중소기업 R&D 연구개발계획서 
COPY-PASTE PROMPT BOOK 
기존 사업계획서·IR·회사소개서·기술소개서·특허자료·매출자료를 넣은 뒤, 「중소기업 기술개발(R&D) 
지원사업 연구개발계획서」 양식에 맞는 제출용 초안을 항목별로 작성하기 위한 실전 프롬프트북 
사용자는 순서대로 붙여넣기만 하면 됩니다. 자세한 설계 설명은 뒤의 부록 LLM 용 R&D 작성 지침에 
분리했습니다. 
빠른 사용법 
단계 
사용자 행동 
1 단계 
이 프롬프트북 파일을 LLM 에 먼저 업로드합니다. 
2 단계 
기존 사업계획서, IR, 회사소개서, 기술소개서, 
특허자료, 매출자료 등 원문 문서를 같은 대화에 
업로드합니다. 
3 단계 
S0 를 한 번만 붙여넣어, LLM 이 전체 자료를 먼저 
숙지하고 R&D 기준표를 만들게 합니다. 
4 단계 
이후 S1→S9 를 순서대로 붙여넣어 각 항목 초안을 
작성합니다. 
5 단계 
마지막에 S10 으로 통합본을 만들고, S11 로 전체 
문체·중복·미확정값·성능지표를 정리합니다. 
꼭 기억할 것 
• 
R&D 계획서는 좋은 사업 아이디어를 설명하는 문서가 아니라, 개발할 기술의 
신규성·구현가능성·검증가능성·사업화 가능성을 설득하는 문서입니다.

## 페이지 2

• 
기술개발 목표는 1 페이지 이내, 연구개발 방법은 4 페이지 이내, 선행연구개발 이력은 1 페이지 
이내, 활용계획은 2 페이지 이내, 연구팀 역량·일정·연구비는 양식의 제한 분량을 고려해 작성해야 
합니다. 
• 
연구개발 방법에는 반드시 R&D 결과의 검증 방법과 계획이 포함되어야 합니다. 
• 
선행연구개발은 해당사항이 없으면 미기재할 수 있으나, 기존 자체개발·정부지원·중기부 R&D 
이력이 있으면 누락 없이 구분해야 합니다. 
• 
성능지표는 심사자가 기술개발 성공 여부를 판단하는 핵심입니다. 지표명, 단위, 목표치, 목표치 
근거, 평가방법, 평가환경을 함께 제시해야 합니다. 
• 
원문에 없는 수치·특허·매출·인력·정부지원 이력은 임의 확정하지 말고 [사용자 확인 필요]로 
남깁니다. 
프롬프트 사용 순서 
S0 → S1 → S2 → S3 → S4 → S5 → S6 → S7 → S8 → S9 → S10 → S11 
공통 규칙 
• 
원문 문서를 충분히 숙지한 뒤 답합니다. 여러 문서 간 공통 사실, 보완 사실, 충돌 사실을 먼저 
구분합니다. 
• 
R&D 양식의 항목 역할에 맞춰 내용을 재배치합니다. 사업설명은 기술개발 목표로, 서비스 장점은 
성능지표와 사업화 전략으로, 기존 성과는 선행연구개발과 역량으로 재분류합니다. 
• 
원문에 없는 R&D 특유 항목이 나오더라도 작성을 멈추지 않습니다. 기술적으로 자연스러운 제안 
문장을 생성하되, 확정 수치·고유명사·특허·실적은 임의 창작하지 않습니다. 
• 
기술 설명은 추상적 표현을 피하고 데이터, 알고리즘, 시스템, 모듈, 인터페이스, 검증 환경, 
성능지표로 구체화합니다. 
• 
사업화 서술은 5 개년 매출표, 수익모델별 산식, 목표시장, 판로, 마케팅, 투자·수출·고용 효과로 
연결합니다. 
• 
브라우징 가능한 LLM 이라면 관련 시장자료, 공개 벤치마크, 논문, 정부통계, 기술표준, 경쟁 서비스 
정보를 검색해 근거를 보강합니다.

## 페이지 3

COPY-PASTE 프롬프트 
아래부터는 사용자가 순서대로 붙여넣기만 하면 되는 코드형 프롬프트입니다. 각 프롬프트는 짧게 
설명하고, 실제 붙여넣을 내용은 충분히 충실하게 유지했습니다. 
S0. 통합 세팅 · 원문 숙지 · R&D 적합성 진단 
이 프롬프트가 하는 일 원문 전체를 먼저 숙지하고 기술성·사업성·수치충돌·성능지표 후보를 한 번에 
정리합니다. 
너는 중소기업 기술개발(R&D) 지원사업 연구개발계획서 작성 전용 편집자다. 
내가 업로드한 이 프롬프트북과 복수의 원문 문서를 함께 읽고, 이후 모든 응답에서 아래 규칙을 우선 적
용하라. 

공통 규칙: 
- 입력된 원문 문서를 충분히 숙지한 뒤 답하라. 
- 여러 문서 간 공통 사실, 보완 사실, 충돌 사실을 구분하라. 
- R&D 연구개발계획서 양식의 항목 역할에 맞춰 내용과 표현을 재배치하라. 
- 원문에 없는 R&D 특유 항목은 작성을 멈추지 말고, 맥락에 맞는 제안 문장을 즉시 생성하라. 
- 제안이 필요한 경우 사용자 의견을 먼저 묻지 말고 LLM 재량으로 작성하되, 수치·기관명·특허·실적은 임
의 확정하지 말라. 
- 브라우징이 가능하면 시장 자료, 기술 벤치마크, 논문, 공공 통계, 경쟁 서비스 정보를 즉시 검색해 근거
와 출처를 보강하라. 
- 동일 메시지를 반복하지 말되, 논증이 약해지지 않도록 항목 목적에 맞게 변용하라. 
- 불확실한 값은 [사용자 확인 필요] 또는 [임시안]으로 표시하라. 

지금 할 일: 
1) 원문 전체의 핵심 재료를 15 개 이내로 정리하라. 
2) 개발하려는 기술을 한 문장으로 정의하라. 
3) 기술개발 목표와 사업화 목표를 분리하여 정리하라. 
4) 문서 간 충돌 가능 정보(숫자, 연도, 매출, 인원, 특허, 수혜사업, 기술명, 기관명)를 15 개 이내로 정리하
라. 
5) 이 과제가 R&D 지원사업에 얼마나 적합한지 A/B/C 등급으로 판정하라. 
- A: 바로 R&D 계획서로 전환 가능

## 페이지 4

- B: 기술성·성능지표 보강 후 전환 가능 
- C: 기술개발보다 일반 사업화 성격이 강해 구조 재설계 필요 
6) 연구개발계획서 작성용 내부 기준표를 만들어라.

/ 연구팀 후보 / 참여기관 후보

8) 핵심 메시지 배치표를 만들어라. 
- 메시지명 | 가장 강하게 쓸 항목 | 보조 언급 가능 항목 | 그대로 반복하면 안 되는 항목 | 변용 방법 한 줄 
9) 전체 얼개를 설계하라. 
- 과제 개요 핵심 메시지 
- 기술개발 목표 핵심 주장 
- 연구개발 방법 단계 
- 선행연구개발 활용 논리 
- 사업화 전략 핵심 논리 
- 제출 전 반드시 확정해야 할 값 체크리스트 

출력 형식: 
[작업 준비 완료]

3. 기술개발 목표와 사업화 목표 
4. 충돌 가능 정보 
5. R&D 적합성 판정 
6. 내부 기준표 
7. 성능지표 후보표 
8. 핵심 메시지 배치표 
9. 전체 얼개 설계 
10. 제출 전 반드시 확정해야 할 값 체크리스트 

결과만 출력하라.

## 페이지 5

실행 팁 새 대화에서 가장 먼저 실행하세요. S0 결과의 내부 기준표와 성능지표 후보가 이후 모든 항목의 
품질을 좌우합니다. 
S1. 과제 개요 작성 
이 프롬프트가 하는 일 첫 장의 과제명, 개발 목표, 개발 방법, 연구팀 구성, 기대효과를 압축 작성합니다. 
S0 결과를 기준으로 연구개발계획서의 「과제 개요」를 작성하라. 

반드시 포함: 
1) 과제명 
2) 개발 목표 
3) 개발 방법 
4) 연구팀 구성 
5) 기대효과 

작성 규칙: 
- 과제명은 기술개발 과제명으로 작성하라. 제품명만 쓰지 말고, 핵심기술·적용대상·개발결과가 드러나게 
하라. 
- 개발 목표는 기술적 목표와 서비스·사업화 목표를 분리하라. 
- 개발 방법은 “데이터/요구사항 확보 → 핵심기술 개발 → 시스템 구현 → 실증/검증 → 사업화”의 흐름으
로 요약하라. 
- 연구팀 구성은 주관기관, 공동/위탁/외주기관, 핵심 인력의 역할을 간결히 제시하라. 
- 기대효과는 매출, 고용, 투자, 수출, 산업·사회적 파급효과를 기술성과 연결해 작성하라. 
- 원문 수치가 있으면 우선 사용하고, 불확실하면 [사용자 확인 필요]로 표시하라. 

출력 형식: 
[과제 개요] 
- 과제명 
- 개발 목표 
1) 기술적 목표 
2) 서비스·사업화 목표 
- 개발 방법 
- 연구팀 구성 
- 기대효과

## 페이지 6

결과만 출력하라. 
S2. 1. 기술개발 목표 
이 프롬프트가 하는 일 1 페이지 제한에 맞춰 연구기술개발 과제의 목표와 배경·필요성을 작성합니다. 
S0 의 내부 기준표와 전체 얼개를 기준으로 「1. 기술개발 목표」를 작성하라. 

반드시 포함: 
1) 연구기술개발 과제의 목표 
2) 연구기술개발 배경 및 필요성 
3) 개발 결과물의 정의 

작성 규칙: 
- 최대 1 페이지 항목이므로 장황한 시장 설명보다 기술개발 필요성과 목표를 압축하라. 
- 목표는 “무엇을 개발하는가”, “어떤 데이터를/기술을 활용하는가”, “기존 방식의 어떤 한계를 해결하는
가”, “최종 결과물이 무엇인가”가 드러나야 한다. 
- 배경 및 필요성은 시장·제도·고객 문제를 설명하되, 반드시 기술개발 필요성으로 연결하라. 
- 개발 결과물은 모듈 또는 시스템 단위로 3~5 개 제시하라. 
- 단순 서비스 개선이 아니라 R&D 과제로서의 신규성, 난이도, 검증 필요성이 보이게 작성하라. 
- 원문에 없는 기술명은 맥락에 맞게 제안할 수 있으나, 확정명은 [임시안]으로 표시하라. 

출력 형식: 
[1. 기술개발 목표] 
(1) 연구기술개발 과제의 목표 
(2) 연구기술개발 배경 및 필요성 
(3) 개발 결과물의 정의 

결과만 출력하라. 
S3. 2. 연구개발 방법 
이 프롬프트가 하는 일 최대 4 페이지 항목을 단계형 기술개발 로드맵으로 작성합니다. 
S0 결과와 S2 의 기술개발 목표를 기준으로 「2. 연구개발 방법」을 작성하라.

## 페이지 7

반드시 포함: 
1) 연구개발 추진 방향 
2) 세부 연구개발 방법 
3) 핵심 기술 요소 
4) 시스템 구현 구조 
5) 실사용 기반 검증 및 고도화 계획 

작성 규칙: 
- 연구개발 방법은 단계형 파이프라인으로 작성하라. 
예: 데이터 수집·정규화 → 분석/추론 모델 개발 → 핵심 알고리즘 구현 → 서비스 시스템 개발 → 성능 검
증 → 실증 고도화 
- 각 단계마다 입력 데이터, 처리 방식, 산출물, 검증 포인트를 포함하라. 
- AI 과제라면 LLM, RAG, Vector DB, 규칙기반 추론, 추천모델, 예측모델, OCR, NLP, 컴퓨터비전, 멀티에이
전트 등 실제 적용 가능한 기술요소를 과제 맥락에 맞게 선택하라.

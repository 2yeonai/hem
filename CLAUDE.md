## What this is

This is an **Obsidian vault** (`.obsidian/`), not a single codebase. It holds four independent, unrelated project folders, each a self-contained "AI м§Ғмӣҗ(employee) мҠӨнӮ¬" for a different small business the user runs or is building for:

- `1. Projects/нҒҙлЎңл“ң ai мһҗлҸҷнҷ”/` вҖ” shared schema/charter definitions used by the other three skills (see below). Note: despite the "1. - `1. Projects/нҒҙлЎңл“ң кҪғм§‘ ai/` вҖ” flower shop automation (вҡ пёҸ see Gotchas вҖ” the folder root is mislabeled)

Alongside these, the vault root also holds a general-purpose **PARA + 5. Zettelkasten** note-taking structure (see below) вҖ” unrelated to the four AI-skill folders, used for the user's personal/knowledge-management notes rather than these projects.

- **(RESOLVED 2026-07-20, root cause found) `.git/index.lock` "cannot remove" is a mount-level `unlink()` restriction, not a real concurrent-process conflict.** On this Windows/bash-mount setup, deleting files under `.git/` from the bash sandbox (`rm`, Python `os.remove`, etc.) fails with "Operation not permitted" even when no other git process is running вҖ” but **renaming the same file (`mv`) succeeds**. This is why stale `index.lock.stale-<timestamp>` files accumulate (past sessions worked around this by renaming instead of deleting) and why plain `git commit`/`git status` can hang 40s+ then fail: git's own internal unlink-and-retry cleanup steps stall on the same restriction. **Working fix, fastest to slowest:** в‘  if `.git/index.lock` exists, `mv .git/index.lock ".git/index.lock.stale-$(date +%s)"` (never `rm`) before any git command. в‘Ў If a plain `git commit` still hangs/times out afterward, bypass porcelain commit entirely with plumbing, which only needs rename (not unlink) and finishes in under a second: `git write-tree` вҶ’ `git commit-tree <tree> -p $(git rev-parse HEAD) -m "..."` вҶ’ `git update-ref refs/heads/main <new-sha>`. Loose-object "warning: unable to unlink tmp_obj_..." messages during this are harmless вҖ” the rename into place already succeeded, only the temp-file cleanup fails. в‘ў Pre-commit hooks (`vault_check.py`, `context_budget_check.py`) are NOT the bottleneck вҖ” both run in ~1-1.5s standalone; don't waste time bypassing them with `--no-verify` first. If `mv` itself ever fails too (not yet observed), that's a genuine escalation вҖ” ask нҳңлҜё to delete via Windows Explorer directly (confirmed working 2026-07-20).

## Obsidian PARA + 5. Zettelkasten structure

Separate from the four AI-skill folders above, the vault root also has a general PARA + 5. Zettelkasten structure for personal notes ("м§ҖмӢқ л°ңм „мҶҢ"/knowledge engine vs. "н”„лЎңм қнҠё мһ‘м—…лҢҖ"/project workbench, kept deliberately apart):

- `1. Projects/` вҖ” time-boxed work with a deadline and a Definition of Done
- `2. Areas/` вҖ” ongoing responsibilities with no end date
- `3. Resources/` вҖ” topic reference material
- `4. Archive/` вҖ” completed/inactive items (never deleted, only moved here)
- `5. Zettelkasten/` вҖ” permanent knowledge, split into `00. Inbox/` (fast capture) вҶ’ `10. Literature/` (source-bound notes) вҶ’ `20. Permanent/` (atomic, densely-linked insight notes)
- `6. Templates/` вҖ” note templates and the template registry
- `7. Attachments/` вҖ” binary files (images, PDFs) referenced by notes

Each of these folders has its own `CLAUDE.md` with the detailed rules (frontmatter schema, Dataview queries, tag conventions, maintenance checklist) for that folder вҖ” read the relevant one before creating or filing a note there rather than assuming from the folder name. `5.

# app_completion_criteria.md — 앱 MVP 완성 기준

Hyemi Grant Factory 앱이 "완성"으로 판정되려면 아래 전부가 실제로 작동해야 한다.

## 1. 입력 (필수값 / 선택값)

| 입력 | 구분 | 비어 있을 때 처리 |
|---|---|---|
| 프로젝트명 | 필수 | 생성 거부 |
| 공고문 원문 (raw_text) | 선택 | **[확인 필요] 모드 — 전 판정에 경고, LOCK 금지** |
| 신청자 정보 | 필수(이름) / 나머지 선택 | 미기재 항목 [확인 필요] 누적 |
| 아이디어 후보 1개 이상 | 필수 | 생성 거부 |
| 심사표·배점 | 선택 | 내부 기준(evaluation_rubric) 대체 + [확인 필요] |
| 예산 계획 | 선택 | 예산 판정 불가 경고 + 견적서 필요 표시 |
| 증빙 목록 | 선택 | 확보 0건이면 해당 후보 감점·단정 금지 경고 |

## 2. 출력 15종 (프로젝트별 생성)

공고 요약 / 자격·제외 리스크 / 심사표·배점 분석 / 아이디어 비교표 / LOCK 판정 / 과제명·한줄요약 후보 / 위험 표현 검출 / 예산 리스크 / 누락 증빙 / 심사위원 검수 / 예상 점수표 / 발표 공격 질문 / 재작업 지시서 / HANDOFF / 다음 실행 프롬프트

## 3. LOCK 가능 조건 (전부 충족 시에만 "LOCK 가능" 표시)

1. 공고문 원문 존재 2. 탈락급 리스크 0건 3. 자격 3종 사람 확인 기록 4. 최종 아이디어 사람 확정 5. 예산 확인 기록 6. 위험 표현 0건 7. 선정 아이디어 확보 증빙 1건 이상
→ 충족해도 앱은 REVIEWED까지만. **LOCKED 기재는 사람이 06_locks/LOCK_STATUS.md에.**

## 4. 자동 중단 조건

체납=있음, 중복수혜=있음 → 즉시 BLOCKED 표시 + 이후 판정에 "제출 불가" 고정

## 5. 파일 저장 규칙

- 입력: `01_inputs/<p>/input.json` / 산출물: `04_outputs/<p>/` / 검수: `05_reviews/<p>/` / 인계: `10_handoff/HANDOFF_<p>.md`
- 기계 판독용 통합 결과: `04_outputs/<p>/results.json`
- 재실행 시 동일 파일 덮어쓰기 (입력이 바뀌면 결과도 바뀌는 것이 정상)

## 6. 오류 처리

- 입력 파싱 실패: 화면에 오류 표시, 파일 쓰지 않음
- 판정 불가: 점수 대신 "[확인 필요]" — 임의 추정 점수 금지
- 규칙 기반 한계 명시: 모든 판정 화면에 "규칙 기반 예비 판정 — 사람 확정 전 참고용" 고지

## 7. 완성 판정 체크리스트 (11차 검수에서 사용)

- [ ] 로컬에서 명령 1개로 실행되는가
- [ ] 화면에서 입력→결과까지 클릭만으로 가능한가
- [ ] 샘플 프로젝트가 API 키 없이 돌아가는가
- [ ] 공고 원문 없이 LOCK이 절대 안 되는가
- [ ] 위험 표현·예산·증빙 부족이 화면에서 먼저 보이는가
- [ ] Markdown/JSON/HANDOFF/다음 프롬프트 내보내기가 되는가

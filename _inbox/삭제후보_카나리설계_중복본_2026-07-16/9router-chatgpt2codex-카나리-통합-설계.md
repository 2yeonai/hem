# 9Router · chatgpt2codex 카나리 통합 설계

## 결론

9Router와 chatgpt2codex를 별도 실험 환경이 아니라 **메인 AI OS의 라우팅 레이어에 "카나리 레인(canary lane)"으로 직결**한다. 공식 경로(Anthropic API / OpenAI 공식 API)는 항상 살아있는 fallback으로 유지하고, 실패의 폭발 반경은 권한·브랜치·계정·데이터·킬스위치 5중 격리로만 제한한다. 테스트 성공 시 **코드 재구축 없이 설정값(트래픽 비율 + 작업 클래스 allowlist) 변경만으로** 메인 운영 범위가 확대되는 구조다.

---

## 1. 아키텍처: 단일 진입점, 이중 레인

```
요청 → AI OS Dispatcher (기존 라우팅 정책 그대로)
         │
         ├─ Stable Lane (기본): 공식 API 직결
         │    Haiku / Sonnet 5 / Opus 4.8 / Fable 5
         │
         └─ Canary Lane (트래픽 X%, 작업 클래스 allowlist):
              ├─ 9Router → 하위 모델 라우팅 실험
              └─ chatgpt2codex → Codex 실행 작업
         │
         └─ Fallback Chain: Canary 실패 시 동일 요청을 Stable로 자동 재시도
```

핵심 원칙 세 가지.

- **무접점 실험 금지**: 카나리도 메인 Dispatcher를 통과한다. 동일한 로그 스키마(`why_fable` 로그 포함), 동일한 카드 포맷, 동일한 캐싱 프리픽스를 사용한다. 별도 파이프라인을 만들면 성공해도 이식 비용이 다시 든다.
- **역할 불변**: 기존 라우팅 정책의 모델 역할표는 카나리에서도 유효하다. 9Router는 Haiku/Sonnet 티어의 디스패치를 대체 실험하는 것이지 Fable 5 판단 계층을 건드리지 않는다. chatgpt2codex는 정책상 "실행자(executor)"인 Codex 슬롯에만 꽂힌다 — 전략 판단 경로에는 진입 불가.
- **Fallback은 자동**: 카나리 레인의 오류·타임아웃·품질 게이트 미달 시 사용자 개입 없이 Stable 레인으로 동일 요청이 재시도된다. 카나리 실패는 "지연"으로만 나타나야 하고 "결과 실패"로 나타나면 안 된다.

## 2. 카나리 진입 조건 (작업 클래스 allowlist)

초기 카나리에는 실패해도 복구 비용이 낮은 작업만 흘린다. 기존 정책의 "Never Route to Fable 5" 목록이 그대로 카나리 1차 후보다.

| 단계 | 허용 작업 클래스 | 근거 |
|---|---|---|
| C0 (섀도) | 실트래픽 복제본만 처리, 결과 미사용 | 품질 비교 데이터 수집 |
| C1 | 태깅, 엔티티 추출, 단순 요약, dedupe | 실패 시 재실행 비용 최소 |
| C2 | 초안 작성, 카드 병합, 일반 응답 (Sonnet 티어) | C1 게이트 통과 후 |
| C3 | 코드 편집·테스트 (chatgpt2codex 경유) | 브랜치 격리 하에서만 |
| 제외(영구) | Fable 5 판단 작업, 결제·외부 발신·삭제 등 비가역 작업 | 카나리 대상 아님 |

## 3. 5중 격리 설계

**① 권한**: 카나리 레인 전용 API 키·토큰 발급, least-privilege. 기본 read-only. 쓰기 권한은 아래 브랜치·데이터 격리 범위 안에서만. 외부 발신(메일, 메시지, 배포)은 권한 자체를 미부여.

**② 브랜치**: chatgpt2codex의 모든 코드 작업은 `canary/*` 브랜치에서만 커밋. `main`은 보호 브랜치로 카나리 계정의 push/merge 권한 없음. 머지는 항상 사람(또는 Stable 레인의 리뷰 게이트)이 수행.

**③ 계정**: 카나리 전용 계정/워크스페이스 분리. billing 분리로 비용 폭주가 즉시 가시화되고, 계정 정지·키 폐기만으로 전체 카나리를 물리적으로 차단 가능.

**④ 데이터 접근**: 카나리에는 원문(raw)이 아니라 **카드 + diff + 최대 3개 스니펫**만 전달(기존 카드 스키마 재사용). 민감 데이터는 allowlist 방식 — 명시적으로 허용한 데이터셋 외에는 접근 불가. 카나리가 생성한 산출물은 별도 스테이징 스토어에 격리 저장 후 게이트 통과 시에만 메인 데이터로 승격.

**⑤ 킬스위치**: 환경변수/설정 플래그 1개(`CANARY_ENABLED=false`)로 즉시 Stable 100% 복귀. 수동 스위치에 더해 자동 트립 조건을 건다:

- 오류율 > 5% (5분 이동 윈도우)
- p95 지연이 Stable 대비 2배 초과
- 시간당 비용이 예산 임계치 초과
- 품질 샘플링 점수가 Stable 대비 유의미 하락

자동 트립 시 알림 + 원인 로그 스냅샷 자동 저장. 재활성화는 수동으로만.

## 4. 승격 경로 (재구축 없는 확장)

승격은 배포가 아니라 **설정 변경**이다. 두 개의 다이얼만 움직인다: 트래픽 비율, 작업 클래스 allowlist.

| 게이트 | 트래픽 | 통과 기준 (각 단계 최소 1주 또는 N건 이상) |
|---|---|---|
| G0→G1 | 섀도 → 5% | 섀도 품질 비교에서 Stable 동등 이상, 오류율 <2% |
| G1→G2 | 5% → 25% | 자동 트립 0회, fallback 발동률 <3% |
| G2→G3 | 25% → 50% | 비용/1k 작업이 Stable 대비 절감 확인 |
| G3→G4 | 50% → 100% | 전 지표 안정, 킬스위치 리허설 1회 통과 |

100% 도달 시 해당 작업 클래스에서 카나리 레인이 곧 Stable이 되고, 기존 공식 경로는 fallback 체인으로 강등되어 유지된다(제거하지 않음). 다음 작업 클래스(C2, C3)는 같은 게이트를 반복한다 — 파이프라인·로그·게이트 로직이 전부 재사용되므로 재구축이 없다.

## 5. 관측성 (기존 로그 스키마 확장)

카나리 호출마다 기존 Fable 로그 스키마에 필드 추가:

```json
{
  "lane": "canary|stable",
  "canary_component": "9router|chatgpt2codex",
  "task_class": "C1|C2|C3",
  "fallback_triggered": false,
  "fallback_reason": "",
  "latency_ms": 0,
  "cost_estimate": 0,
  "quality_sample_score": null
}
```

대시보드는 Stable vs Canary를 항상 나란히 비교(오류율, p95, 비용/1k, fallback률). 게이트 판정은 이 대시보드 수치로만 한다 — 인상 평가 금지.

## 6. 실패 시나리오별 대응

| 시나리오 | 영향 범위 | 대응 |
|---|---|---|
| 9Router 오라우팅(비싼 모델로 폭주) | 카나리 계정 billing만 | 비용 트립 → 자동 차단, Stable 무영향 |
| chatgpt2codex가 잘못된 코드 커밋 | `canary/*` 브랜치만 오염 | 브랜치 폐기, main 무영향 |
| 인증 토큰 만료/차단 (비공식 경로 리스크) | 해당 요청만 지연 | fallback 자동 재시도, 트립 카운트 증가 |
| 업스트림 약관/차단 이슈로 서비스 자체 중단 | 카나리 레인 전체 | 킬스위치 → Stable 100%, 운영 무중단 |
| 데이터 유출 의심 | allowlist 데이터셋만 | 카나리 키 폐기 + 해당 데이터셋 감사 |

주의: chatgpt2codex류의 비공식 브리지는 업스트림 약관 변경·차단에 상시 노출된다. 그래서 이 설계에서 공식 경로는 "실험 기간의 임시 fallback"이 아니라 **영구 fallback**이며, 카나리가 100%가 되어도 제거하지 않는다.

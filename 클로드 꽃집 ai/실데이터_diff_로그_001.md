- 날짜: 2026-07-07
- 근거: decision-log 5번 항목(꽃집 복제 진행 + 수정 diff 로그 기록), Fable plan step 2
- 원칙: "복제본에서 발견한 결함을 복제본에서만 수정 금지 — 원본 수정 후 반영 경로를 남길 것" (stop_rule) 준수 — 여기서는 아직 코드를 고치는 단계가 아니라, 실제 데이터를 12봇 설계에 통과시켜보며 gov-support 코드와 얼마나 다른지 기록하는 단계.

## ✅ 이름 뒤바뀜 — "오타"가 아니라 정상적인 리스크 패턴으로 확정 (사용자 확인 완료, 2026-07-07)
받은 데이터가 두 가지였습니다.

1. **스크린샷(실제 수신 문자 원문)**: `창녕군청 재무과 백승흔10만원 난 승진축하 부곡면장 박혜미`
2. **직접 타이핑해주신 버전**: `창녕군청 재무과 박혜미 10마넌 난으로 보내줘 승진축하합니다 부곡면장 백승흔`

위치 기반으로 읽으면(국내 화환 주문 문자의 관용적 어순 — "수신 기관/부서 + 수신자" 다음 "발신 직책 + 발신자") 두 버전에서 수신자/발신자 이름이 정반대로 배정됩니다.

**사용자 확인**: "모르는 사람이 보내면 헷갈리는 게 맞음." 즉 이건 둘 중 하나의 오타를 찾아 "정답"을 확정하는 문제가 아니라, **발신자를 모르는 경우(낯선 번호로 처음 오는 주문 등) 이름의 상대적 위치만으로는 수신인/발신인을 확실히 구분할 수 없다는 것 자체가 이 도메인의 정상적인 리스크 패턴**이라는 뜻입니다. 이 사례를 "확인 필요"로 남겨서 사람이 실제로 전화해서 재확인하도록 만드는 게 올바른 동작이지, 파서가 어느 한쪽으로 자신 있게 확정해버리면 그게 오히려 버그입니다. 아래 골든셋 항목에 이 리스크 패턴을 정식 필드로 기록했습니다(스크린샷 원문을 대표값으로 채택하되, 리스크 자체는 "미해결로 남기는 것이 정답"인 케이스로 표시).

## 중요한 발견 1 — "STT/음성" 전제가 이 케이스에는 안 맞음
이 주문은 **문자로 직접 수신**됐습니다. 통화 녹음이 아니라서 받아쓰기봇(STT)이 낄 자리가 없고, 수집봇 출력이 곧바로 말귀봇 입력이 됩니다. 이건 지난번 `manifest.schema.v2.yaml`에 남겨뒀던 ASSUMPTION("주문 채널이 텍스트/음성 둘 다 있다고 가정")이 실제로 맞는 가정이었음을 확인해주는 사례입니다 — 텍스트 전용 주문이 실제로 존재하고, 이 경우 STT는 pass-through조차 아니라 **아예 건너뜀**이 맞습니다(pass-through라면 "빈 값을 그대로 통과"인데, 이 케이스는 애초에 호출될 이유가 없음). `12봇_kind분류표.yaml`의 `transcription_bot.depends_on: [collection_bot]`은 유지하되, 파이프라인 실행기가 `source_type == "sms"|"kakao"`일 때 이 stage를 스킵하는 조건부 라우팅이 필요하다는 게 이번에 새로 드러난 요구사항입니다(이전엔 몰랐던 것 — v2 스키마의 `depends_on`만으로는 "조건부 스킵"을 표현 못함, open_issue로 추가 필요).

## 중요한 발견 2 — 리본 문구 조립에 실제로 필요한 파싱 난이도
"난으로 보내줘 승진축하합니다"에서 실제로 뽑아야 하는 것:
- 상품: "난" (동양란/서양란 등 세부 품종은 이 문구만으로 확인 불가 → `missing_fields`에 들어가야 함)
- 리본 문구 원문: "승진축하"
- 리본 문구 출력형: 설계보고서 사전에 있던 "승진축하 → 승진을 축하드립니다." 규칙 그대로 적용 가능
- 금액: "10만원"(스크린샷) vs "10마넌"(타이핑) → 둘 다 100000으로 정규화되어야 함. 설계보고서의 금액 파서 규칙("십마넌"/"10마넌" → 100000)이 정확히 이 케이스를 커버함 — 이 부분은 사전에 이미 설계돼 있던 규칙이 실데이터에서도 그대로 들어맞은 몇 안 되는 사례.

## 파일·함수 단위 diff — gov-support-matching-skill `scripts/run.py` 기준
아래는 이 실제 주문 1건을 처리하려고 시도했을 때, gov-support의 각 함수가 그대로 재사용 가능한지 확인한 결과입니다.

| gov-support 함수 | 역할 | 이 꽃집 주문에 재사용 가능? | 근거 |
|---|---|---|---|
| `collect_and_extract_announcements()` | 공고문 JSON 로드 | **0% — 대응 개념 자체가 없음** | "공고"라는 개념이 꽃집 주문에는 없음. 이 문자 자체가 이미 "공고"에 해당하는 게 아니라 "신청 대상"에 해당 — 아예 다른 층위 |
| `check_eligibility_and_disqualification()` | 자격/결격 요건 판정 | **0%** | 신청자격이라는 개념이 꽃집 주문에 없음 |
| `match_programs()` | 공고-사업자 매칭 | **0%** | 매칭 대상(공고 여러 건 중 최적)이 꽃집엔 없음 — 꽃집은 "이 주문 1건을 어떻게 처리할지"이지 "여러 후보 중 고르기"가 아님 |
| `check_budget_compatibility()` | 예산 총액/제외항목 검사 | **0%, 개념적 유사물도 없음** | "예산 한도"/"집행 제외 항목"이라는 개념이 꽃집 주문엔 대응 개념이 아예 없음(가격은 있지만 "한도 초과 검사"할 대상이 없음) — Fable의 blind_spot 예측(check_budget_compatibility가 도메인 종속)보다 한 단계 더 심각: 이식이 아니라 "삭제 대상" |
| `draft_application()` | 신청서 6개 섹션 초안 작성 | **0%** | "사업동기/팀역량/예산계획" 같은 섹션 자체가 꽃집 주문에 없음. 리본상품금액봇이 만드는 "출력문구"가 형태상 가장 가까운 대응물이지만 내용/목적이 완전히 다름 |
| `judge_mode_self_review()` | 감점위험/과장표현/증빙부족 탐지 | **모양(shape)만 재사용 가능, 내용 0%** | "risks 목록 + pass 추천"이라는 출력 형태는 검수매니저에도 그대로 쓸 수 있음. 그러나 `EXAGGERATION_WORDS`("최초", "유일" 등 과장 표현 사전), `RUBRIC_SYNONYMS`(심사배점 동의어) 같은 내용은 전부 정부지원사업 전용 — 꽃집엔 "과장 표현 탐지"라는 개념 자체가 없고, 대신 "수신자/발신자 이름 혼동"처럼 완전히 다른 리스크 유형이 필요 |
| `apply_quality_gate()` | confidence + judge_review → lock_state | **패턴은 재사용 가능** | "숫자 기준 + 질적 기준 둘 다 통과해야 확정"이라는 게이트 구조 자체는 검수매니저→사람검수자 흐름에 그대로 대응됨(v2 스키마의 self_check/human_gate 모드가 이미 이 구조를 일반화해둔 상태) |
| `log_failure()` / `failure_log_format` | 실패 로그 남기기 | **100% 재사용 가능** | timestamp/step/input_summary/error/confidence_at_failure 필드는 완전히 도메인 무관 |
| `today()` / `parse_date()` | 날짜 유틸 | **부분 재사용 가능** | 함수 자체는 재사용되지만, "마감일 지남 여부"라는 판단 로직(공고 도메인 전용)은 안 쓰이고, 꽃집에서는 "배송 희망일시가 과거인지" 같은 전혀 다른 판단에 쓰여야 함 — 유틸은 같아도 판단 규칙은 새로 짜야 함 |

### 요약
5개 핵심 판단 함수(수집/자격체크/매칭/예산체크/초안작성) 중 **재사용 가능한 것은 0개**입니다. `judge_mode_self_review`와 `apply_quality_gate`도 "출력 모양"만 재사용되고 내용물은 전부 새로 짜야 합니다. 실제로 재사용되는 건 로깅 포맷과 날짜 유틸 함수 뿐입니다 — 코드량 기준으로는 전체의 10~15% 정도로 추정됩니다(정확한 라인 수 비교는 아직 안 함).

이건 Fable의 blind_spot 예측(`check_budget_compatibility`, 배점 동의어 테이블, eligibility 필드가 도메인 종속적일 것)보다 실제로는 더 심각합니다 — 저 3개만 종속적인 게 아니라 **핵심 판단 로직 전체**가 종속적이고, 공유되는 건 파이프라인의 "모양"(quality_gate 게이트 구조, 로깅 포맷)뿐입니다. 이건 오히려 Fable의 원래 결론("manifest 하나 유지, core는 아직 만들지 말 것")을 더 강하게 뒷받침하는 근거이기도 합니다 — 지금 억지로 공통 코드를 뽑아내려 했다면 "로깅 포맷 공유하자고 두 도메인을 억지로 묶는" 꼴이 났을 것입니다.

## 채널 우선순위 정정 (사용자 확인, 2026-07-07)
문자는 **보조 수단**이고 실제 주문은 **대부분 전화(음성)로 옴**. 즉 받아쓰기봇(STT)이 파이프라인의 "가끔 스킵되는 부가 stage"가 아니라 **메인 흐름의 핵심 stage**라는 뜻 — 지난번 `manifest.schema.v2.yaml`의 ASSUMPTION("텍스트/음성 두 채널 모두 존재")은 방향은 맞았지만 비중을 정반대로 가정하고 있었음(텍스트가 기본, 음성이 예외인 것처럼 서술됨). 아래 "다음에 필요한 것"에 반영.

이번 텍스트 케이스(#001)는 이걸로 충분히 기록됐다고 보고 텍스트 채널 조사는 여기서 마무리. 다음은 음성 주문(STT 필요한 실제 메인 흐름) 검증 차례.

## 다음에 필요한 것
- **[우선순위 최상위] 통화 녹음 데이터 확보** — 실제 주문의 대부분을 차지하는 채널인데 아직 골든셋에 1건도 없음. 이게 있어야 받아쓰기봇(STT)/말귀봇의 실제 병목(STT 오인식률, 사투리/전화음질 등)을 확인할 수 있음 — 텍스트 케이스만으로는 이 파이프라인의 핵심 stage를 전혀 검증 못한 상태
- 골든셋을 최소 5~10건으로 늘리기 — **음성 케이스 위주로**. 텍스트는 이번 1건으로 보류(추가 텍스트 케이스는 후순위)

- 실제 코드로 이 주문을 처리해보려면 correction_bot/order_draft_bot/ribbon_price_bot에 해당하는 함수를 처음부터 새로 작성해야 함(재사용 가능한 골격 없음) — 아직 작성 안 함, 골든셋이 더 쌓인 뒤(특히 음성 케이스) 스키마부터 확정하는 게 순서상 맞다고 판단됨(임의로 코드부터 짜지 않음)

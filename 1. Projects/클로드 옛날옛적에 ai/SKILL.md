# 옛날옛적에 — AI 키즈 동화·동요 공장 (kids-story-ost-factory)

이 파일은 2026-07-14 처음 작성됨. 정부지원사업/꽃집/방역 스킬의 SKILL.md 구성(트리거 조건/하는 일/알려진 한계/실행방법/관련 문서)을 그대로 따름. **1차 스캐폴드 — 완성된 서비스 아님.**

## 언제 이 스킬을 쓰는가 (트리거 조건)
manifest.yaml의 `triggers` 섹션과 동일 — event 트리거 1종만 존재.

- **event 트리거 (source: story_request)**: 보호자가 "아이가 이런 이야기를 해줬다 / 이 캐릭터로 동화 만들어줘" 식으로 요청을 넣으면 즉시 시작. 진입 stage는 `발화수집봇`.
- 정기 스케줄 트리거(예: 매주 자동 추천)는 아직 없음 — 사용 패턴상 필요성이 확인되지 않았음(open_questions 참고).

## 언제 쓰면 안 되는가
- **아이 본인이 직접 실시간으로 자유 텍스트를 입력하는 용도** — 이 공장은 "보호자가 아이가 말한 것을 대신 입력"하는 형식으로만 설계됐다(2026-07-14 혜미 결정, decision-log §9). 아동이 직접 기기를 조작해 입력하는 시나리오는 전제 자체가 다르므로 지금 구조로는 다루지 않는다.
- **실존 IP 캐릭터를 다수 사용자 대상으로 서비스화(배포)하는 용도** — 지금은 "개인소유+테스트" 단계라 실존 캐릭터명 입력을 막지 않지만, 배포 시점에는 저작권·라이선스 재검토가 필수다. 재검토 없이 배포용으로 그대로 쓰면 안 된다.
- **실제 오디오(노래) 파일이 나온다고 안내하는 것** — 지금 OST생성봇은 가사·스타일노트 "텍스트"만 mock 생성한다. 실제 작곡/TTS 합성은 이 스캐폴드 범위 밖.
- **원본 기획서("전체본.md") 없이 UI/전체 플로우/타겟 연령대까지 다 확정됐다고 안내하는 것** — 아래 "알려진 한계" 참고, 이 스캐폴드는 세션로그 요약만으로 1차 추정한 것.

## 이 스킬이 하는 일 (한 문단 요약)
보호자가 아이가 한 말(발화)을 대신 입력하면(`발화수집봇`), 그 내용에서 캐릭터·주제·아이 이름/나이·톤·OST 포함 여부 같은 구조화된 요청 필드를 뽑아내고(`입력정리봇`), 최소 수준의 부적절 표현 필터를 거치며(`안전선별봇` — 입력 주체가 보호자로 바뀌었어도 콘텐츠 최종 소비자가 아동이라는 사실 때문에 필터는 유지), 요청된 캐릭터 중 실존 IP가 있는지 감지해 지금은 통과시키되 배포 게이트용 플래그만 남긴다(`캐릭터_IP_점검봇`). 이후 동화 본문을 생성하고(`동화생성봇`), OST/노래 기능이 켜져 있으면(기본값 켜짐) 가사·스타일노트를 생성한 뒤(`OST생성봇`), 보호자가 최종 확인해 승인/반려하고(`보호자검수` — 반려 시 동화생성봇부터 재작업), 승인된 동화+OST+배포게이트 메타데이터를 최종 산출물로 묶는다(`산출물포맷봇`).

## 실행 방법
**mock 실행기(`scripts/run.py`)가 있다 — 실제 AI 모델(동화 창작/작사/TTS)은 아직 호출하지 않고, manifest.yaml의 stages/triggers/run_if를 그대로 따라가며 "구조가 처음부터 끝까지 한 번 통과되는지"를 확인하는 목적의 스텁 실행기다.**
```bash
# 스키마·manifest 정합성 검증 (vault root에서 실행)
python3 ai공장짓기/scripts/validate_manifest.py "1. Projects/클로드 옛날옛적에 ai/manifest.yaml"

# 컴파일/문법 확인
python3 -m py_compile "1. Projects/클로드 옛날옛적에 ai/scripts/run.py"

# mock 파이프라인 실행 (샘플 입력)
python3 "1. Projects/클로드 옛날옛적에 ai/scripts/run.py" "1. Projects/클로드 옛날옛적에 ai/test/sample_input.json"

# 또는 스킬 폴더 안에서
cd "1. Projects/클로드 옛날옛적에 ai" && python3 scripts/run.py test/sample_input.json
```
2026-07-14 기준: `validate_manifest.py` FAIL 0 / WARN 0. `run.py`로 샘플 입력 실행 시 8개 stage(발화수집봇→입력정리봇→안전선별봇→캐릭터_IP_점검봇→동화생성봇→OST생성봇→보호자검수→산출물포맷봇) 전부 통과, `산출물포맷봇`까지 도달해 PASS 확인.

## 알려진 한계
- **원본 기획서("전체본.md")가 vault에 없다.** 2026-07-13 세션에서 혜미가 업로드했다고 기록돼 있으나 실제 파일은 vault에 저장되지 않아 지금 시점엔 찾을 수 없음 — `2. Areas/Claude 세션로그/2026-07-13.md` "### 기타" 섹션의 짧은 요약이 유일한 근거다. **이 manifest.yaml/SKILL.md의 필드 구성(자유서술 1개 입력, child_name/child_age 필드, 톤 옵션 등)은 전부 1차 추정이며, 혜미 확인이 필요하다.** 실제 UI 세부사항, 전체 사용자 플로우, 타겟 연령대는 알 수 없는 상태로 남겨둠(임의로 지어내지 않음).
- **아직 실제 AI 모델은 호출하지 않는다.** run.py는 전부 mock 함수로 동작 — 골든셋/실사용 확보 후 실제 모델 호출로 교체 필요.
- **OST 저작권 문제가 미해결이다.** 생성된 가사/스타일이 기존 곡을 표절할 위험을 걸러낼 장치가 아직 없음 — 배포 전 재검토 항목 후보로 open_questions에 남겨둠.
- **캐릭터 IP 충돌은 "테스트 단계 허용"일 뿐, 배포 시점엔 재검토가 필수다.** 2026-07-14 혜미 결정(decision-log §9): 지금은 개인소유+테스트 용도라 실존 캐릭터명(예: mock 사전의 뽀로로/핑크퐁 등) 사용을 막지 않지만, 다수 사용자 대상 서비스화 시점에는 캐릭터별 라이선스를 다시 확인해야 한다. `캐릭터_IP_점검봇`이 남기는 `deployment_gate_required` 플래그가 이 재검토 대상을 표시하는 용도다.
- **안전선별봇이 단순 키워드 매칭 수준으로 가정돼 있다.** 실제로 문맥형 판단이 필요하다고 판명되면 model stage로 격상 검토가 필요하다 — 지금은 local(정형 매칭)로 잠정 분류.
- **tier(입력정리봇=low_cost, 동화생성봇/OST생성봇=mid)는 꽃집/방역 사례를 참고한 잠정값이다.** 골든셋이 없어 실측되지 않았다.
- **정기 스케줄 트리거가 없다.** 필요해지면 방역 리마인더봇 패턴을 참고해 추가 검토.
- **review_result는 가벼운 human_gate 패턴만 쓴다.** 방역의 문서승인처럼 버저닝/잠금(approval_block)까지는 아직 필요하다고 판단하지 않았음 — 필요해지면 승격 검토.
- 미결정 사항 전체는 manifest.yaml의 `open_questions` 참고.

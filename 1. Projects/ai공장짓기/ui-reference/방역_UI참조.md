# 방역 UI 참조 #방역

근거 문서: `1. Projects/클로드 방역 ai/방역&클린 모바일 AI 운영관리 MVP 계획안.md`. 이 문서는 그 계획안에 있는 기능 블록·화면 흐름·필드 그룹을 실제 화면 구성으로 옮긴 제안입니다. 공통 규칙(색상/승인카드/모바일 우선)은 `공통_패턴.md`를 그대로 따릅니다.

## 이 앱의 성격

방역은 4개 공장 중 가장 "현장형"입니다. 담당자가 폰을 들고 현장에서 입력하고, 대표자가 나중에(또는 그 자리에서) 승인하는 구조입니다. 그래서 화면 설계의 우선순위는 예쁨보다 **오프라인에서도 안 죽는 것, 사진 첨부가 쉬운 것, 실수로 승인 없이 발송되지 않는 것** 세 가지입니다.

## 화면 목록 (계획안의 "최소 기능 블록" 기준)

| 화면 | 목적 | 사용자 |
|---|---|---|
| ① 홈/오늘일정 | 오늘 방문 예정 현장 목록, D-day 리마인더 | 현장입력자 |
| ② 오더메모 입력 | 신규 문의를 구조화된 메모로 저장 | 현장입력자/대표자 |
| ③ 고객DB 상세 | 고객·현장·방역이력·문서이력 조회 | 대표자 |
| ④ 현장 입력 | 방문 체크인, 사용 약품, 처리내용, 사진 첨부 | 현장입력자 |
| ⑤ 문서 초안(완료보고서/소독증명서) | AI가 만든 초안 확인, 누락 경고 확인 | 현장입력자→대표자 |
| ⑥ 대표자 검수(승인 카드) | 승인/반려, 반려사유 입력 | 대표자 |
| ⑦ 보관함 | 승인·잠금된 문서 목록, 검색 | 대표자 |
| ⑧ 사후관리 | 만족도·불편사항 입력 | 현장입력자/대표자 |

## 화면별 구성 제안

### ① 홈/오늘일정

- 상단: 오늘 날짜 + 오늘 방문 예정 건수
- 카드 리스트(세로): 현장명, 고객명, D-day 배지(`--accent` 임박, `--fail` 당일 초과), 최근 방역이력 한 줄 요약
- 카드를 누르면 ④ 현장 입력으로 바로 진입
- 하단 고정 버튼: [+ 오더메모 입력] (`--primary`)

### ② 오더메모 입력

- 단일 화면 폼(마법사형 아님 — 항목이 적어서 한 화면이면 충분): 고객명, 연락처, 현장명/주소, 요청사항(자유 텍스트), 문서요청 여부(완료보고서/소독증명서 체크박스)
- 자유 텍스트 입력칸은 크게(현장 통화 내용을 그대로 붙여넣는 경우가 많음을 고려)
- 저장 즉시 AI가 구조화한 결과를 아래에 미리보기로 보여주고 "이대로 저장" 확인 받음(AI 판단을 사람이 확인 후 저장 — 헌장 원칙)

### ③ 고객DB 상세

- 상단: 고객명, 연락처, 현장 주소 (고정)
- 탭 대신 세로 스크롤 섹션: 방역이력(방문일·사용약품·문서상태 리스트), 다음 방문 예정(D-day), 문서 보관함 링크, 만족도 이력
- 조사 근거: 현장서비스 앱의 상세화면은 탭 뒤에 정보를 숨기지 않고 스크롤로 다 보여주는 구조가 표준적이라는 패턴 조사 결과를 반영(체크리스트/이력을 탭 안에 숨기지 않음)

### ④ 현장 입력

- 최상단에 카메라 버튼 크게 배치(작업 전/후 사진) — 메뉴 안에 숨기지 않음
- 사용 약품(드롭다운 또는 최근 사용 이력에서 선택), 처리 구역, 주요 작업내용(텍스트)
- 오프라인 상태면 화면 상단에 회색 배지 "오프라인 - 임시저장됨" 표시, 네트워크 복구 시 배지가 자동으로 "동기화 완료"(`--ok`)로 바뀜
- 저장 버튼은 오프라인에서도 항상 활성화(로컬 저장이 먼저, 서버 전송은 나중)

### ⑤ 문서 초안 화면

- 문서 종류 탭(완료보고서/소독증명서) — 이 화면만 예외적으로 탭 사용 가능(문서 두 종류가 서로 독립적인 문서라서 섞어 보여주면 오히려 헷갈림)
- AI가 채운 필드는 배경을 살짝 다른 톤(`--card-hover`)으로 구분해서 "AI가 채움"을 표시
- 누락 필드는 `--warn` 색 테두리 + "확인 필요" 라벨
- 하단: [검수 요청] 버튼 — 누르면 ⑥ 대표자 검수로 상태 전환 (누락 필드가 있으면 이 버튼이 비활성화되고 안내 문구 표시)

### ⑥ 대표자 검수 (승인 카드) — 공통_패턴.md의 승인 카드 그대로 적용

- 머리말: "완료보고서 - 해오름카페 (2026-07-17)"
- 요약: 현장명/방문일/사용약품/작성자/누락 항목 점검 결과(0건이면 초록 체크)
- [전체 문서 보기] 펼치기
- 버튼: [승인] `--ok`, [반려] `--fail` (반려 누르면 사유 입력 필수), [보류]는 만들지 않음(회사 헌장상 승인/반려만 있고 애매한 상태를 남기지 않는 게 원칙)
- 승인 시: 문서가 잠금 상태로 전환되고 카드에 자물쇠 아이콘 표시. 이 시점부터 [발송]/[출력] 버튼이 비로소 활성화됨.

### ⑦ 보관함

- 리스트(카드 아님, 표 형태 허용 — 데스크톱에서 대표자가 검색 목적으로 쓸 가능성 높음): 문서번호, 고객명, 승인일자, 문서종류, 상태(전부 "승인완료·잠금" 상태만 존재)
- 검색창(고객명/현장명), 기간 필터

### ⑧ 사후관리

- 별 5점 만족도 입력, 불편사항 자유텍스트, 재방문 필요 체크박스
- 체크 시 자동으로 홈 화면 리마인더 목록에 추가

## 색상/상태 적용 요약 (방역 전용)

| 상태 | 화면 | 색 |
|---|---|---|
| D-day 임박(3일 이내) | 홈 | `--accent` |
| D-day 초과 | 홈 | `--fail` |
| 필수항목 누락 | 문서 초안 | `--warn` |
| 대표자 승인완료·잠금 | 검수/보관함 | `--ok` |
| 반려 | 검수 | `--fail` |
| 오프라인 임시저장 | 현장입력 | `--text-dim` |

## 아직 확정 안 된 부분 (계획안의 "스키마 관련 유보 사항" 그대로 승계)

고객/현장/방문 엔터티 분리 여부, 약품 마스터 테이블 여부, 문서 버전 관리 방식, 오프라인 동기화 충돌 정책은 계획안에서 "unspecified—require confirmation"으로 남아 있습니다. 이 UI 참조 문서도 그 부분을 화면 레이아웃 수준에서만 다루고 데이터 구조를 임의로 확정하지 않았습니다. 실제 구현 시 이 부분은 먼저 확인이 필요합니다.

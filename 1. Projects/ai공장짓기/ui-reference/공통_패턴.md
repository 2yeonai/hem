# 공통 UI 패턴 #플랫폼

이 문서는 공장이 달라도 항상 똑같이 적용해야 하는 규칙만 모아둔 것입니다. 공장별 화면 구성은 각 `OOO_UI참조.md`에 따로 있고, 색상 코드 등 세부 값은 `1. Projects/ai공장짓기/UI_기획서.md`(2026-07-12 혜미 확정본)를 그대로 씁니다. 여기서는 값을 새로 정하지 않고, "언제 어떤 값을 쓰는가"를 정리합니다.

## 1. 디자인 토큰은 이미 확정돼 있다 — 재사용만 한다

`UI_기획서.md`에서 이미 다크 글래스 테마로 확정됐습니다 (2026-07-12, 혜미 선택). 요약:

- 배경 `--bg:#0f1220`, 카드 `--card:#1a1f35`, 카드 호버 `--card-hover:#232a4a`
- 본문 글자 `--text:#e8ecff`, 흐린 글자(부가 정보) `--text-dim:#9aa3c7`
- 강조색 `--accent:#7fd4ff`, 주요 버튼 `--primary:#2f6fed`
- 상태색: 통과/성공 `--ok:#3a9d5f`, 경고 `--warn:#d9a53a`, 실패/반려 `--fail:#e05a5a`
- 폰트: Pretendard, 본문 15px/줄간격 1.7, 제목 22px 굵게
- 카드 모서리 14px, 버튼 높이 40px, 모바일(640px 이하)에서는 카드 1열 + 버튼 전체폭

새 공장 앱을 만들 때 이 값을 다시 고민하거나 새 색을 추가하지 않습니다. 색이 부족하다고 느껴지면 이 문서를 고치는 게 아니라 `UI_기획서.md` 원본을 먼저 검토하고 혜미에게 확인받습니다(디자인 토큰은 "Fable 확정" 문서라 임의 변경 금지).

## 2. 상태 색상 사용 규칙 (4개 공장 공통)

방역·꽃집·정부지원·콘텐츠 전부 "초안 → 검수 → 승인/반려 → 확정" 흐름을 공유합니다. 화면에서 상태를 보여줄 때는 항상 아래 매핑을 따릅니다.

| 상태 | 색상 토큰 | 의미 | 공장별 예시 |
|---|---|---|---|
| 초안/대기 | `--text-dim` (회색) | 아직 사람이 안 본 상태 | 꽃집: 검수 전 주문 / 콘텐츠: draft |
| 진행중/검토중 | `--accent` (하늘색, 점멸 가능) | 지금 처리되고 있음 | 방역: AI 초안 생성 중 / 정부지원: 매칭 계산 중 |
| 승인/통과 | `--ok` (초록) | 사람이 확정함, 잠김 | 방역: 대표자 승인 완료 / 콘텐츠: 게시 승인 |
| 주의/확인필요 | `--warn` (주황) | 필수항목 누락, 확인 필요한 값 | 꽃집: 확인 필요 필드 있음 / 정부지원: 마감 임박 |
| 반려/실패 | `--fail` (빨강) | 되돌아가야 함 | 방역: 대표자 반려 / 콘텐츠: 게시 실패 |

우선순위 배지(꽃집 review_manager_bot의 초록/노랑/빨강/파랑 등 공장별 자체 우선순위 체계가 있는 경우)는 위 5색과 억지로 통일하지 않아도 되지만, 최소한 "승인=초록, 반려=빨강"의 의미는 절대 바꾸지 않습니다. 색을 헷갈리게 쓰면 현장에서 실수로 반려 건을 승인 건으로 착각할 위험이 있기 때문입니다.

## 3. 승인 카드 공통 패턴 (approval card)

4개 공장 모두 "AI가 초안을 만들고 → 사람이 승인/반려한다"는 구조를 갖고 있습니다(회사 헌장의 human 게이트 원칙). 웹 검색 조사 결과, 승인/반려 UI는 업계에서도 대체로 아래 구조로 수렴합니다 — 리뷰 대상이 무엇인지 한눈에 보여주는 카드, 명확한 승인/반려 버튼, 반려 시 사유를 반드시 남기게 하는 것, 반려된 건은 처음부터 다시 하지 않고 맥락과 함께 되돌리는 것.

권장 카드 구성 (위→아래 순서):

1. **머리말**: 무엇에 대한 승인인지 한 줄 (예: "김OO님 방문 완료보고서", "박혜미 주문 #0231")
2. **핵심 정보 요약**: 사람이 승인 여부를 판단하는 데 필요한 값만 3~6개 (전체 데이터를 다 펼치지 않음)
3. **AI 신뢰도/우선순위 배지**: 있는 경우 색상 배지로 표시 (예: 꽃집의 초록/노랑/빨강/파랑)
4. **원문/상세 보기 링크 또는 펼치기**: 필요할 때만 펼쳐서 전체 내용 확인
5. **행동 버튼 2~3개**: [승인/확정] `--ok` 색, [반려/수정요청] `--fail` 색, 필요시 [임시저장] `--text-dim` 톤. 버튼은 최소 2개(승인/반려)이며, 반려를 누르면 사유 입력칸이 필수로 뜬다.
6. **잠금 표시**: 승인 완료된 건은 카드 자체가 "잠김" 상태로 바뀌어(자물쇠 아이콘 등) 재수정이 기본적으로 막혀 있음을 보여준다.

이 카드 패턴은 특히 방역의 대표자 검수, 꽃집의 human_reviewer, 정부지원의 문서승인, 콘텐츠의 게시승인에 공통으로 씁니다. 회사 헌장 원칙("승인 전 절대 발송 금지")이 화면에도 보이도록, 승인 전에는 [발송]/[게시]/[출력] 버튼을 아예 비활성화(회색, 클릭 불가) 상태로 둡니다.

## 4. 한글 타이포그래피·표기 규칙

- 본문 15px, 줄간격 1.7 — 한글은 영문보다 줄간격이 좁으면 답답해 보이므로 이 값을 줄이지 않는다.
- 날짜 표기는 `2026-07-17` 형식(ISO)으로 통일. "7월 17일" 같은 자연어 표기는 알림 문구에서만 쓰고 데이터 표시 영역에서는 쓰지 않는다.
- 금액은 천단위 콤마 + "원" (예: `100,000원`), 소수점 없음.
- 상태값은 한글 명사형으로 통일: "승인완료", "승인대기", "반려", "초안" — "승인됨/승인함" 같은 동사형 혼용 금지(방역 문서에서 실제로 있었던 표기 불일치 사례, 재발 방지).
- 버튼 문구는 동사+명사 짧은 조합: "검수저장", "주문확정", "출력준비" (꽃집 예시) — "저장하기", "저장을 눌러주세요" 같은 긴 존댓말 문구는 쓰지 않는다.

## 5. 모바일 우선 원칙

방역과 꽃집은 현장/카운터에서 폰으로 쓰는 게 기본이므로 모바일 우선(mobile-first)으로 설계합니다. 정부지원과 콘텐츠는 데스크톱에서 쓰는 비중이 높지만, 그래도 아래 규칙은 4개 공장 공통으로 유지합니다.

- 640px 이하 화면에서는 카드가 1열로 쌓이고, 버튼은 화면 전체 폭을 채운다 (엄지로 누르기 쉽게).
- 사진 첨부가 필요한 화면(방역 현장사진, 꽃집 배송사진)은 카메라 바로 호출 버튼을 최상단 근처에 둔다 — 메뉴 깊숙이 숨기지 않는다.
- 오프라인 대응이 필요한 화면(방역 현장입력)은 "임시저장" 상태를 명확한 배지로 보여주고, 동기화 전에는 승인 화면에 노출하지 않는다.
- 리스트/큐 화면(승인 대기 목록 등)은 위에서 아래로 스크롤만으로 훑을 수 있게 하고, 가로 스크롤이나 여러 열 테이블은 모바일 화면에서 쓰지 않는다.

## 6. 조사 근거 요약 (2026-07-17)

웹 검색으로 조사한 일반적 UI 패턴 자료 중 이 문서에 반영한 것들:

- 승인/반려 워크플로우는 "무엇을 승인해야 하는지 보여주는 카드 + 명확한 승인/수정 액션 + 반려 시 사유 필수"로 수렴한다는 공통 의견 (Cflow, AI UX Playground 등).
- 현장 서비스 앱의 작업 상세 화면은 개요-작업목록(체크리스트)-자산-이력-문서-완료(서명/사진) 구조가 표준적이며, 체크리스트는 탭 뒤에 숨기지 않는 것이 권장된다는 조사 결과 (Team 400 field service 아티클).
- 주문/운영 대시보드는 카드형 레이아웃, 부드러운 카드 테두리, 절제된 강조색, 명확한 타이포그래피로 "한눈에 훑기"를 우선시하는 경향 (Design4Users, Muzli 대시보드 갤러리).
- 콘텐츠 캘린더는 날짜/시간/플랫폼/본문/자산/링크를 최소 필드로 하고 상태를 draft→검토중→승인→예약→게시 단계로 명시하는 패턴이 일반적 (Sprinklr, Hootsuite 자료).
- 정부지원 등 복잡하고 드물게 쓰는 신청 절차는 마법사(wizard)형 단계별 입력이 오류를 줄인다는 공통 권고 (NN/g, Eleken 등).

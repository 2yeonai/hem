# UI 참조 가이드 #플랫폼

## 이 폴더가 왜 있는가

혜미 님은 개발자가 아니라서, 새 공장(방역/꽃집/정부지원/콘텐츠) 앱을 만들 때마다 "화면을 어떻게 생기게 할지"를 처음부터 다시 정하는 건 비효율적입니다. 이 폴더(`1. Projects/ai공장짓기/ui-reference/`)는 **"앱을 만들 때마다 자동으로 참조하는 UI 사전"** 역할을 합니다. 새 세션에서 어떤 공장이든 앱 화면을 만들거나 고칠 일이 생기면, 담당 Claude는 이 폴더의 관련 문서를 먼저 읽고 그 안의 패턴을 그대로 따르면 됩니다 — 매번 "예쁜 UI가 뭘까"를 새로 판단하지 않아도 되게 하는 게 목적입니다.

**중요**: 이 폴더는 `1. Projects/ai공장짓기/UI_기획서.md`(2026-07-12, 혜미 확정: 다크 글래스 테마 디자인 토큰)를 대체하지 않습니다. `UI_기획서.md`는 **색·폰트·버튼 같은 확정된 디자인 토큰**(코드 값)을 담은 상위 문서이고, 이 폴더는 그 토큰을 **어떤 화면 구성(레이아웃·컴포넌트 종류)에 적용할지**를 공장별로 구체화한 실무 참조서입니다. 둘 다 필요하면 같이 읽으세요.

## 폴더 구조

```
ui-reference/
├── UI_참조_가이드.md      ← 지금 이 문서 (총 안내, 여기서 시작)
├── 공통_패턴.md            ← 4개 공장 전부에 적용되는 공통 규칙 (상태색, 승인카드, 모바일 우선 등)
├── 방역_UI참조.md          ← 방역&클린 모바일 운영관리 앱 화면 패턴
├── 꽃집_UI참조.md          ← 꽃집(온천꽃식물원) 주문 파이프라인 앱 화면 패턴 (human_reviewer 승인화면 와이어프레임 포함)
├── 정부지원_UI참조.md      ← 정부지원사업 매칭·초안·발표자료 앱 화면 패턴
└── 콘텐츠_UI참조.md        ← 콘텐츠(마시땅/이연/네이버블로그) 콘텐츠 캘린더 앱 화면 패턴
```

## 사용법 — 새 앱 화면을 만들 때 이 순서로 읽는다

1. **`공통_패턴.md`을 먼저 읽는다.** 상태 색상(승인/대기/반려), 승인 카드 공통 모양, 한글 타이포그래피 규칙, 모바일 우선 원칙이 여기 다 있습니다. 이걸 안 지키면 공장마다 앱이 제각각으로 보입니다.
2. **해당 공장의 `OOO_UI참조.md`를 읽는다.** 예: 꽃집 승인 화면을 만든다면 `꽃집_UI참조.md`를 읽습니다. 이 문서에는 그 공장에 필요한 화면 목록, 화면별 구성요소, 참고할 만한 실제 앱 패턴(모빈/드리블 등에서 조사한 일반적 패턴)이 정리돼 있습니다.
3. **`1. Projects/ai공장짓기/UI_기획서.md`의 디자인 토큰(색상 코드, 폰트, 버튼 규격)을 그대로 적용한다.** 새로운 색이나 폰트를 임의로 추가하지 않습니다.
4. 화면을 다 만들고 나면, 원래 계획 문서(예: 방역은 `방역&클린 모바일 AI 운영관리 MVP 계획안.md`, 꽃집은 `HANDOFF.md`)에 실제 구현 여부를 갱신합니다.

## 왜 "자동 참조"가 되는가

이 폴더의 파일명과 배치 자체가 규칙입니다 — 공장 이름 그대로 파일명을 지었기 때문에, "꽃집 앱 화면 만들어줘" 같은 요청을 받으면 담당 Claude는 벡터 검색이나 파일명 패턴 매칭만으로 `꽃집_UI참조.md`를 바로 찾을 수 있습니다. 새 공장(5번째 공장 등)이 추가되면 같은 규칙으로 `OOO_UI참조.md`를 이 폴더에 추가하고, 이 가이드의 "폴더 구조" 표에 한 줄만 추가하면 됩니다.

## 리서치 방법과 한계 (2026-07-17 작성 근거)

이 폴더의 화면 패턴 제안은 실제로 모빈(mobbin.com)·유아이볼(uibowl.com)·핀터레스트 화면을 직접 캡처해서 만든 게 아닙니다. **브라우저 접근 없이 웹 검색으로 "이런 종류의 앱에는 보통 이런 화면 구성 요소가 있다"는 글·아티클을 찾아 일반적인 패턴만 종합**한 것입니다. 즉 특정 앱을 그대로 베낀 디자인이 아니라, "현장 서비스 앱은 보통 체크리스트+사진첨부+일정 화면이 있다"는 수준의 일반 원칙입니다. 실제 화면 디자인(픽셀 단위)은 여전히 `UI_기획서.md`의 디자인 토큰과 구현 담당자의 판단으로 확정해야 합니다.

# 꽃집 UI 참조 #꽃집

근거 문서: `1. Projects/클로드 꽃집 ai/HANDOFF.md`, `12봇_kind분류표.yaml`. 지금 4개 공장 중 **승인 화면이 하나도 없는 유일한 공장**이 꽃집입니다 — 14단계 파이프라인 중 local 7개·model 6개는 이미 규칙기반으로 구현됐는데, kind:human인 `human_reviewer` 단계만 화면 없이 `run_pipeline.py`가 자동으로 통과시키는 목업 상태입니다. 그래서 이 문서는 "지금 당장 필요한" 화면을 가장 구체적으로 다룹니다.

## human_reviewer 승인화면 — 와이어프레임 (가장 급함)

### 이 화면이 받는 데이터 (review_manager_bot이 만들어 넘겨줌)

`12봇_kind분류표.yaml`의 shared_context 기준, human_reviewer가 읽는 값은 `review_checklist`, `review_priority`(초록/노랑/빨강/파랑), `editable_fields`이고, 그 안에는 상류 단계에서 채워진 아래 필드들이 들어 있습니다.

| 필드 | 출처 | 비고 |
|---|---|---|
| recipient_org / recipient_title | order_draft_bot | 받는사람 기관·직함 (이름은 의도적으로 미추출 — 아래 "확인 필요" 참고) |
| sender_org / sender_title | order_draft_bot | 보내는사람 기관·직함 (이름 동일하게 미추출) |
| ribbon_message_final | ribbon_price_bot | 리본 문구 (예: "승진을 축하드립니다") |
| product_name / price / quantity | ribbon_price_bot | 상품명·가격·수량 |
| review_priority | review_manager_bot | 초록/노랑/빨강/파랑 우선순위 색 — **매핑 기준 자체가 미확정 제안값**(HANDOFF §7 참고) |
| review_checklist | review_manager_bot | 확인해야 할 항목 리스트(예: "받는사람 이름 확인 필요") |

**주의(그대로 화면에 반영해야 함)**: recipient_name/sender_name(받는 분·보내는 분 "이름")은 order_draft_bot이 의도적으로 추출을 시도하지 않습니다(golden_set 001에서 실제 이름이 뒤바뀐 사례가 있어서). 따라서 승인화면은 이름 필드를 **빈칸 + "확인 필요" 배지**로 항상 보여줘야 하고, 사람이 직접 채워 넣는 입력칸을 반드시 제공해야 합니다. 또한 **배송지 주소/연락처는 지금 스키마에 별도 필드로 명시돼 있지 않습니다** — order_draft의 org/title 정보로 대략 유추만 가능한 상태이니, 화면에는 "배송지 확인 필요" 자유입력칸을 넣고 이 구조 자체를 다음 세션에서 스키마 담당자와 확인하는 것을 권장합니다(임의로 주소 필드명을 만들어 구현하지 않기).

### 화면 레이아웃 (세로 스크롤 1화면, 모바일/데스크톱 겸용)

```
┌──────────────────────────────────────────┐
│ ← 뒤로       주문 검수                     │
│                                            │
│  [●빨강] 확인 필요 3건        주문 #0231   │  ← review_priority 배지 + 체크리스트 건수
├──────────────────────────────────────────┤
│  받는 분 (recipient)                       │
│  ┌──────────────────────────────────────┐│
│  │ 기관/직함: 북면사무소 면장   [AI추정]  ││  ← order_draft_bot 값, 배경 톤 다르게
│  │ 이름: [________________] ⚠확인필요    ││  ← 빈칸, 사람이 직접 입력
│  │ 배송지 주소: [________________] ⚠확인필요││ ← 빈칸, 자유입력
│  │ 연락처: [________________] ⚠확인필요   ││
│  └──────────────────────────────────────┘│
├──────────────────────────────────────────┤
│  보내는 분 (sender)                        │
│  ┌──────────────────────────────────────┐│
│  │ 기관/직함: 부곡면사무소 면장 [AI추정]  ││
│  │ 이름: [________________] ⚠확인필요    ││
│  └──────────────────────────────────────┘│
├──────────────────────────────────────────┤
│  리본 문구 / 상품 / 가격                    │
│  ┌──────────────────────────────────────┐│
│  │ 리본 문구: "승진을 축하드립니다"  [AI]  ││  ← 수정 가능한 텍스트칸
│  │ 상품명: 축하화환 3단          [AI]     ││
│  │ 수량: 1개                     [AI]     ││
│  │ 가격: 100,000원               [AI]     ││
│  └──────────────────────────────────────┘│
├──────────────────────────────────────────┤
│  검수 체크리스트 (review_checklist)         │
│  ☐ 받는분 이름 확인                        │
│  ☐ 배송지 주소 확인                        │
│  ☐ 리본 문구 어투 확인                     │
├──────────────────────────────────────────┤
│  원문 보기 ▾ (통화/문자 원본 펼치기)        │
├──────────────────────────────────────────┤
│                                            │
│  [ 검수저장 ]   [ 주문확정 ]   [ 출력준비 ] │  ← approval_action 3가지 값 그대로 버튼화
│   (회색/보류)     (초록/ok)      (파랑/추가) │
│                                            │
│  ※ 확인필요 항목이 남아있으면 [주문확정]     │

```

### 버튼 3개 = approval_action 스키마 값 그대로

`manifest.yaml`의 `approval_action` 필드는 정확히 `검수저장 | 주문확정 | 출력준비` 세 값을 가지므로, 화면 버튼도 이 3개 그대로 씁니다 — 공통_패턴.md의 "승인/반려 2버튼" 기본형과 다르게, 꽃집은 반려 개념이 없고 대신 3단계 진행 버튼입니다(2026-07-16 기준 구조).

- **검수저장**: 지금까지 고친 내용만 저장, 아직 확정 아님. 색 `--text-dim` 톤(중립 회색 버튼).
- **주문확정**: `confirmed_fields`/`manual_edits`를 확정, 이후 storage_bot으로 넘어감. 색 `--ok`(초록). **체크리스트에 미확인 항목이 남아있으면 비활성화** — 헌장의 "승인 전 발송 금지" 원칙을 여기서도 지켜야 함.
- **출력준비**: 확정된 주문을 인쇄 준비 상태로 넘김(print_prep_bot으로). 색 `--primary`(파랑) — 주문확정 이후에만 눌릴 수 있게 순서 강제.

### review_priority 색상 배지 표시 규칙

review_priority는 초록/노랑/빨강/파랑 4색이지만 정확한 매핑 기준(신뢰도 수치→색)은 아직 미확정 제안값입니다(HANDOFF.md §7). 화면에서는 일단 이 4색을 있는 그대로 배지로 노출하되, 매핑 로직 자체를 화면이 재정의하지 않습니다 — 색이 이상하게 보이면 review_manager_bot.py의 규칙을 고치는 게 맞고, 화면 쪽에서 임의로 다른 색을 쓰지 않습니다.

## 이 화면 말고 나머지 파이프라인 관련 화면 (있으면 좋음, 급하지 않음)

| 화면 | 목적 | 우선순위 |
|---|---|---|
| 대기 큐 목록 | 검수 대기 중인 주문 카드 리스트(review_priority 색으로 정렬) | 승인화면 다음으로 중요 |
| 보관함/이력 | 확정·출력완료된 주문 검색 | 낮음 |
| 다중주문 묶음 보기 | bundle_id로 묶인 여러 주문(call_007 같은 사례)을 한 화면에서 순차 처리 | 낮음(order_split_bot이 아직 자동분리 못 하는 한계와 맞물림) |

### 대기 큐 목록 (제안)

```
검수 대기 (5건)
┌────────────────────────────┐
│ [●빨강] #0231 북면사무소 →  │
│ [●노랑] #0230 해오름카페 →  │
│ [●초록] #0229 청솔유치원 →  │
└────────────────────────────┘
```
카드를 누르면 바로 위 human_reviewer 승인화면으로 진입. 빨강 우선순위가 위로 정렬.

## 조사 근거

주문/승인 카드 패턴은 일반적인 주문관리 대시보드 조사에서 "카드형 레이아웃 + 부드러운 카드 테두리 + 상태 배지"가 표준적이라는 점, 승인 워크플로우 조사에서 "무엇을 승인하는지 요약 + 명확한 액션 버튼 + 반려/보류 시 사유·체크리스트 기록"이 공통 패턴이라는 점을 반영했습니다. 다만 꽃집은 반려 대신 3단계 진행 버튼(검수저장/주문확정/출력준비)이라는 스키마상 고유 구조가 있어, 일반적인 승인/반려 2버튼 패턴을 그대로 쓰지 않고 스키마에 맞춰 조정했습니다.

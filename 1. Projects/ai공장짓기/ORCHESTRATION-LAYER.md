# 상위 AI 오케스트레이션 계층 (설계서 보완)

설계서 v2의 §3 조직도·장기 원칙·실행본부 내부 오케스트라는 그대로 유지한다.
이 문서는 그 위에 빠져 있던 **제품 간(cross-product) 오케스트레이션 계층**만 추가한다.

---

## 1. 계층 구조

```
[관제층]  나 + routing-policy.md (판정표)
          └ 보조 관제: Claude Cowork  ← 패킷 생성·카드 취합·사용량 기록을 대행
                │
[프로토콜층]  TASK-PACKET.md (입력 규격) ←→ RESULT-CARD.md (출력 규격)
          ※ 제품 간에는 "대화"가 아니라 "패킷과 카드"만 오간다
                │
[실행 노드 5]
  ChatGPT      전략·기획·Deep Research·출처 비교·최종 종합/감사
  Claude Code  코드 실행 + 내부 서브에이전트 지휘 (실행본부 노드)
  Codex        병렬 구현·코드베이스 검증·교차 리뷰
  Claude Cowork 문서·파일·비개발 업무 실행 (+ 보조 관제 겸직)
  Gemini       Google 생태계·초장문·멀티모달 보조 조사·교차검증
                │
[기억층]  vault (ops/tasks/ decisions/ ops/usage.md)
          + git repo (코드 노드 간 공유 상태)
```

**이 계층이 4가지 낭비를 줄이는 방식:**

| 낭비 | 해결 장치 |
|---|---|
| 매번 "어디에 시킬까" 판단 비용 | routing-policy.md 판정표 — 판단을 조회로 바꾼다 (30초) |
| 같은 설명 반복(토큰 낭비) | TASK-PACKET — 자기완결 500단어, 원문은 경로로. 병렬 시 **비범위가 배타적인 파생 패킷**으로 분할해 중복 조사까지 차단 |
| 전달·취합 누락 | 모든 결과는 RESULT-CARD 강제, 작업 폴더의 파일명이 곧 상태(00_→1x_→2x_→90_→99_). 폴더만 봐도 누락이 보인다 |
| 구독 한도·강점 비효율 | usage.md 신호등(🟢🟡🔴) + 카드의 "사용량 체감" 란 + 정책표의 대체 규칙(R8) |

**핵심 설계 결정 2가지 — 역할이 겹치지 않게 나눈다.**
- **총괄(지적 종합) = ChatGPT.** 파생 패킷 분할, 카드 결론 비교·상충 판정, 90_FINAL 작성, 외부 감사.
- **사무국(파일 실무) = Claude Cowork.** 5개 노드 중 로컬 파일을 직접 읽고 쓰는 범용 노드는 Cowork뿐이므로, 사람이 하던 잡무를 위임한다: ① 한 문장 → 00_PACKET 생성 ② 회수된 카드의 저장·폴더 상태 관리 ③ usage.md 갱신·주간 vault 정리. 이로써 사람의 역할은 "붙여넣기와 판정"만 남는다.

※ 예외 — **AI OS 설계서 프로젝트 자체**에서는 총괄·구현이 Claude(현 Fable)이고 ChatGPT는 감사 3항목만 수행한다. 근거와 승계 규칙은 설계서 §18 거버넌스.

## 2. 자동화 가능 / 반자동 / 현재 불가능

| 구분 | 항목 |
|---|---|
| **자동 (사람 개입 0)** | Claude Code 내부의 모델·서브에이전트 전환(✅공식) · 코드 노드 간 공유 상태(git) · Cowork의 패킷 생성·카드 취합·파일 정리 (한 번 지시하면 폴더 기준으로 반복 수행) |
| **반자동 (조회·붙여넣기만 사람)** | 라우팅 판정(표 조회 — Cowork에게 "이 작업 판정해줘"로 위임 가능) · 패킷을 각 제품에 붙여넣기 · 카드 회수(각 제품에서 복사 → Cowork에 투하) · usage.md 신호등 갱신(주 1회 5분) |
| **현재 불가능 (설계로 우회 중)** | 제품 간 자동 호출/푸시(구독 결박+제3자 통합 금지) · 구독 잔여 한도의 프로그램 조회 · 제품 간 컨텍스트 자동 동기화(→ 패킷이 대체) · "한 화면" 통합 대시보드(→ usage.md가 대체) |

불가능 3종은 전부 벤더 정책 제약이지 설계 결함이 아니다. 정책이 바뀌면 프로토콜층(패킷·카드)은 그대로 두고 "붙여넣기"만 자동 전송으로 바뀐다 — 이 문서의 구조가 그날을 위한 준비다.

## 3. 격리된 최소 검증안 2건

### 3-1. 9Router (목적: 통합 대시보드 UX와 풍향계 관찰 — 그 이상 금지)

- **환경:** 별도 폴더, 메인 워크플로와 무접점. **메인 계정 OAuth 연결 절대 금지.**
- **재료:** 무료 Provider(일회용 계정) 1–2개만.
- **기간:** 2주 관찰 + 분기 재평가.
- **측정 (예/아니오로 답할 수 있게):**
  1. 쿼터 대시보드가 usage.md 수동 기록(주 5분)을 대체할 만큼 나은가?
  2. "자동 폴백" 체감이 정책표 조회(30초)보다 실질 이득이 있는가?
  3. Provider 목록에 열림/닫힘 변화가 있었는가? (벤더 정책 풍향계)
- **산출:** `vault/decisions/9router-실험-{날짜}.md` 3문항 답 + 유지/폐기 판정.

### 3-2. chatgpt2codex (목적: "ChatGPT 외부감사에게 로컬 눈 달아주기" 1개 시나리오만)

- **환경:** 테스트용 repo 1개만 연결(실 프로젝트 금지). Owner Token은 비밀번호급 관리.
- **시나리오 (단 1개):** Claude Code가 구현·커밋 → ChatGPT가 chatgpt2codex로 그 repo를 읽고 테스트 실행 + 스크린샷 → **RESULT-CARD 형식으로 반환하는지** 확인.
- **성공 기준:** 카드의 "근거" 란에 테스트 출력·스크린샷 증거가 들어오고, 내가 코드를 안 보고도 판정 가능하면 성공.
- **판정:** 성공 시 외부감사 절차에 정식 편입(월 사용 빈도 기록), 2회 연속 실패 또는 6개월 무업데이트 시 폐기.

## 4. 실제 사용 예시 3건 (전 과정)

### 예시 A — "꽃집 발렌타인 인스타 문구 10개" (비코딩·저난도)
1. 판정표 R6: 시작 ChatGPT(기본 티어), 병렬 없음, 검수 나.
2. 패킷 생략 가능(저난도 단발 — START-HERE §5 예외 규칙). ChatGPT에 바로 요청.
3. 결과가 좋으면 끝. 마음에 드는 문구는 `vault/prompts/`에 저장.
→ 오케스트레이션 비용 0. **모든 작업에 절차를 씌우지 않는 것도 규칙이다.**

### 예시 B — "정부지원사업 공고 3건 비교 후 지원서 초안" (대규모 조사+문서·중난도)
1. 판정표 R4+R5 복합: 시작 ChatGPT(Deep Research 티어), **병렬 Gemini**(공고 원문이 hwp/pdf 장문 — 초장문·멀티모달 강점), 종합·검수 ChatGPT.
2. Cowork에게: "이 문장으로 TASK-PACKET 만들어줘" → `vault/ops/tasks/2026-07-14_지원사업비교/00_PACKET.md` 생성.
3. 총괄 ChatGPT에게 파생 분할 지시(비범위 배타: GPT=일반 웹·종합 / Gemini=공고 원문·Google 소스) → `10_`, `13_` 파생 패킷을 각자에게 붙여넣기 (설명 반복 0).
4. 두 결과를 RESULT-CARD로 회수 → `2x_RESULT-*.md` 저장(Cowork 위임 가능).
5. 카드 2장을 ChatGPT에 주고 취합 절차 실행 → `90_FINAL.md`(상충 판정 포함) → 지원서 초안.
6. Cowork가 초안을 .docx로 정리, 나 승인(`99_APPROVED.md`). 재사용 결정은 decisions/로 승격.

### 예시 C — "꽃집 주문 폼 버그 수정 + 재발 방지" (코딩·중난도·실패위험 중)
1. 판정표 R1+R3: 시작 Claude Code(내부 오케스트라: planner→developer→tester→reviewer 자동), **교차 Codex**, 최종 승인 나.
2. 패킷 작성(버그 재현 절차가 "입력 자료"에 필수) → Claude Code에 투입.
3. Claude Code가 feat 브랜치에 수정 + 내부 리뷰 + RESULT-CARD 커밋.
4. Codex CLI를 같은 repo에서 열고 패킷+카드 전달: "이 수정을 검증하고 빠진 엣지케이스만 보고" → 두 번째 카드.
5. 카드 2장으로 나 판정 → merge. 재발 방지 규칙은 CLAUDE.md에 추가.

## 5. 이 계층의 유지보수

- 판정표·티어표가 현실과 어긋나면(새 모델, 한도 변경) **routing-policy.md만** 고친다. 다른 파일은 불변.
- 분기 재평가 때 이 문서의 §2 "불가능" 표를 다시 본다 — 벤더 정책 변화로 칸이 옮겨지는 것이 승격 신호다.
- 실전 배치 위치: `vault/` 루트에 START-HERE, `vault/prompts/`에 routing-policy, `vault/templates/`에 TASK-PACKET·RESULT-CARD, `vault/ops/`에 usage.md. (설계서 §9 폴더 구조와 동일)

## 0. 핵심 결론

1. **최종 구조는 "이중 두뇌 – 단일 손"** — 당신의 실제 방식(GPT 기획 / Claude Code 실행)을 공식 구조로 채택하고, Anthropic Orchestrator 패턴은 실행층 내부에서만 쓴다. 비교 분석은 §2.
2. **이 OS는 구독 전용으로 설계한다 (API 사용 계획 없음)** — 구독 통합·전환의 유일한 안전 경로는 공식 클라이언트다. Anthropic이 2026-02-19 문서로 "구독 OAuth의 제3자 사용 = 약관 위반"을 명시하고 04-04부터 차단 중이므로, 라우터 계층은 구독 목적으로 Core에 넣을 수 없다. 벤더 간 전환은 자동화 대신 "정책 라우팅 + vault 인터페이스 + 동일 repo 멀티 CLI"로 수동 전환 비용을 최소화한다. §6-4.
3. **Fable 5 의존 지점 0으로 설계** — 상시 구조는 Sonnet+Opus로 완결. 7/19까지의 무료 기간은 "한 번 하면 오래 남는 판단 작업"에만 집중 투입. §13.
4. **5년 생존의 열쇠는 등급제** — 코어(마크다운·Git·Obsidian·역할정의 텍스트)는 소유, 지능(모델)은 임대, 실험 기능은 격리. §1.
5. **부서 간 직접 호출 금지** — 모든 통신은 사람 또는 vault 파일 경유. 글루 코드 0이 비개발자 유지보수와 5년 생존의 조건이다.

## 1. 5년을 버티는 전제: 3등급 의존성 관리

AI 모델은 6개월마다 바뀐다. 5년을 버티는 것은 모델이 아니라 **구조와 데이터**다.

| 등급 | 구성요소 | 방침 |
|---|---|---|
| **코어 (5년+)** | 마크다운 파일, Obsidian vault, Git/GitHub, 폴더 구조, CLAUDE.md·PRD·에이전트 정의 등 "역할 텍스트" | 소유한다. 여기에만 투자 축적 |
| **교체 부품** | ChatGPT, Claude, Gemini, Codex, Claude Code, n8n, 라우터 | 임대한다. 언제든 갈아끼움 |
| **실험 계층** | Advisor(⚠️실험적), chatgpt2codex, Agent Teams, 9Router류 | 격리한다. 죽어도 본체 무손상 |

원칙: **지능은 임대하고, 기억·규칙·판단의 결과는 소유한다.** 인터페이스는 5년 뒤에도 살아있을 확률이 가장 높은 두 표준만 쓴다 — ① 파일(마크다운), ② OpenAI 호환 API.

## 2. 구조 비교: Anthropic 패턴 vs 내 작업 방식

### 구조 A — Anthropic Orchestrator 패턴 그대로 (Claude Code가 기획+실행 전부)

```
나 → Claude Code(기획·분해·실행·검증) → 서브에이전트들
```

| 장점 | 단점 |
|---|---|
| 단일 허브, 유지보수 최소 | 기획과 실행이 같은 벤더 → 같은 계열 모델의 편향을 아무도 못 잡는다 |
| 공식 기능만으로 완결 (서브에이전트·훅·권한) | 이미 결제한 ChatGPT Business ×2가 논다 |
| 복붙 없음, 컨텍스트 일관성 최고 | Claude 한도에 전부 집중 → 계정 축소 시 타격 최대 |
| — | 당신의 실제 사고 습관(GPT로 발상)과 어긋남 → 도구가 습관을 못 이긴다 |

### 구조 B — 내 실제 방식 (GPT 전략·기획 / Claude Code 실행)

```
나 ↔ ChatGPT(전략·PRD) → [전달] → Claude Code(실행) → 결과 → ChatGPT(외부 감사)
```

| 장점 | 단점 |
|---|---|
| **기획 두뇌 ≠ 실행 두뇌** → 상관 오류가 구조적으로 줄어든다. "작성자≠검토자" 원칙의 상위 버전 | GPT가 코드 현실을 못 본다 → 기획-코드 괴리 가능 |
| ChatGPT ×2 구독 최대 활용, Claude 한도 절약 | 전달이 수동(복붙) → 마찰 |
| Claude 1계정으로 줄어도 기획 축 무손상 | 전달 형식이 매번 다르면 실행 품질 불안정 |
| 실제 습관과 일치 → 계속 쓰게 된다 | 완전 자동화 불가 구간 존재 |

### 판정: B 채택 + A는 실행층 내부에서 사용

**"이중 두뇌 – 단일 손"**: B가 회사의 조직도, A는 손의 해부학이다. B의 3가지 단점은 각각 장치로 보정한다.

1. **GPT가 코드를 못 본다** → ① 실행 후 doc-writer가 만드는 **결과 카드**(변경점·결정·미해결)를 GPT에 회신 ② 필요 시 chatgpt2codex로 GPT가 직접 로컬 테스트·스크린샷 검증(읽기·검증 한정).
2. **복붙 마찰** → 전달 통로를 vault로 단일화. GPT 산출물은 `inbox/`, 실행 결과는 작업 폴더(`ops/tasks/`)의 RESULT-CARD. 파일이 곧 프로토콜.
3. **형식 불안정** → **PRD 템플릿 고정**(§9). GPT가 항상 같은 틀로 뽑으면 planner가 기계적으로 받는다. 이 템플릿이 두 두뇌 사이의 계약서다.

## 3. 최종 조직도 — 전 도구 포함

```
                        ┌──────────────────────────┐
                        │    나 (CEO — 승인·방향)     │
                        └────────────┬─────────────┘
        ┌────────────────────────────┼────────────────────────────┐
        │                            │                            │
┌───────▼──────────┐      ┌─────────▼──────────┐      ┌──────────▼───────┐
│ 전략실 ChatGPT     │      │ 실행본부 Claude Code │      │ 총무부 n8n        │
│ 사업전략·기획·PRD   │      │ (Orchestrator)      │      │ 공고수집·알림      │
│ 사업계획서·마케팅    │      │  ├ planner          │      │ 꽃집 웹훅         │
└───────┬──────────┘      │  ├ researcher       │      │ 백업 트리거        │
        │                 │  ├ developer─tester  │      └──────────┬───────┘
┌───────▼──────────┐      │  ├ reviewer         │                 │
│ 조사국 Gemini      │      │  ├ doc-writer       │                 │
│ 대량조사·멀티모달   │      │  └ knowledge        │                 │
│ 초장문 분석        │      └───┬─────────┬───────┘                 │
└───────┬──────────┘          │         │ MCP(커넥터 표준)          │
        │               ┌─────▼───┐  ┌──▼───────────┐             │
┌───────▼──────────┐    │ GitHub  │  │ (예약석) 라우터 │◄────────────┘
│ 외부감사실         │    │ 원장:    │  │ 벤더 정책 변화   │  현재는 빈 자리
│ ChatGPT +        │    │ 코드·PR  │  │ 시에만 (§6-4)  │
│ chatgpt2codex    │    │ ·이력    │  └──────────────┘
│ Codex(2차 구현)   │    └────┬────┘
└──────────────────┘         │
                    ┌────────▼──────────┐
                    │ 기록보관소 Obsidian  │ ← 모든 부서의 공용 기억이자
                    │ inbox/PRD/reports  │    부서 간 공식 문서 통로
                    │ decisions/prompts  │
                    └───────────────────┘
```

**표준 업무 사이클 (누가 누구를 호출하는가):**

1. 나+ChatGPT: 전략 대화 → PRD 작성 → `vault/inbox/`
2. Claude Code planner: PRD 읽고 작업 분해 → 나에게 계획 승인 요청
3. 승인 후 researcher/developer 실행 (GitHub feat 브랜치, developer는 tester를 중첩 호출)
4. reviewer 리뷰(읽기 전용) → doc-writer가 RESULT-CARD 작성 → 작업 폴더(`vault/ops/tasks/…/2x_`) + PR
5. 나: PR 확인. 중요 건은 결과 카드를 ChatGPT(외부감사)에 전달, 필요 시 chatgpt2codex로 직접 검증
6. **merge·배포·삭제는 나만.** knowledge가 결정·교훈을 `vault/decisions/`에 적립
7. n8n: 사이클 밖에서 수집·알림·백업만 반복 (판단 없음)

**호출 규칙:** 부서 간 직접 호출 금지. 통신은 ① 사람 경유 ② vault 파일 경유 둘뿐. 실행본부 내부만 위→아래 호출 허용(수평 호출 금지, 중첩은 developer→tester 한 곳).

## 6. 라우터: 9Router 기능 구분 검토, 자체 구축, LiteLLM 재평가

(2026-07-14 재검토. 9Router README 직접 확인 + Anthropic 정책 변경 뉴스 검증 반영)

### 6-1. 9Router 기능의 정확한 구분

9Router는 "유료 API 키 라우터"가 아니라 3종의 연결 방식을 모두 제공하는 것이 맞다. 구분하면:

**① 별도 API 키 없이 구독 계정 OAuth만으로 가능한 것 (기술적으로)**
- Claude Code(Pro/Max), Codex(Plus/Pro), GitHub Copilot, Cursor 구독을 OAuth로 연결
- 쿼터 추적(5시간/주간 리셋), 자동 토큰 갱신, 계정별 라운드로빈
- 그 구독 쿼터를 다른 도구(Cursor에서 Claude 구독, n8n에서 Codex 구독 등)에서 사용
- ※ 단, Claude 경로는 아래 6-2에서 보듯 현재 **약관 금지 + 서버 차단 대상**

**② 무료 Provider로 가능한 것 (기술적으로)**
- Kiro AI(AWS Builder ID 등 OAuth): Claude 4.5·GLM-5·MiniMax "무제한" 주장
- OpenCode Free(무인증 패스스루), Vertex AI(신규 GCP $300 크레딧)
- ※ README 스스로 기록한 사실: iFlow·Qwen 무료 티어는 1년 새 종료, Gemini CLI 경로는 "계정 밴 가능성" 경고. 무료 경로의 수명은 남의 결정에 달려 있다.

**③ API 키가 필요한 것**
- 40+ 프로바이더(OpenRouter, GLM, MiniMax, Kimi, OpenAI, Anthropic, Gemini, DeepSeek…) 라우팅
- 3단 폴백(구독→저가→무료), 예산 상한

**연결 방식과 무관한 라우터 자체 기능:** RTK 압축(입력 20–40% 절감 주장), 포맷 변환(OpenAI↔Claude↔Gemini), 요청 로깅, 사용량 분석 — 이것들은 ①②③ 어느 경로에서든 동작한다.

### 6-2. 기술적 가능 vs 약관상 안전 (핵심 구분)

| 경로 | 기술적으로 가능? | 약관상 안전 확인? |
|---|---|---|
| Claude 구독 OAuth → 제3자 라우터 | 과거 가능, **현재 서버 차단 진행** | ❌ **명시적 금지.** Anthropic이 2026-02-19 문서로 "Free/Pro/Max OAuth 토큰의 타 제품 사용은 소비자 약관 위반" 명시, 2026-04-04부터 구독의 제3자 도구 사용 차단 시행 |
| Codex/Copilot/Cursor 구독 OAuth → 라우터 | ✅ README 기준 제공 중 | ⚠️ **확인 안 됨.** 각사 약관은 공식 클라이언트 사용을 전제. 명시적 허용 문서 없음. Anthropic 선례처럼 언제든 금지·차단될 수 있는 구조 |
| 무료 Provider(Kiro 등) 백엔드를 라우터로 사용 | ✅ 제공 중 | ⚠️ **확인 안 됨.** 해당 서비스(예: AWS Kiro)의 백엔드를 지정 클라이언트 밖에서 쓰는 것의 허용 여부 불명. 지속성도 불안정(②의 종료 사례) |
| API 키 → 라우터 | ✅ | ✅ **안전.** API 키는 프로그램적 접근용으로 판매되는 상품이라 프록시 경유가 약관과 충돌하지 않음 |

즉 이전 판의 "구독 OAuth 통과는 리스크"라는 추정은, Claude에 한해서는 추정이 아니라 **확인된 금지 사실**로 격상됐다. 나머지 경로는 "기술적으로 되지만 안전이 확인된 바 없음"이 정확한 표현이다.

### 6-3. 재평가: "내 보유 구독 활용" 목적에는 9Router인가 LiteLLM인가

**둘 다 아니다 — 이 목적의 정답은 라우터가 아니다.**

- **LiteLLM:** 구독을 아예 다루지 못한다(API 키 전용). "구독 활용" 목적에는 원천 부적합. 용도는 오직 API 자동화의 게이트웨이.
- **9Router:** 구독을 다루는 기술을 가진 유일한 후보지만, 당신 구독의 핵심(Claude ×2→1)이 정확히 **금지·차단된 경로**다. ChatGPT/Codex 경로는 아직 되지만 ⚠️ 미확인 지대이고, Anthropic 선례가 보여주듯 "구독 재판매 차익(token arbitrage)"을 벤더가 방치하지 않는다. 5년 설계의 기반으로 삼을 수 없다.
- **구독 활용의 실제 정답:** 구독은 **공식 클라이언트 안에서 최대로 쓴다** — Claude 구독은 Claude Code(=실행본부, 서브에이전트·훅 전부 구독 한도 내), ChatGPT 구독은 전략실+외부감사+chatgpt2codex(공식 커넥터 방식), Codex 구독은 Codex CLI에서 2차 구현. 본 설계 §3이 이미 "구독 100% 활용 구조"다. 라우터가 더해줄 수 있었던 것(공식 클라이언트 밖 사용)이 바로 금지된 부분이다.

**남는 결론:** 자체 네트워크 프록시 구축은 여전히 비추천 — 금지 범위(구독 우회)는 만들어도 차단 대상이고, 이 OS는 API를 쓰지 않으므로 합법 범위(API 라우팅)는 만들 이유 자체가 없다.

### 6-4. 구독 전용 전제의 재설계 — 공식 클라이언트로 어디까지 가능한가

**전제 확정: 이 OS는 API를 사용하지 않는다.** Router 계층을 "향후 API 자동화 대비"로 설계하지 않는다. 목적은 두 가지뿐이다 — (a) 보유 구독 계정(ChatGPT·Claude·Gemini·Codex)의 관리, (b) 작업별 모델 전환.

#### (1) 공식 클라이언트만으로: 가능 / 불가능 / 수동 대체선

| 원하는 기능 | 공식 클라이언트로 | 불가능하다면 수동 대체선 |
|---|---|---|
| **Claude 생태계 내부의 작업별 모델 자동 전환** | ✅ **완전 자동.** 서브에이전트 `model` 필드(haiku/sonnet/opus 자동 위임), `/model`, Advisor(⚠️) — 본 설계 §8·§12가 이미 이것 | 대체 불필요 |
| **벤더 간 자동 전환** (한 지점에서 GPT↔Claude↔Gemini 자동 배분) | ❌ **불가.** 각 구독은 공식 클라이언트에 결박되며, 제3자 통합은 금지·차단(§6-2) | "전환"을 3요소로 분해하면 대부분 대체된다 — 아래 (2) |
| **구독 통합 사용량 대시보드** | ❌ 통합 화면 없음. 각자 확인(Claude `/usage`, ChatGPT·Gemini 설정) | 주간 5분 기록 `vault/ops/usage.md` (§16 주간 루틴에 이미 포함). 한도 임박 시 정책 라우팅의 기본값을 바꾸는 것으로 대응 |
| **컨텍스트를 유지한 채 벤더 전환** | ❌ 자동 이전 없음 | **vault 표준 문서(PRD·결과 카드)가 컨텍스트 운반체.** 복붙 1회 = 이 설계가 처음부터 선택한 인터페이스 |
| **같은 코드베이스에서 벤더 전환** | 부분 ✅ — **공식 CLI 3종(Claude Code, Codex CLI, Gemini CLI)은 모두 같은 repo에서 동작하며 git이 공유 상태다.** 각자 자기 구독으로 로그인 = 전부 약관 청정 | 앱 전환만 수동(초 단위). "Claude가 구현 → Codex로 열어 교차 리뷰"가 이 패턴 |
| **멀티 계정 관리** (ChatGPT ×2 등) | 각 클라이언트 내 로그인 전환(수동) | 계정별 용도 고정(예: 1번=전략실, 2번=외부감사)으로 전환 자체를 없앤다 |

#### (2) "자동 전환"의 실체 분해 — 잃는 것은 수 초뿐이다

전환 = ①어디로 보낼지 **판단** + ②컨텍스트 **이동** + ③**실행**.

- **① 판단 → 자동화됨 (정책 라우팅).** `vault/prompts/routing-policy.md` 한 장:

```
| 작업 신호            | 담당                          |
| 전략·기획·사업계획     | ChatGPT (전략실)              |
| 구현·리뷰·문서화      | Claude Code (내부 전환은 자동)  |
| 대량 조사·멀티모달     | Gemini                       |
| 교차 구현·2차 의견     | Codex CLI                    |
| 애매하면             | Claude Code → 불만족 시 교차    |
```

  판단이 표 조회가 되는 순간 "자동 선택"의 인지 비용은 0이다. 새 모델 등장 = 표 한 줄 수정.
- **② 이동 → 표준화됨.** PRD·결과 카드가 운반체(§2). 복붙 1회.
- **③ 실행 → 유일하게 수동.** 앱을 여는 수 초.

**결론: 구독 체계에서 자동 라우터가 없어 잃는 것은 ③의 수 초뿐이며, 이것이 현 약관 환경에서 도달 가능한 최적점이다.** 자동화하려던 ①②는 이미 정책 문서와 vault가 해결한다.

#### (3) 9Router — "구독 통합·자동 전환 검증" 실험 계층으로서의 평가

실험 계층 유지에 동의한다. 단, 이 실험이 **지금 검증해줄 수 있는 범위를 정직하게 좁혀야** 실험이 된다.

- **검증 불가:** Claude 구독 통합 — 실험하면 재현되는 것은 "차단"이라는 이미 아는 사실이다. 메인 계정 기반의 모든 것 — 계정이 담보라서 실험 비용이 아니라 도박이다.
- **제한적 검증 가능:** ⓐ 쿼터 추적·통합 대시보드 UX가 주간 수동 기록 대비 실제로 나은가(관리 화면 관찰) ⓑ "자동 전환"의 체감 가치 — 일회용 계정+무료 Provider만으로 ⓒ **풍향계 관찰** — 어떤 Provider가 열리고 닫히는지는 벤더 정책 변화의 조기 신호다.
- **실험 규칙(불변):** 메인 워크플로에서 격리, 메인 계정 OAuth 절대 금지, 결과는 `vault/decisions/`에 기록, 분기 재평가로 유지/폐기 판정.
- **이 실험의 산출물 정의:** "도입 여부"가 아니라 **"구독의 멀티클라이언트 사용이 합법화되는 날, 하루 만에 켤 수 있는 지식"**이다. 그 이상을 기대하는 순간 실험이 아니라 리스크가 된다.

#### (4) 장기: Router 계층의 Core 승격 조건 (구독 전용 전제)

구판의 API 기반 승격 조건은 **폐기**한다. 구독 전용 전제에서 남는 승격 조건은 하나뿐이다:

> **벤더가 구독의 제3자/멀티클라이언트 사용을 공식 허용하는 정책 변화가 있을 때.**

그 전까지 Router 계층은 조직도의 예약석(빈 자리)으로만 존재한다. 필요 여부가 전적으로 벤더 정책에 종속이므로, 우리가 미리 쌓을 것은 라우터 기술이 아니라 (a) 정책 라우팅 문서의 성숙, (b) vault 인터페이스의 표준화다 — 정책이 바뀌는 날, 라우터는 이 둘 위에 그냥 꽂힌다. (참고: 훗날 API를 쓰기로 전제 자체가 바뀌면 그때의 후보는 LiteLLM·OpenRouter류지만, 이는 현 설계 범위 밖이다.)

## 8. Agent 설계서 (실행본부 내부 — `.claude/agents/`)

전부 ✅ 공식 문법. 조사·기록은 `haiku`(저비용), 구현·리뷰는 `inherit`. 모델 교체는 frontmatter 한 줄.

**planner.md** — PRD를 받아 분해 (이중 두뇌의 접점):
```markdown
---
name: planner
description: vault/inbox의 PRD를 읽고 작업 분해·리스크 식별. 새 작업 시작 시 사용.
tools: Read, Grep, Glob
model: inherit
---
vault/inbox/에서 지정된 PRD를 읽어라. PRD 템플릿의 각 항목을
작업 단위로 분해하고, 불명확한 요구는 "질문" 목록으로 분리하라.
분해 결과는 승인용 계획으로 제시하고, 승인 전에는 어떤 실행도 위임하지 마라.
코드 현실과 PRD가 어긋나는 지점을 발견하면 최우선으로 보고하라.
```

**researcher.md** — 저비용 조사 격리:
```markdown
---
name: researcher
description: 기술 조사, 공식 문서 수집. 대량 검색 시 사용.
tools: Read, Grep, Glob, WebFetch, WebSearch
model: haiku
---
공식 문서와 GitHub를 우선 출처로 하라. 블로그·SNS 주장은 "미검증" 표시.
결과는 출처 링크와 함께 요약만 반환하고, 원문 덤프를 반환하지 마라.
```

**developer.md** — 쓰기 허용 + 훅 방어 + tester 중첩:
```markdown
---
name: developer
description: 승인된 계획의 구현. 계획 승인 후에만 사용.
tools: Read, Write, Edit, Bash, Grep, Glob, Agent
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/block-dangerous.sh"
---
승인된 계획 범위만 구현하라. main 브랜치 직접 커밋 금지, feat/ 브랜치에서만.
완료 후 tester 서브에이전트로 테스트를 실행하고 실패만 보고받아라.
계획에 없는 파일을 수정해야 하면 중단하고 사유를 보고하라.
```

**tester.md** — developer가 중첩 호출:
```markdown
---
name: tester
description: 테스트 실행과 실패 분석. developer가 구현 완료 후 호출.
tools: Read, Bash, Grep, Glob
model: haiku
---
테스트를 실행하고 실패한 항목만 에러 메시지와 함께 보고하라.
통과 항목은 개수만 요약하라. 코드를 수정하지 마라.
```

**reviewer.md** — 읽기 전용 강제:
```markdown
---
name: reviewer
description: 코드 변경 후 품질·보안 리뷰. 코드 수정 직후 능동적으로 사용.
tools: Read, Grep, Glob, Bash
model: inherit
---
git diff로 변경분을 확인하고 치명적/경고/제안 3단계로 피드백하라.
노출된 시크릿, 에러 처리 누락, 입력 검증을 반드시 확인하라.
코드를 직접 수정하지 마라.
```

**doc-writer.md** — 결과 카드 생산 (외부감사로 가는 출력):
```markdown
---
name: doc-writer
description: 작업 완료 후 결과 카드·PR 본문 작성.
tools: Read, Write, Grep, Glob
model: haiku
---
vault/templates/RESULT-CARD.md 형식(v1)으로 작성하라. 저장은 해당 작업 폴더
vault/ops/tasks/{작업폴더}/에 2x_RESULT-claudecode.md로 하라.
이 카드는 취합자·외부감사에 그대로 전달되므로 코드 없이도 이해되게 써라.
```

**knowledge.md** — 세션을 넘는 조직 기억:
```markdown
---
name: knowledge
description: 결정사항·교훈을 vault에 기록·조회.
tools: Read, Write, Edit, Grep, Glob
model: haiku
memory: project
---
작업 완료 시 결정 내용·이유·기각된 대안을 vault/decisions/에 날짜별 기록하라.
새 작업 시작 전 관련 과거 결정을 먼저 검색해 planner에 제공하라.
```

```
    └── n8n/                     # 워크플로 export(JSON) 보관 = 버전 관리
```

### PRD 템플릿 (vault/templates/PRD.md — 두 두뇌 간 계약서)

```markdown
# PRD: {프로젝트/기능명}
- 작성: {날짜} / ChatGPT 세션: {링크 or 요약}
## 1. 목적 (왜 하는가, 1문단)
## 2. 성공 기준 (측정 가능하게, 3개 이하)
## 3. 범위 (이번에 하는 것)
## 4. 비범위 (이번에 안 하는 것 — 필수)
## 5. 제약 (예산·기한·기술)
## 6. 미해결 질문 (planner가 답하거나 나에게 되물을 것)
```

**PRD ↔ TASK-PACKET 관계 (계층 정의):** PRD는 프로젝트 단위의 계약서, TASK-PACKET(v1)은 작업 단위의 발주서다. planner가 PRD를 분해한 결과가 곧 패킷들이다 (1 PRD → n PACKET, 각 패킷의 "입력 자료"에 PRD 경로 기재). 단발 업무는 PRD 없이 패킷만으로 발주한다 — 기준은 START-HERE의 예외 규칙과 동일.

### CLAUDE.md 표준 템플릿 (각 프로젝트 루트)

```markdown
# 프로젝트: {이름}

## 조직 규칙 (모든 프로젝트 공통)
- 너는 실행본부 오케스트레이터다. 기획·전략 결정은 하지 않는다.
  PRD가 불명확하면 구현하지 말고 질문 목록을 반환하라.
- 새 작업은 반드시 vault/prd/의 PRD에서 시작한다. PRD 없는 요청은
  "PRD 먼저"라고 답하라 (긴급 버그 수정 제외).
- main 브랜치 직접 커밋 금지. 모든 변경은 feat/* 브랜치 → PR.
- merge, 배포, 파일 삭제, 대량 수정(10파일+)은 사람 승인 필수.
- 작업 완료 시: doc-writer로 결과 카드, knowledge로 결정 기록.
- 과거 결정과 충돌하는 구현 전에 vault/decisions/를 검색하라.

## 이 프로젝트 특화 규칙
- 기술 스택: {…}
- 테스트 명령: {…}
- 절대 건드리지 말 것: {.env, 결제 로직, …}

## 금지 사항
- rm -rf, git push --force, 시크릿 파일 읽기/쓰기
- 외부 API 신규 결제가 필요한 결정
```

## 10. GitHub 전략

- **브랜치:** `main`(보호, 직접 push 전원 금지) + `feat/작업명`. 에이전트는 feat에서만.
- **PR:** 모든 변경은 PR 경유. 본문은 doc-writer 결과 카드 재사용.
- **리뷰 3중:** ① reviewer 서브에이전트(로컬) → ② PR에서 Claude 자동 리뷰(claude-code-action) 또는 Codex 교차 리뷰 → ③ merge 버튼은 항상 사람.
- **Actions:** 테스트/린트 CI + vault 자동 백업까지만. 배포 자동화는 구축 5단계 이후.
- **vault도 private repo** — 코드와 기억이 모두 이력에 남으면 "실수 복구"가 항상 가능하다.

## 13. 비용 최적화

1. **구독 우선, API 최후.** 대화·개발은 구독 앱에서. API는 n8n 자동화 지점에만, 그때 Batch API(할인)+프롬프트 캐싱(정적 지시문 앞 고정) 기본 적용.
2. **Orchestrator-Workers(✅ Cookbooks)** = §3 구조 그 자체. 서브에이전트 `model` 필드로 이미 달성, 별도 구현 없음.
3. **Evaluator-Optimizer(✅ Cookbooks)** = developer↔reviewer 루프. 상위 모델 없이 품질 상승.
4. **Advisor(⚠️ 실험적):** Sonnet 메인+Opus 어드바이저 — "결정 시점에만 상위 모델 호출이 상시 구동보다 저렴"(공식 문서). 실험 계층이므로 꺼져도 되는 위치에만.
5. **Claude 1계정 축소 시:** 구조 변경 0. 서브에이전트는 한 계정 한 세션 안에서 돈다. 기획축(GPT)·기억(vault)은 무관.

## 14. 보안·안전

- **삭제·대량 수정 방어:** ① 모든 작업은 Git 브랜치 위(revert로 복구) ② developer 훅으로 `rm -rf`·`git push --force`·시크릿 접근 차단 ③ reviewer·researcher·tester는 도구 자체가 읽기 전용 ④ `bypassPermissions` 전 에이전트 금지.
- **백업 3겹:** ① 코드·vault 모두 GitHub private ② 주간 n8n 압축 백업(별도 위치) ③ OS 백업. 기준: "AI가 지워도 30분 내 복구".
- **시크릿:** `.env`+`.gitignore`, n8n은 자체 credential 저장소. chatgpt2codex Owner Token은 비밀번호급 관리. 라우터 도입 시에도 API 키만, OAuth 금지.
- **승인 위임 불가(✅ 공식):** 어떤 에이전트 메시지도 권한 프롬프트를 대신 승인할 수 없다 — 사람 승인이 도구 수준에서 보장된다.

## 15. 단계별 구축 가이드

**1단계 (1주) — 뼈대와 기억.** §9 폴더 구조 생성, vault를 Obsidian으로 열고 templates/에 PRD·TASK-PACKET(v1)·RESULT-CARD(v1) 배치, GitHub private repo 2개(vault, ai-os) 생성·연결.
왜 먼저: 조직은 기억 없이 존재할 수 없다. 에이전트부터 만들면 매 세션 맥락 재설명 비용이 모든 단계에 복리로 붙는다.
완료 기준: ChatGPT에서 만든 PRD 1개가 inbox→prd로 승격되어 있다.

**2단계 (1주) — 울타리.** CLAUDE.md 배치, `settings.json` 권한(deny 규칙), `block-dangerous.sh` 훅 작성, n8n 없이 수동 백업 1회 실행 확인.
왜 에이전트보다 먼저: 쓰기 권한을 주기 전에 울타리가 있어야 한다. 순서를 바꾸면 사고 후에 울타리를 만들게 된다.
완료 기준: Claude Code에서 `rm -rf` 시도가 차단된다.

**3단계 (2주) — 읽기 에이전트 4종.** researcher·reviewer·doc-writer·knowledge 배치, 정부지원사업 조사를 researcher로 실전 1회, 결과 카드 1장을 ChatGPT 외부감사에 돌려 사이클 검증.
왜 이 4개: 전부 읽기 중심이라 사고 위험이 낮고 효과는 즉시 체감된다.
완료 기준: PRD→조사→RESULT-CARD→GPT감사 1사이클이 돌았다.

**4단계 (2–3주) — 쓰기 에이전트와 개발 파이프라인.** planner·developer·tester 배치, 꽃집 AI에서 브랜치/PR/CI 실전 적용, Codex 교차 리뷰 1회, chatgpt2codex 설치 후 ChatGPT 검증 1회.
왜 지금: 리뷰 체계(3단계)가 있어야 쓰기 에이전트를 받아낼 수 있다.
완료 기준: feat 브랜치→PR→3중 리뷰→사람 merge가 1회 완주됐다.

**5단계 (2주) — 반복 자동화.** n8n에 공고 수집·아침 요약·주간 백업 워크플로 구축, 워크플로 JSON을 automation/n8n/에 export(버전 관리), 꽃집 웹훅 연결.
왜 마지막: 자동화는 안정된 수동 프로세스의 복제다. 프로세스가 굳기 전 자동화하면 잘못된 프로세스가 고착된다.
완료 기준: 1주일간 사람 개입 없이 수집·알림·백업이 돌았다.

**6단계 (상시) — 분기 재평가.** §17 확장 계획의 체크리스트로 실험 계층 승격/폐기 결정.

## 16. 운영 매뉴얼

**일일 루틴 (10분):** n8n 아침 요약 확인 → inbox 새 파일 분류(PRD 승격/보류/폐기) → 열린 PR 확인·merge 결정.
**주간 루틴 (30분):** 백업 성공 확인(복원 테스트는 월 1회) → 사용량 점검(Claude `/usage`, ChatGPT·Gemini 한도) → decisions/ 훑고 반복되는 실수는 CLAUDE.md에 규칙으로 추가(=조직 학습).
**월간 루틴:** 에이전트 프롬프트 개정(잘 틀리는 지시 수정) → 비용 결산: API 비용 발생 시 §6 라우터 트리거 판단.

**사고 대응:**
- 파일 삭제/오염 → `git revert`/`git checkout`, vault는 Git 이력에서 복구. 원인 명령을 훅 차단 목록에 추가.
- 에이전트 폭주(무한 루프·과도 호출) → 세션 중단 → 해당 에이전트 `maxTurns` 설정 추가 → decisions/에 기록.
- 계정 한도 도달 → 급하지 않으면 대기, 급하면 그 작업만 Codex/Gemini로 우회. 라우터로 우회하지 않는다(§6).
- 벤더 장애/서비스 종료 → 코어(vault·Git)는 무손상. 해당 부서의 역할 텍스트를 새 도구에 이식.

**품질 유지 원칙:** 같은 실수 2회 = CLAUDE.md 규칙화. 같은 질문 2회 = vault/prompts/에 템플릿화.

## 17. 향후 확장 계획

**분기 재평가 체크리스트 (실험 계층 → 승격/유지/폐기):**

| 대상 | 승격 조건 | 폐기 조건 |
|---|---|---|
| Advisor | "experimental" 문구 제거 + 요금 안정 | 가격 급등 또는 기능 제거 |
| chatgpt2codex | 릴리즈 지속 + 커뮤니티 성장 | 6개월 무업데이트 |
| Agent Teams | 병렬 프로젝트가 실제 3개+ 상시 가동될 때 | 필요 없으면 영구 보류 |
| Router 계층 (9Router 실험) | 벤더가 구독의 제3자/멀티클라이언트 사용을 공식 허용하는 정책 변화 시에만 (§6-4) | 실험 유지 비용이 관찰 가치를 넘어서거나, 프로젝트가 6개월+ 무업데이트일 때 |

**성장 단계별 확장:**
1. **지금(1인+AI):** 본 설계 그대로.
2. **서비스 출시(꽃집 AI 등 외부 사용자 발생):** 해당 프로젝트만 Managed Agents/서버 배포 검토(✅ 공식 API). 로컬 AI OS는 개발·운영 본부로 유지.
3. **협업자 합류:** vault·프로젝트 repo 권한 분리, `.claude/agents/`가 이미 repo에 있으므로 조직 규칙이 자동 공유된다.
4. **새 모델/벤더 등장 시 교체 절차:** ① 역할 텍스트(에이전트 md·CLAUDE.md)는 그대로 ② `model:` 필드/부서 담당만 교체 ③ 2주 병행 운용 후 decisions/에 비교 기록 ④ 완전 전환. — 이 절차가 "5년 이상"의 실체다: 도구는 흐르고, 조직은 남는다.

## 18. 설계 거버넌스 (이 설계서 자체를 만들고 고치는 규칙)

배경: 설계-구현이 사용자를 통해 ChatGPT↔Claude 사이를 오가며 반복 수정되는 낭비가 있었다. 역할을 고정한다.

- **기획·구현 (단독):** Claude 측 — 현재는 Fable 5. vault의 실물 파일·v1 규약을 직접 확인한 상태에서 설계와 파일 구현을 전담한다.
- **Fable 접근 종료 시 승계 (7/19 이후 미결제 전제):** 승계자는 Claude Code/Cowork의 **Opus(판단·감사성 검토)+Sonnet(구현·문서 수정)**. 승계 절차는 "이 설계서 + vault/decisions/를 읽는 것"으로 완료 — 별도 이관 작업이 없다. **역할은 모델이 아니라 파일에 귀속된다**(§1 원칙)는 것이 이 승계 설계의 근거다.
- **ChatGPT (감사 한정):** 완성본을 받은 뒤 3가지만 판정한다 — ① 원래 목표 반영 여부 ② 실제 사용 가능성 ③ 앞뒤 모순. 새 설계·프레임 추가는 하지 않는다. ※ 이 제한은 "설계서 프로젝트"에 한정된다. 일반 업무에서의 전략실·총괄 역할(§3, ORCHESTRATION-LAYER)은 불변.
- **사용자:** 승인과 방향 결정만. 중간 재설계 요청은 하지 않고, 새 아이디어는 vault/inbox/에 적립했다가 분기 재평가 때 일괄 반영한다.
- **변경 관리:** v1 규약(TASK-PACKET·RESULT-CARD)의 변경은 "원래 요구사항 → 가능한 구현 → 불가능한 부분과 이유 → 대체안" 순서로 보고 후 사용자 승인 시에만. 그 외 문서는 기획자가 정합성 판단으로 직접 수정하되 변경 사유를 vault/decisions/에 남긴다.

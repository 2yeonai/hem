# 정부지원 승인화면 프로토타입 (v1)

`승인화면_프로토타입_v1.html` — 정부지원사업 매칭 AI 파이프라인에서 **대표자검수**(kind:human 승인 지점)가
실제로 어떤 화면일지 보여주는 단일 파일 HTML 목업. 외부 CDN/폰트 없이 브라우저에서 더블클릭으로 바로 열림
(백엔드 연결 없음, 전부 화면 시뮬레이션).

## 무엇을 기준으로 만들었나 (지어낸 값 없음)

이 목업의 큐 3건은 전부 실제 파일에서 그대로 가져왔다.

1. **"온천꽃식물원" 사업계획서 초안 (DRAFT 케이스)** — 이 작업 중 직접
   `python3 scripts/run.py test/blind_test_oncheon_saenghwalmunhwa.json scripts/sample_announcements.json`
   을 target_period만 미래(2026-07-17~2026-09-30)로 넓혀 재실행해서 얻은 **실제 run.py 출력**이다.
   business_profile(업종/지역/사업설명 등)은 `test/blind_test_oncheon_saenghwalmunhwa.json` 원문 그대로 —
   온천꽃식물원의 실제 사업자 정보(화훼 도소매, 2015년 창업, 경상남도 창녕군)이며 지어낸 항목이 없다.
   결과가 `lock_state: DRAFT`(AI 자체검수 미통과)로 나온 것도 실제 판정 그대로이며, 일부러 "이상적인
   케이스"만 골라 보여주지 않았다 — 확인 필요 항목이 많은, 실제로 대표자 개입이 필요한 상태를 그대로 보여준다.
2. **"신청기업 A" 사업계획서 초안 (READY_FOR_APPROVAL 케이스)** — `test/sample_output_ideal.json`
   (회귀테스트 fixture)을 그대로 사용. 이 스킬은 애초에 상호·대표자 이름을 스키마에 넣지 않는
   블라인드 심사 원칙을 따르므로(`scripts/run.py`의 `draft_application()` bullets4 주석 참고),
   이 fixture에도 회사명이 없다 — 화면에도 "신청기업 A(회귀테스트 fixture)"로 정직하게 표시했다.
3. **"온천꽃식물원" 발표자료 패키지 (PPT초안·발표대본·예상QNA)** — `test/backend_result_2026-07-17.json`
   (발표 뒷단 8-stage mock 실행 결과)을 그대로 사용. PPT/대본이 한 번 반려됐다가("슬라이드 10장 초과",
   "발표시간 배분 오류") 재작업 후 승인된 실제 이력이 그대로 들어 있다.

## 화면 구성

- **검수 대기 큐**: 위 3건을 카드로 나열. 사업계획서 초안은 매칭 프로그램명·마감일 D-day·AI 자체검수
  통과여부·적합도(%) 배지, 발표자료 패키지는 PPT/대본/QNA 3종 각각의 상태 배지.
- **사업계획서 초안 상세**:
  - AI 자체검수(quality_gate_result) 배너 — 통과/미통과와 상관없이 항상 사람이 최종 승인/반려를 누른다
    (회사 헌장: 최종 판단은 대표자).
  - 매칭 결과 · 제외된 공고 목록(접기/펼치기)
  - 신청서 초안 6개 섹션(①사업개요~⑥기대효과), `○ 소제목 : 내용` 서식 그대로 렌더링
  - 섹션별 배지: **AI 초안**(값 그대로 채워짐) / **AI 제안 포함**(`[AI 제안 — 실제 집행 전 사업주 확정 필요]`,
    2026-07-17 신설된 예산배분·기대효과 자동제안) / **⚠ 확인 필요 있음**(`[확인 필요]`, 정말 모르는 항목)
    — `scripts/run.py`가 실제로 만드는 이 두 라벨을 화면에서도 섞지 않고 구분했다.
  - 필수서류 준비 현황(준비완료/미준비 배지) + 서류명 매칭 실패 사유
  - AI 심사위원 모드 자기검수: 감점위험/과장표현/근거없는서술, PSST 4구간(문제·해결·증빙·성과) pass/fail,
    가상 심사위원 9인(8인 + 홍재우) 점수·즉시경고, 감점 전파(설명 전용)
  - 확인 필요 항목 목록(접기/펼치기)
  - **승인 처리**: [승인]/[반려] 버튼, 반려 시 사유 필수 입력 모달, 처리 이력(approval 블록:
    status/version/approved_by/rejection_reason/version_history 시뮬레이션), 승인 후 잠금 배너,
    "제출은 대표자가 직접" 고정 안내
- **발표자료 패키지 상세**: 발표 요건 요약 → PPT초안/발표대본/예상QNA 3개 문서 각각 승인상태·버전·
  이전 반려 이력 → 발표패키지저장 완료 배너.

뺀 것(이번 v1 범위 밖, 다음 버전 후보):
- ① 매칭 시작 입력 폼, ③ 신청서 마법사(5단계 단계별 입력 UI), ⑥ 마감 캘린더 — 이번 지시가
  "대표자검수 화면 중심"이었으므로 검수/승인 흐름에 집중하고 입력 마법사는 만들지 않았다.
  (마법사가 필요해지면 정부지원_UI참조.md ③번 설계를 그대로 따르면 됨.)

  아직 "반려 후 어느 단계로 되돌아가는지"가 파이프라인에 정해져 있지 않아(아래 알려진 한계 참고),
  화면에서 값을 고쳐도 실제로 재생성되지 않는다는 오해를 줄 수 있어 이번 v1에서는 보기 전용으로 뒀다.

## 알려진 한계

- **승인/반려는 화면 시뮬레이션이며 `scripts/run.py`와 연결돼 있지 않다.** 특히 사업계획서 초안 쪽은
  `정부지원_manifest_v2.yaml`의 `open_issues` 1번에 이미 기록돼 있듯 "반려 후 어느 stage로 되돌아가는지"
  자체가 아직 미정이다 — 이 화면의 반려 버튼은 그 사실을 안내 문구로 그대로 노출한다.
- 발표자료 패키지 쪽(`PPT승인`/`발표패키지승인`)은 매니페스트에 `kind: human`과 `return_to`가 이미
  정의돼 있어 사업계획서 쪽보다 구조적으로 더 성숙하지만, 실제 재승인 시 `stage_ppt초안생성()` 등을
  다시 호출하는 배선은 이 화면 밖(run.py 쪽)의 일이다.
- PPT초안 본문은 "제출확정본 PDF에서 구간을 그대로 잘라낸(psst_anchor_split)" 실제 데이터라 안내문·
  개인정보동의서 문구가 섞여 있다 — `기능_인덱스.md`에 이미 기록된 알려진 한계이며, 화면에도 그 사실을
  그대로 안내했다(정제해서 숨기지 않음).
- 실 LLM/STT 연동 없음 — `run.py`가 만드는 값을 그대로 보여줄 뿐, 이 화면 자체는 아무것도 생성하지 않는다.

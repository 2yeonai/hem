# KNOWN_LIMITATIONS.md — 한계 고지 (필독)

## 1. 규칙 기반 분석이다 (AI 아님)

현재 앱의 모든 점수·판정은 **키워드·수치 존재 여부 등 규칙 기반 추정**이다.
- 공고 적합성 = 어휘 겹침 세기 → 의미를 이해하지 못한다. 동의어를 놓치고, 우연 일치에 속을 수 있다.
- 예상 점수표는 "감점 요인이 몇 개나 남았는지"의 지표이지 실제 심사 점수 예측이 아니다.
- 정밀 판단(차별성 실질 평가, 스토리 일관성)은 Export 탭의 프롬프트로 AI 세션에, 확정은 사람에게.

## 2. 실제 공고의 최종 판단은 사람이 한다

- 신청자격·지원제외 해당 여부는 반드시 **공고 원문 조항과 서류(납세증명·수혜이력·사업자등록증)**로 사람이 확인한다. 앱의 "이상 없음"은 입력값 기준일 뿐이다.
- 법률·세무·행정 판단을 자동 확정하지 않는다. 이 앱은 행정사·세무사·전문 컨설팅을 대체하지 않는다.

## 3. 공고 원문이 없으면 LOCK 금지

raw_text가 비어 있으면 전 판정이 [확인 필요] 모드가 되고 LOCK·"제출 가능" 판정이 구조적으로 막힌다. 이것은 버그가 아니라 안전장치다. 우회하지 마라.

## 4. 예산·증빙 미확인 시 제출 불가 또는 보완 필요

- 견적서 없는 예산 항목은 "발화 금지"로 표시된다 — 근거 없는 금액이 더 위험하기 때문.
- 확보 증빙 0건인 후보의 주장은 단정 금지. 앱은 이를 감점으로만 표시할 수 있고, 증빙 실물의 존재는 확인하지 못한다.

## 5. 기술적 한계

- 위험 표현 스캐너는 부분 문자열 매칭 — 방어 문맥("확정 표현 금지")도 검출할 수 있다(오탐). 검출 결과는 삭제 지시가 아니라 검토 지시다.
- 공고문 추출기는 줄 단위 키워드 탐색 — 표·붙임파일 구조는 못 읽는다. 핵심 조항은 입력 필드에 직접 붙여넣어라.
- 동시 편집 보호 없음(로컬 1인용). input.json을 손으로 고칠 때 JSON 문법 오류는 화면 오류로 이어진다.
- run_factory.py(CLI)와 engines.py(앱)는 위험 표현 파서만 공유 규칙이 같고 코드가 분리돼 있다 — 통합은 NEXT_ROADMAP #3.

## 6. 샘플의 한계

sample-demo는 가상 공고다. 수치·배점·조항 전부 지어낸 값이며, 실제 공고 작업의 근거로 쓰면 안 된다 (D-006).

# 혜미식 AI 작업 공장 — 전체 설계서 (1차 공정 산출물)

작성일: 2026-07-03
상태: LOCK (구조 변경 시 DECISIONS.md에 기록 후 변경)

## 1. 전체 시스템 개요

이 공장은 단발성 산출물이 아니라 **반복 생산 구조**를 만든다.

- 입력: 공고문 / 신청자 정보 / 아이디어 후보 / 심사표 / 예산 기준 / 제출양식 / 발표 조건 / 증빙자료
- 처리: 워크플로우(02_workflows) + 규칙(00_rules) + 템플릿(03_templates)에 따라 단계별 생산
- 출력: 04_outputs에 산출물, 05_reviews에 심사위원 검수, 06_locks에 LOCK 기록
- 원칙: **매번 현재 공고 기준으로 다시 판단한다. 이전 사례를 기본값으로 쓰지 않는다.**
- 관점: Gemini식 확장성(성장성 살리기) + Monday식 방어성(탈락 리스크 막기)를 항상 병행한다.

핵심 파이프라인:

```
공고문·신청자 입력
  → [A] 정부지원사업 공장 (서류에서 점수를 만든다)
  → [B] 발표자료 공장   (발표자료에서 구조를 보여준다)
  → [C] 발표연습기 공장 (실제 말로 방어한다)
  → [D] 콘텐츠 변환 공장 (선정 후 홍보·후기 재생산)
  → [E] 업무자동화 설계 공장 (사업 아이템 자체의 MVP 설계)
모든 단계 사이: LOCK → 검수 → 다음 단계
```

| 품질 기준 | evaluation_rubric.md + 공고 심사표 기준. 배점 높은 항목에 분량·증빙 집중, 주장마다 증빙 연결, 위험 표현 0건 |
| 실패 위험 | 이전 공고 기준을 재사용, 자격 오판으로 접수 탈락, 증빙 없는 주장, 예산이 단순 구매·외주로 보임 |

### B. 발표자료 공장

| 항목 | 내용 |
|---|---|
| 생산라인명 | 발표자료 공장 (presentation_workflow) |

| 검수 지점 | 제출본과 숫자·과제명·예산 일치 여부(사람 확인), 발표자가 실제로 말할 수 있는 표현인지 |
| 자동화 가능 지점 | 목차 생성, 슬라이드 문구 초안, 불일치 대조 |
| 품질 기준 | 1슬라이드 1주장, 제출본과 충돌 0건, 배점 상위 항목이 핵심 슬라이드에 배치 |

| 후속 확장 | 슬라이드 자동 생성(PPT 파일), 심사표별 목차 프리셋 |

## 3. 입력 → 처리 → 출력 흐름

```
01_inputs/   입력자료 (공고문, 신청자 정보, 아이디어 후보, 증빙 목록)
    ↓
02_workflows/ 해당 워크플로우 선택 (factory_orchestrator_workflow.md가 라우팅)
    ↓  (00_rules/ 규칙 + 03_templates/ 템플릿 적용)
04_outputs/  산출물 생성 (프로젝트별 폴더)
    ↓
05_reviews/  심사위원 모드 검수 (칭찬 금지, 감점 먼저)
    ↓
06_locks/    통과 시 LOCK 기록 → 다음 단계
    ↓
10_handoff/  세션 종료 시 인계문
```

## 4. 사람이 반드시 검수할 지점 (자동 확정 금지)

1. 신청자격·지원제외 최종 판정 — 오판 시 접수 탈락
2. 최종 아이디어 선정 — 사업 방향 결정
3. 예산 금액·항목 확정 — 실제 견적·집행 책임
4. 증빙자료의 사실 여부 — AI는 존재를 확인할 수 없음
5. 제출 직전 최종본 — 제출 책임은 사람
6. 개인정보·법률 관련 표현

## 5. 자동화 가능한 지점

- 공고 핵심 기준 추출·요약, 배점 분석
- 아이디어 후보 비교표, 감점 위험 표시
- 과제명·한줄요약 후보 다중 생성
- 위험 표현 스캔·대체 표현 제안 (danger_words.md 기준)
- 체크리스트 대조 (evidence, budget, final_review)
- 발표 공격 질문 생성, 대본 압축·쉬운 말 변환
- 콘텐츠 채널별 변환

## 6. 단계별 LOCK 구조

lock_rules.md에 상세. 요약: 공고 기준 → 자격·리스크 → 아이디어 후보 → 심사표 분석 → 최종 아이디어 → 과제명 → 작성전략 → 본문 → 증빙 → 예산 → 분량 → 위험표현 제거 → 제출본 → 발표자료 → 대본 → Q&A 순으로 LOCK. **LOCK 이후 새 아이디어는 본문에 섞지 않고 확장 아이디어로 분리한다.**

## 7. 품질관리 구조

- 모든 산출물은 05_reviews에서 심사위원 모드(감점 우선) 검수를 통과해야 LOCK
- 검수 기준: 공고 적합성 / 심사표·배점 / 아이디어 선정 / 실행 가능성 / 예산 / 증빙 / 위험 표현 / 발표 방어 (8종)
- 샘플·가정값은 반드시 "확인 필요" 태그를 달고, 확정처럼 쓰지 않는다
- 검수 실패 시 revision_order.md(재작업 지시서)를 생성하고 우선순위대로 수정

## 8. 후속 확장 구조 (지금 하지 않는 것)

- 공고문 PDF 자동 파싱 도구
- 라벨 PDF 실제 생성 스크립트 (E라인 구현)
- 발표 녹음·텍스트화 비교 도구
- 과거 제출본·선정 사례 DB
- 웹 UI 앱화

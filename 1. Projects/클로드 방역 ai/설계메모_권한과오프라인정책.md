# 설계 메모 — 권한체계와 오프라인 정책 (2026-07-07)

**이 문서는 설계 메모입니다. 코드나 manifest.yaml에는 아직 반영하지 않았습니다.**
지금은 "새 문서가 뭐라고 제안하는지"와 "지금 구조와 어떻게 다른지"만 정리한 것이고,
실제로 넣을지 말지는 사용자 결정 이후에 진행합니다.

---

## 1. 권한체계 (기사/운영/대표/관리자)

### 새 문서가 제안하는 3역할 모델

| 역할 | 할 수 있는 일 | 제한 |
|---|---|---|
| 현장입력자 | 상담/방문 입력, 사진 첨부, 초안 생성 요청 | 최종 승인·잠금 불가 |
| 대표자 | 전체 조회, 수정, 승인, 반려, 잠금, 재발행 | 없음 |
| 관리자 | 사용자관리, 기준값 관리, 보관 정책 관리 | 현장 기록 임의변경은 제한 권장 |

(사용자가 언급한 "기사/운영/대표/관리자" 4개 역할과, 새 문서의 "현장입력자/대표자/관리자" 3개 역할은
이름이 완전히 일치하지는 않습니다 — "기사"↔"현장입력자", "운영"에 해당하는 역할이 새 문서엔 별도로
없어 보입니다. 이 차이 자체를 지금 결정하지 않고 그대로 적어둡니다.)

### 지금 구조와의 차이

- 지금 manifest.yaml에는 `대표자검수`라는 human stage 하나만 있고, "현장입력자"나 "관리자"가
  별도 역할/stage로 구분돼 있지 않습니다(사업계획서 원문 기준 "대표/직원 2인 체제" 전제로 설계됐음).
- 위 3~4개 역할을 실제로 구분하려면 최소한 "누가 이 데이터를 입력/수정/승인할 수 있는가"를
  stage 단위가 아니라 사용자 단위로 관리하는 개념이 추가로 필요해 보입니다.
- 이건 1단계 대조표의 "구조 변경이 필요할 수도 있는 것" 추가 후보 항목과 같은 문제입니다.

### 열린 질문 (지금 결정 안 함)

- "운영" 역할이 정확히 무엇을 하는 역할인지 (관리자와 어떻게 다른지) 확인 필요
- 권한을 사람 단위로 관리할지, 역할 단위로 관리할지
- 관리자가 "현장 기록 임의변경 제한 권장"이라고만 돼 있는데, 실제로 막을지 말지

## 2. 오프라인 정책

### 새 문서가 제안하는 내용

| 항목 | 권장 동작 |
|---|---|
| 오프라인 저장 | 입력 즉시 로컬 임시저장, 네트워크 복구 시 동기화 |
| 충돌 처리 | 최신 수정본 자동 병합이 아니라 대표자 확인 트리거 |
| 사진 첨부 | 현장 사진은 작업 전/후 또는 핵심 구역 기준으로 첨부 |
| 승인 흐름 | 현장 입력자 승인 불가, 대표자만 최종 승인 |
| 반려 흐름 | 반려사유 필수 입력 |
| 잠금 | 승인된 문서는 수정 불가, 재발행 시 새 버전 생성 |
| 감사로그 | 작성·수정·승인·반려·잠금 이벤트 기록 |

### 지금 구조와의 차이

- 지금 manifest.yaml과 run.py 어디에도 "오프라인 임시저장"이나 "동기화" 개념이 없습니다.
  모든 stage가 항상 네트워크가 연결돼 있다고 가정하고 설계돼 있습니다.
- "승인 흐름"과 "잠금"은 이미 지금 구조(대표자검수 → 고객DB저장봇 순서)의 방향과 크게 다르지
  않습니다 — 다만 "문서 잠금 후 수정 불가, 재발행 시 새 버전"은 3단계 테스트에서 확인한
  "템플릿버전" 갭과 같은 문제입니다.

### 열린 질문 (지금 결정 안 함)

- 오프라인 임시저장을 어디에 담을지(로컬 파일? 별도 임시 스토리지?)
- 동기화 충돌이 났을 때 "대표자 확인 트리거"를 구체적으로 어떻게 구현할지
- 현장 통신이 자주 끊기는 게 실제로 얼마나 자주 있는 일인지(골든셋 확보 후 확인 필요 — 골든셋 템플릿
  "추가_확인사항"에 이 질문을 추가할지도 사용자 확인 필요)

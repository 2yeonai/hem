# 방역&클린 문서승인 화면 프로토타입 (v1)

이 폴더의 `승인화면_프로토타입_v1.html`은 방역&클린 18단계 파이프라인(`1. Projects/클로드 방역 ai/manifest.yaml`) 중
`kind: human`인 두 게이트 가운데 **문서승인**(완료보고서·소독증명서 승인/반려)이 실제로 어떤 화면일지 미리 보기 위한
**목업**입니다. 백엔드·AI 모델·데이터베이스 없이, 파일 하나만으로 브라우저에서 즉시 동작합니다. (대표자검수 — 고객 문의/방문확정
승인 — 는 이 화면의 대상이 아닙니다. manifest.yaml에 두 human 게이트가 별도로 정의돼 있고, 이 프로토타입은 그중 문서승인만 다룹니다.)

## 여는 방법

`승인화면_프로토타입_v1.html` 파일을 **더블클릭**하면 기본 웹 브라우저(Chrome, Edge 등)로 바로 열립니다.
인터넷 연결이 없어도 열리고 작동합니다(외부 폰트·이미지·스크립트를 전혀 불러오지 않음 — 파일 하나에 모든
화면·글꼴 설정·동작 코드가 들어 있습니다).

열리면 "문서승인 대기" 목록이 먼저 보입니다. 카드를 누르면 그 건의 상세 승인 화면으로 들어가고, 왼쪽 위 ← 버튼을
누르면 목록으로 돌아갑니다.

## 데이터 출처

- **대기 큐 12건**: `test/새문서_합성테스트_12건.yaml`을 그대로 옮겼습니다(고객명/현장명/방문일/사용약품/문서요청/
  사진첨부/승인상태/만족도). 이 파일 자체가 "run.py 구조 테스트 전용 합성 데이터"라고 주석에 명시돼 있어, 실제
  방역&클린 고객 데이터가 아닙니다. 픽스처의 "승인상태: 승인"은 스키마 enum(초안/승인대기/승인완료/반려)에 맞춰
  "승인완료"로만 명칭을 정규화했고 값 자체는 바꾸지 않았습니다.
- **사업자 정보**: `golden_set.yaml`의 2026-07-15 대표님 인터뷰 실답변 — 상호 방역&클린 / 대표자 김남오 /
  사업자등록번호 404-46-00762.
- **문서 필드 구조**: `templates/완료보고서_템플릿A_간단형.md`, `templates/소독증명서_템플릿.md`의 항목 구성을
  그대로 따랐습니다(소독증명서 법정 9필드 포함).
- **반려사유 2갈래(표현오류/데이터오류)**: `manifest.yaml`의 문서승인 stage `on_fail.human_reject.return_to`
  매핑(표현오류→문서생성봇, 데이터오류→현장완료입력봇)을 그대로 반영했습니다.
- 픽스처에 아예 없는 값(실시면적/소재지(현장)/관리자확인/담당자/다음 방문 예정일)은 지어내지 않고 빈 칸 +
  "⚠ 확인 필요" 표시로만 남겨뒀습니다. 처리내용·소독내용종류처럼 픽스처의 실제 필드(현장명·사용약품·방문일)에서
  조합해 만든 문구는 "AI 초안" 배지로 구분했습니다.

## 화면에 있는 내용

- **대기 큐**: 12건이 상태(승인대기 → 반려 → 초안 → 승인완료 순)로 정렬되고, 상단에 문서 기준 상태별 건수
  배지가 나옵니다. 각 카드에 요청된 문서 종류(완료보고서/소독증명서)별 배지가 따로 붙습니다 — 한 건에서
  완료보고서는 승인, 소독증명서는 반려인 것처럼 문서 종류마다 독립적인 상태를 가질 수 있다는 manifest.yaml의
  설계를 그대로 화면에 반영했습니다.
- **상세 승인 화면**: 문서 두 종류를 모두 요청한 건은 상단에 탭(완료보고서/소독증명서)이 뜹니다 — 방역_UI참조.md의
  "이 화면만 예외적으로 탭 사용 가능" 규칙을 따른 것입니다. 사업자 정보, 소독증명서 법정 9필드(1.상호~9.소독실시자),
  완료보고서 필드가 카드별로 나뉘고, AI가 채운 값은 "AI 초안" 배지, 픽스처에 없는 값은 주황 테두리 + "⚠ 확인 필요"
  배지로 구분됩니다(공통_패턴.md 규칙 그대로). 모든 입력칸은 승인완료 전까지 직접 수정 가능합니다.
- **승인/반려**: [승인확정](초록)/[반려처리](빨강) 두 버튼만 있고 "보류" 상태는 만들지 않았습니다(회사 헌장 원칙 —
  방역_UI참조.md ⑥). 반려를 누르면 사유(5자 이상 필수)와 반려 종류(표현오류/데이터오류)를 반드시 선택해야 하고,
  선택에 따라 되돌아갈 stage가 화면에 바로 표시됩니다. 표현오류는 이 화면에서 [문서 재생성] 버튼으로 재생성까지
  시뮬레이션할 수 있지만, 데이터오류는 현장완료입력봇(현장 재입력) 화면이 필요해 이 프로토타입 범위 밖이라
  재생성 버튼을 의도적으로 비활성화해뒀습니다 — 안 되는 걸 되는 것처럼 보이지 않게 하기 위해서입니다.
- **잠금**: 승인 완료되면 🔒 배지가 붙고 모든 입력칸이 비활성화됩니다.
- **발송 게이트**: 각 건 하단에 "완료 안내 발송(문자장부봇)" 카드가 있고, 요청된 문서가 전부 승인완료 상태여야만
  [완료안내발송] 버튼이 켜집니다. manifest.yaml이 "가장 중요한 안전장치"라고 부르는 "승인완료 아닌 문서는 절대
  발송하지 않는다" 철칙을 화면에서도 그대로 강제합니다.
- **초안 → 검수 요청 다리**: 이 프로토타입은 문서 초안 화면(⑤)을 따로 만들지 않았지만, "초안" 상태 문서에
  [검수 요청] 버튼을 남겨 ⑤→⑥ 전환만 최소한으로 시뮬레이션할 수 있게 했습니다.

## 방역_UI참조.md 대비 구현한 것 / 뺀 것

**구현한 것** (방역_UI참조.md ⑥ 대표자 검수/문서승인 부분)
- 대기 큐 목록, 문서 상세 + approval 블록(status/version/approved_by/rejection_reason/version_history) 시뮬레이션
- 승인/반려 동작, 반려 시 사유 입력 + return_to 되돌리기(manifest.yaml 정의 그대로)
- 큰 글씨·모바일 우선 반응형·기술 용어 없는 화면(공통_패턴.md 규칙 준수: 상태색 매핑, 승인카드 6요소, 한글
  명사형 상태값, 동사+명사 버튼 문구)
- 소독증명서 법정 9필드 전체, 완료보고서 필드, 사업자 정보 자동 채움(상호=상호명 동일값 안내 포함)

**뺀 것** (범위 밖, 별도 화면으로 남김)
- ① 홈/오늘일정, ② 오더메모 입력, ③ 고객DB 상세, ④ 현장 입력, ⑤ 문서 초안(전용 화면 — 이 프로토타입은
  "초안" 상태를 목록에 보여주기만 하고 최소한의 [검수 요청] 다리만 둠), ⑦ 보관함, ⑧ 사후관리는 만들지
  않았습니다. 요청 범위가 "문서승인 화면을 중심으로"였기 때문입니다.
- 오프라인 배지(현장입력 전용 UI라 이 화면과는 무관 — 방역_UI참조.md에도 ④ 현장 입력 화면 소관으로 명시)
- D-day 배지: 새문서_합성테스트_12건.yaml의 D_day 필드는 "오늘(2026-07-07) 대비"로 계산된 값인데 시연 시점
  기준으로는 의미가 달라지고, 애초에 방역_UI참조.md에서도 D-day는 ① 홈 화면 소관이라 이 화면에서는 뺐습니다.

## 목업의 한계 (실제로 안 되는 것)

1. **실제 데이터와 연결되지 않습니다.** 화면의 12건은 새로고침하거나 파일을 다시 열면 전부 초기 상태로
   되돌아갑니다. 승인/반려/검수요청/발송 결과는 이 브라우저 탭 메모리에만 잠깐 남고 저장되지 않습니다 —
   실제 서비스에서 필요한 고객DB저장봇 연결은 없습니다.
2. **문서생성봇/현장완료입력봇은 실행되지 않습니다.** [문서 재생성], [검수 요청], [완료안내발송] 버튼은
   모두 상태 전환만 시뮬레이션할 뿐, 실제 AI 모델 호출이나 문자 발송은 일어나지 않습니다.
3. **실시면적/소재지/관리자확인/담당자/다음 방문 예정일은 항상 빈 칸입니다.** 픽스처 데이터 자체에 이 값들이
   없어서인데, 실제 운영에서는 현장완료입력봇 단계에서 채워질 값입니다.
4. **문서 종류별 승인상태를 픽스처가 따로 주지 않아, 완료보고서/소독증명서를 둘 다 요청한 건은 두 문서에 같은
   초기 상태를 넣었습니다.** 승인/반려는 각 문서를 화면에서 개별적으로 눌러야 서로 달라집니다(그렇게 되도록
   설계는 이미 문서 종류별 독립 상태로 만들어져 있습니다).

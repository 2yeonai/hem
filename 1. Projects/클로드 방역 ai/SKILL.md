# 방역&클린 AI 운영관리 스킬 (pest-control-ai-ops)

> **새 기능을 설계·제안하기 전에 [[1. Projects/클로드 방역 ai/기능_인덱스|기능_인덱스]]부터 확인할 것.** "이미 있는 기능"을 다시 설계하는 사고를 막기 위한 단일 진입점(정부지원 스킬에 이미 있는 패턴, decision-log §19 판정2로 이 스킬에도 신설, 2026-07-17).

이 파일은 이전에 존재하지 않았음 — 이번(2026-07-07) 작업에서 처음 작성함. 정부지원사업/꽃집 스킬의 SKILL.md 구성(트리거 조건/하는 일/알려진 한계/유지보수)을 그대로 따름.

## 언제 이 스킬을 쓰는가 (트리거 조건)
manifest.yaml의 `triggers` 섹션과 동일 — 두 가지 방식으로 시작된다.

- **event 트리거 (source: inquiry)**: 전화/카톡/문자 등으로 신규 견적 문의·방역 요청이 들어왔을 때. 진입 stage는 `수집봇`.
- **schedule 트리거 (interval: visit_cycle)**: 정기 고객사(약 110곳)의 방문주기가 도래했을 때. 진입 stage는 `리마인더봇`. 실제 날짜 확인은 이 스킬 밖의 별도 소형 스케줄러 프로그램이 주기적으로(예: 매일 1회) 호출해서 트리거함 — 이 스킬 자체에는 스케줄링 엔진이 없음.
- 부가적으로 파이프라인 도중 `report_requested`(완료보고서 요청) 또는 `certificate_requested`(소독증명서 요청) 중 하나라도 참이면 `문서생성봇`이 조건부로 실행되고(run_if), 승인이 필요한 문서(approval_required=true — 완료보고서·소독증명서 둘 다 해당)가 있으면 곧바로 `문서승인`(kind: human) 게이트로 이어짐. 반려되면 문구/서식 문제는 `문서생성봇`부터, 데이터 자체가 틀린 경우는 `현장완료입력봇`부터 재작업.

## 언제 쓰면 안 되는가
- **완료보고서·소독증명서를 법적 효력이 있는 최종 문서로 그대로 제출/발급하는 용도** — 아래 "알려진 한계" 참고, 법정 필수 기재항목이 아직 확정되지 않아 초안(draft) 이상으로 취급하면 안 됨.
- **여러 사용자(대표/직원 외 추가 인력)의 권한을 구분해야 하는 운영** — 지금 설계는 "대표자 1인 최종검수" 전제(사업계획서 실제 인력 2인 체제 기준)라 권한체계가 필요한 조직에는 적용 범위 밖.
- **현장 인터넷 연결이 불안정하거나 완전 오프라인인 환경에서의 필수 의존** — 오프라인 동작 범위가 설계되지 않음.
- 현장분리봇/견적금액산정봇/현장사진봇 세 가지 기능이 필요한지 여부를 아직 검증하지 않은 상태로 "이 스킬은 다중현장 분리나 견적 자동산정까지 다 된다"고 안내하는 것 — 지금은 자리만 마련된 상태(run_if로 꺼져 있음).

## 이 스킬이 하는 일 (한 문단 요약)
전화/카톡/문자로 들어온 신규 견적 문의를 받아쓰기·판정(방역요청/견적문의/일반문의/스팸/확인필요)한 뒤, 문맥 보정과 필드 구조화(고객명/공간유형/방문희망일/요청사항/문서요청여부/후속연락여부)를 거쳐 검수매니저가 confidence를 종합해 자동확정 또는 대표자 에스컬레이션으로 분기한다(1단계). 대표자가 최종 검수해 고객DB에 저장하면(2단계), 방문일정이 담당자에게 전달되고(3단계) 현장 방역 수행 완료 후 완료 메모(사용 약품, 사업자 정보 포함)가 입력된다(4단계). 고객이 요청한 경우에만 완료보고서·소독증명서 초안이 생성되고(5단계), 승인이 필요한 문서는 대표자가 별도 게이트(`문서승인`)에서 승인/반려하며 — 반려되면 문구 문제는 문서 재생성만, 데이터 문제는 현장 입력부터 다시(5.5단계) — 승인완료된 문서만 완료 안내 문자·알림톡에 실려 발송되고 만족도·불편사항을 수집한다(6단계, 사후관리). "승인완료 아닌 문서는 절대 발송하지 않는다"는 규칙은 발송 단계 자체에서 다시 한번 강제로 검증한다. 이 선형 흐름과 별개로, 정기 고객사(약 110곳)의 방문주기가 도래하면 별도 스케줄러가 리마인더봇을 깨워 같은 방문알림 단계로 합류시킨다.

## 실행 방법
**mock 실행기(`scripts/run.py`)가 있다 — 실제 AI 모델(STT/LLM)은 아직 호출하지 않고, manifest.yaml의 stages/triggers/run_if/entry_points/승인워크플로우를 그대로 따라가며 "구조가 처음부터 끝까지 한 번 통과되는지"를 확인하는 목적의 스텁 실행기다.**
```
# 스키마·manifest 정합성 검증
python3 ai공장짓기/scripts/validate_manifest.py "1. Projects/클로드 방역 ai/manifest.yaml"

# 컴파일/문법 확인
python3 -m py_compile "1. Projects/클로드 방역 ai/scripts/run.py"

# event 트리거(신규 문의) mock 실행
python3 "1. Projects/클로드 방역 ai/scripts/run.py" --trigger event

# schedule 트리거(정기 방문주기) mock 실행
python3 "1. Projects/클로드 방역 ai/scripts/run.py" --trigger schedule

# 12건 합성 테스트 + 승인워크플로우 3개 시나리오 배치 실행
python3 "1. Projects/클로드 방역 ai/scripts/test_12cases_batch.py"
```
2026-07-07 기준 배치 테스트 결과: 검증1(문서종류구분)/검증2(승인워크플로우 A·B·C 시나리오)/검증3(entry_points+날짜계산) 전부 PASS.

## 알려진 한계
- **현장분리봇/견적금액산정봇/현장사진봇 3개는 아직 켤지 안 켤지 미정이다.** manifest.yaml에 run_if 자리만 만들어 뒀고(`run_if: 미정_골든셋수집후결정`), 실제 on/off와 조건 문자열은 방역 골든셋을 확보한 뒤 결정한다. 지금 이 세 기능이 있다고 안내하면 안 된다.
- **법정 필수 문서 기재항목, 권한체계, 오프라인 동작 범위는 아직 미확정이다.** 방역&클린 사업계획서 및 지금까지의 설계 문서 어디에도 완료보고서·소독증명서의 법정 필수 기재사항, 대표자 외 추가 사용자 권한 구분, 오프라인/저연결 환경에서의 동작 범위가 명시돼 있지 않다 — 임의로 가정하지 않고 미확정으로 남겨둔다.
- **아직 실제 AI 모델(STT/LLM)은 호출하지 않는다.** run.py는 전부 mock(목업) 함수로 동작 — 골든셋 확보 후 실제 모델 호출로 교체 필요.
- **문의판정봇/보정봇의 tier(sonnet)는 꽃집 사례를 따른 잠정 판단이며 방역 데이터로 실측되지 않았다.**
- **golden_set이 아직 없다.** 스키마의 io/reads/writes 정합성은 validate_manifest.py로, 파이프라인 흐름·승인워크플로우는 mock 배치 테스트로 검증했지만, 실제 방역 텍스트/통화로 판정·구조화가 잘 되는지는 전혀 검증되지 않았다.
- **estimated_price(견적금액) 개념이 실제로 필요한 도메인 개념인지조차 확인되지 않았다** — 표준 가격표 존재 여부 미확인.
- **만족도수집봇의 타이밍**(방문 직후인지 시차를 둔 별도 트리거가 필요한지)이 불확실하다.
- **문서승인 반려 재시도 최대 횟수(현재 mock은 3회)와 version_history 보존 기간·형식이 미정이다** — 실 운영 기준 확정 필요(manifest.yaml open_questions 참고).
- **business_info(상호명/대표자명/사업자등록번호)는 시스템이 값을 저장·전달만 할 뿐, 그 값이 맞는지 검증하지는 않는다.** 완료보고서·소독증명서 템플릿과 document_draft에 실제로 연결돼 있지만(2026-07-07), 실제 발급 전 국세청 공식 사업자등록증명과의 대조는 여전히 사람(대표자) 몫이다.
- **아직 구조변경 후보로 남아 사용자 판단을 기다리는 카드가 5개 있다** — 고객/현장 분리, 법정 주기 저장방식, AI 전달 필드 최소화, 권한체계, 오프라인 정책(`fable5_카드_사전정리_2026-07-07.yaml` 참고).

## 유지보수
- 스키마는 `ai공장짓기/manifest.schema.v2.yaml`을 따른다 — 이 스킬만의 스키마 변경이 필요하면 먼저 공용 스키마 파일을 고치고 그 변경 이력을 스키마의 open_issues에 남긴다(이번 작업의 triggers/run_if/entry_points 추가가 그 선례).
- 이 폴더(`클로드 방역 ai`)에서 큰 파일을 Edit할 때는 알려진 동기화 버그(파일이 중간에 잘리는 현상)가 재현된 적이 있다 — **2026-07-07부터 자동 점검으로 대응 중**: `ai공장짓기/scripts/verify_write.py`가 Write/Edit 직후 "의도한 내용"과 "실제 디스크 내용"을 바이트 단위로 비교해, 다르면 자동으로 `.stale-*` rename + 재작성하고 HANDOFF.md에 한 줄 기록까지 남긴다. 다만 이건 진짜 후크(hook)가 아니라 "매번 호출하는 루틴"이라 호출 자체를 빼먹으면 소용없다 — 큰 파일을 고칠 때마다 이 스크립트를 부르는 걸 잊지 않는 게 핵심이다.
- 골든셋이 쌓이면 `1. Projects/클로드 꽃집 ai/golden_set.yaml`과 같은 형식(원문 보존 + 불확실 값은 null+확인필요 표시)으로 이 폴더에 별도 파일을 만든다. 빈 템플릿은 `golden_set_template.yaml` 참고.

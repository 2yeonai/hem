# 법정 요구사항 vs 기존 설계 대조표 (2026-07-07)

새 문서(`방역&클린 모바일 AI 운영관리 MVP 계획안.md`)의 법정 서식·보존기간·소독주기·권한체계·오프라인정책 내용을
기존 `manifest.yaml`(17 stage, triggers/run_if/entry_points 포함)과 대조한 결과입니다.

**이 문서는 판단만 정리한 것이며, 어떤 것도 아직 결정하지 않았습니다.** (c)그룹은 근거 1~2문장만 적었고
사용자 승인 없이는 스키마를 바꾸지 않습니다.

---

## (a) 기존 구조 그대로 반영 가능

| 새 문서 요구사항 | 대응하는 기존 구조 |
|---|---|
| 오더메모(상담/현장 입력 구조화) | 수집봇 → 전사봇 → 문의판정봇 → 보정봇 → 문의정리봇 |
| 고객DB 누적관리 | 고객DB저장봇 |
| 정기방문 D-day 리마인더(자리) | 리마인더봇 (schedule 트리거의 entry_stage로 이미 선언돼 있음) |
| 완료보고서·소독증명서 초안 생성 | 문서생성봇 (`run_if: 고객이_문서를_요청함`) |
| 대표자 승인 후에만 확정 | 대표자검수(human) — 이미 "승인 전 발송 불가" 순서로 구성돼 있음 |
| 만족도·불편사항 수집 | 만족도수집봇 |
| "AI는 초안·구조화만, 확정은 대표자" 원칙 | 전체 stage 순서 자체가 이미 이 원칙을 따름(문서생성봇은 대표자검수 이후 단계) |

## (b) 필드만 추가하면 되는 것 (구조는 안 바꿔도 됨)

| 새 문서 요구사항 | 추가할 위치 | 비고 |
|---|---|---|
| 소독증명서 9개 법정 필드(상호/실시면적/소재지/관리자확인/소독기간/소독내용종류/약품사용내용/발급일/소독실시자) | `document_draft` 객체 내부 필드 | 객체 내부 필드 추가라 shared_context 구조 자체는 안 바뀜 |
| 사용 약품명 | 신규 필드(예: `used_chemicals`) | 현재는 `site_completion_note`(자유 텍스트)에 뭉쳐 있음 — 별도 필드로 분리 가능 |
| 사진 첨부 개수·설명 | `site_photo`를 배열/객체로 확장 | 현재 uri 문자열 하나뿐 — 필드 타입 확장 수준 |
| 문서 잠금 여부·보관 위치·보존기간(2년) | `document_draft` 내부 필드 | 신규 필드 추가로 충분 |
| 반려사유 | `대표자검수`의 `manual_edits` 옆에 신규 필드 | `approval_action`에 "반려" 값 추가는 enum 값 확장 수준 |
| 감사로그 상세화(누가/언제 수정·승인·반려) | 기존 `history_log` 필드 내용 확장 | 필드는 이미 있음, 안에 담는 내용만 늘어남 |

## (c) 구조 변경이 필요할 수도 있는 것 — 근거만, 결정 안 함

새 문서의 "스키마 관련 유보 사항"에 해당하는 5가지입니다. 각 1~2문장 근거만 적었습니다.

| # | 항목 | 근거 (1~2문장) |
|---|---|---|
| 1 | **고객/현장 분리** | 지금 `customer_record`는 고객 정보와 현장 정보가 한 덩어리 object로 저장돼, 고객 한 명이 현장을 여러 곳 갖거나 한 현장에 방문 이력이 여러 번 쌓이는 경우를 표현하려면 엔터티를 나눠야 할 수 있음. |
| 2 | **법정주기 저장방식** | 지금 `visit_cycle`은 "격월/분기" 같은 문자열 하나뿐인데, 시설유형별로 법정 소독주기가 다르게 적용된다면 시설유형-주기 매핑 구조가 별도로 필요할 수 있음(다만 새 문서에도 시설별 법정주기 원문은 확인되지 않음). |
| 3 | **승인이력** | 지금 `approval_action`은 그 순간의 액션 하나만 담는 단일 필드라, "누가 언제 승인·반려했는지"를 누적 기록하려면 리스트/로그 형태의 별도 구조가 필요할 수 있음. |
| 4 | **템플릿버전** | 지금 `document_draft`는 매번 새로 만들어지는 단일 object라 버전 번호나 재발행 시 이전 버전과의 관계(v1→v2)를 추적할 개념이 없어, 새 문서의 "재발행 시 새 버전 생성" 요구를 반영하려면 구조 확장이 필요할 수 있음. |
| 5 | **AI 전달 필드 최소화** | 지금 일부 stage(예: 검수매니저)는 원본 필드를 폭넓게 읽는데, 새 문서의 "최소수집·목적제한" 원칙을 지키려면 어떤 stage가 어떤 필드까지 볼 수 있는지 접근범위를 다시 설계해야 할 수 있음. |

### (c) 추가로 발견된 후보 (5개 목록 밖, 3단계 테스트에서 재확인 예정)

| 항목 | 근거 |
|---|---|
| 문서요청 단일 boolean(`document_requested`) | 완료보고서만/소독증명서만/둘 다 요청하는 3가지 경우를 구분할 방법이 지금은 없음(불리언 하나뿐) — 3단계 테스트에서 실제로 확인함. |
| 승인상태 4종(승인완료/반려/승인대기/초안) | 지금 `approval_action`은 "검수저장\|방문확정" 두 값만 정의돼 있어 4가지 상태를 다 표현할 수 있는지 3단계에서 확인함. |
| 권한체계(현장입력자/대표자/관리자 3역할) | 지금은 `대표자검수`(human) stage 하나만 있고 "현장입력자"가 별도 역할/stage로 구분돼 있지 않음 — 4단계 설계메모에서 별도로 다룸(코드 반영은 안 함). |
| 오프라인 동기화 정책 | 지금 manifest/run.py 어디에도 오프라인 임시저장·충돌처리 개념이 없음 — 4단계 설계메모에서 별도로 다룸(코드 반영은 안 함). |

## 참고 — 새 문서 자체도 확정하지 않은 부분

새 문서 원문에 `unspecified—require confirmation`으로 명시된 항목(법정 서식 별지번호, 보존기간 조문 표현 등)은
이 대조표에서도 그대로 미확정으로 남겼습니다. 국가법령정보센터 원문·관할 보건소 서식으로 재확인 전에는
(b)그룹의 "법정 필드"도 최종 확정이 아니라 초안 수준입니다.

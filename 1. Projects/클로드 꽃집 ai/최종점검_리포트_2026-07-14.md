# 꽃집 파이프라인 local 6봇 + run_pipeline.py 구현 최종점검 리포트
- 날짜: 2026-07-14
- 점검 대상: 12봇_kind분류표.yaml의 kind:local 7개 stage 및 신규 run_pipeline.py

## 무엇을 했나

12봇_kind분류표.yaml을 전체 정독해 14 stage의 kind/depends_on/io.reads/io.writes를 확인했다. `summary.local` 목록(collection_bot, transcription_bot, storage_bot, print_prep_bot, dispatch_bot, delivery_photo_bot, sms_ledger_bot, 총 7개)에서 storage_bot은 이미 구현·테스트돼 있었으므로, 나머지 6개를 새로 구현했다.

**주의**: 작업 지시문에는 "order_classification_bot/order_split_bot도 kind=local 참고 패턴"이라는 설명이 있었으나, yaml 원문을 확인한 결과 이 둘은 실제로는 **kind: model**(tier: sonnet)이다 — 다만 실제 LLM 연동 전 규칙기반 임시 버전으로 이미 구현돼 있어서 "코드 스타일 참고용"으로는 여전히 유효했다. 추측하지 않고 yaml을 기준으로 판단했다는 점을 기록해둔다.

## run_pipeline.py 실행 결과

신규 작성한 `code/run_pipeline.py`는 12봇_kind분류표.yaml의 depends_on 그래프를 위상정렬해 실행 순서를 계산한다(순서를 코드에 하드코딩하지 않음 — stop_rule 1 준수). kind별 처리:
- kind:local 7개 → 실제 함수 호출
- kind:model 중 이미 구현된 2개(order_classification_bot, order_split_bot) → 규칙기반 버전 실제 호출
- kind:model 중 미구현 4개(correction_bot, order_draft_bot, ribbon_price_bot, review_manager_bot) → "여기서 실제 LLM 호출 자리" 로그만 남기고 값 임의로 안 채움
- kind:human(human_reviewer) → "여기서 사람 승인 대기" 로그 남기고 데모용으로만 자동 통과(실제 사람 승인 아님을 로그에 명시)

golden_set.yaml 6건(001~006, 005/006은 같은 call_007 원문을 공유하므로 기존 테스트 관례대로 5개 케이스로 실행)을 전부 실행한 결과:

- **001(SMS 주문)**: collection→transcription→classification(주문전화)→split(완료,1개)→[mock 4단계]→[mock 사람승인]→storage 저장→print_prep→dispatch→delivery_photo→sms_ledger(bundle_status=완료) 전 stage 통과.
- **002(안부 전화)**: classification에서 "일반통화"로 판정 → stop_condition에 따라 여기서 정상 종료(설계대로, 저장 없음).
- **003("꽃 심으러 감")**: classification이 "주문가능성있음"으로 오판정(규칙기반의 기존 알려진 한계, test_order_classification_bot.py의 KNOWN_LIMITATIONS과 동일 케이스) → 끝까지 진행됨. 새로운 버그 아님.
- **004(행사 일정 문의)**: "일반통화"로 정상 종료.
- **005/006(call_007, 부곡면장/북면장 통화)**: "주문전화"로 판정, split_status는 "확인필요"(규칙기반이 여러 건 가능성만 감지 — 기존 알려진 한계)로 나오지만 segment 1개로 끝까지 진행, storage 저장까지 정상 통과.

최종 출력: `전체 파이프라인 스모크 테스트: PASS (알려진 한계 1건 제외)`, 종료 코드 0.

## 아직 안 된 것 (알려진 한계, 이번 세션 범위 밖)

1. kind:model 4개(correction_bot/order_draft_bot/ribbon_price_bot/review_manager_bot) — 실제 sonnet/haiku LLM 호출 없음. run_pipeline.py에서 로그만 남기고 값은 None/원문 그대로 pass-through.
2. kind:human(human_reviewer) — 실제 사람 승인 화면/버튼 없음. run_pipeline.py에서 데모 목적으로만 자동 "검수저장" 처리.
3. transcription_bot의 STT/OCR — 실제 API 키 미연동, 항등(identity) 목업.
4. sms_ledger_bot의 문자 발송 — 실제 게이트웨이(알리고 등) 미연동, mock 항상 성공.
5. order_classification_bot/order_split_bot의 규칙기반 한계(003 오판정, call_007 자동 2분리 불가) — 2026-07-07부터 이미 문서화된 기존 한계, 이번에 새로 발견된 것 아님.

## 환경 이슈

run_pipeline.py 및 decision-log_12to14봇.md를 작성하는 과정에서 CLAUDE.md에 문서화된 mid-byte 파일 잘림 버그가 재현됐다(Edit 도구에서 1회, **Write 도구에서도 1회** — 기존 문서는 Edit만 언급했으나 이번에 Write도 같은 증상을 보이는 것을 확인). 규칙대로 `.stale-<timestamp>`로 rename 후 bash heredoc으로 재작성해 복구했다. 백업 파일(삭제하지 않음): `code/run_pipeline.py.stale-1783960900`, `decision-log_12to14봇.md.stale-1783961098`.

또한 `code/` 폴더 안의 일부 테스트 산출물(test_dispatch.json, test_print_prep.json, test_sms_ledger.json, run_pipeline_demo.json)이 fuse 마운트 권한 문제로 완전 삭제가 안 되고 빈 DB({"orders": {}, "workflow_events": []}) 상태로 남아있다 — 각 테스트가 시작 시 reset_db()로 초기화하므로 실행 결과에는 영향 없다(방역/정부지원 프로젝트에서도 동일하게 문서화된 환경 제약).

## 별도 발견 (이번 점검 범위 밖, 알려드리기만 함)

프로젝트 루트의 `HANDOFF.md`가 여전히 gov-support-matching-skill 내용이다. 2026-07-13 R4 정리에서 SKILL.md/manifest.yaml/scripts/test/gov-support-skill/은 `4. Archive/꽃집폴더_정부지원사본_2026-07/`로 이동됐는데 HANDOFF.md만 그때 누락된 것으로 보인다. 이번 작업 결과는 이 리포트와 decision-log_12to14봇.md에 기록했고, 잘못 남은 HANDOFF.md는 손대지 않았다.

## 결론

kind:local 7개 stage 전부 구현·테스트 완료(신규 6개, 40/40 체크 PASS + 기존 storage_bot 12/12 PASS 유지). run_pipeline.py로 14 stage 파이프라인 전체가 golden_set 6건 기준 end-to-end로 최소 1회 정상 실행됨을 실측으로 확인했다(PASS, 알려진 한계 1건 제외). 남은 일은 kind:model 4개 + kind:human 1개에 실제 LLM/승인 UI를 붙이는 것 — 전부 known limitation으로 명시했고 임의로 값을 지어내지 않았다.

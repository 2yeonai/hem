## 이번 세션에서 한 일 (요약)

1. **PDF 40페이지 전량을 원본 해상도로 추출.** `pdftoppm`/`pdfimages`(poppler-utils)는 이 환경 bash PATH에 실제로는 없었음(당초 지시와 다름) — 대신 `pip`이 이미 갖고 있던 **PyMuPDF(`fitz`)**로 각 PDF 페이지의 embedded JPEG을 원본 그대로(재압축 없이) 추출. 실제 파이썬 실행파일 경로 주의: `C:\Users\82106\AppData\Local\Programs\Python\Python314\python.exe`(기본 `python3`/`pip` 커맨드는 WindowsApps 스텁이라 패키지가 안 잡힘 — 반드시 이 전체경로로 호출해야 pymupdf/pypdf/pillow가 보임).
2. **각 페이지 방향 확인.** 원본 이미지는 세로(portrait)로 저장돼 있지만 실제로는 가로로 눕혀 찍은 스프레드 사진 — `PIL Image.rotate(90, expand=True)`(반시계 90도)로 회전해야 정방향(제목이 위, 페이지 번호가 아래)이 됨.
3. **PDF 페이지 순서 ↔ 물리 페이지 번호 매핑을 40장 전부 육안 확인으로 완성.** 목차 페이지 3개(Cervical Part=물리33p, Trunk Part=물리48p, Hip Part=물리62p)를 찾아 "기법명→페이지" 전체 목록을 확보했고, 목차가 없는 Shoulder 구간(물리3~32p)은 순서대로 나열돼 있어 직접 다 읽으며 확인. **PDF 페이지 순서와 물리 페이지 순서가 다르다는 조사보고의 결론이 재확인됨**(예: PDF 15페이지=물리31-32, PDF 16페이지=물리35-36으로 점프).
4. **소책자 전체 구조 확정: Shoulder(3~32p) → Cervical(33~47p) → Trunk(48~61p) → Hip(62~83p) 4개 섹션뿐**. Knee/Ankle/Foot 파트, 복부/골반기저 개별 근육, 다열근 등은 이 소책자에 **원래 없음** — 카드 중 가자미근·비복근·전경골근·후경골근·장/단비골근·슬와근·장무지굴근/신근·장지굴근/신근·중간광근·내측광근·봉공근·복직근·복횡근·내복사근·외복사근·골반기저근·다열근·단내전근(단독)은 이번 소책자에서 이미지를 못 찾음 — **억지로 만들지 않고 텍스트만 유지**(지시사항 7번 그대로 이행).
5. **누락 확인(촬영 자체가 안 된 물리 페이지):** 물리 71~72페이지(Hamstring MET2-knee, Adductor magnus ART)가 40장 스캔 어디에도 없음 — 앞뒤 페이지(69-70, 73-74)는 다 있는데 이 스프레드만 원본 PDF에 없음(촬영 누락으로 추정). 대내전근 단독 ART, 햄스트링 무릎단계 MET는 이미지 확보 못 함.
6. **크롭:** 모든 필요한 물리 페이지를 좌/우 절반으로 분할(스프레드 폭 절반 기준)까지 완료. **사진 박스만의 2차 크롭(둥근 모서리 마커 안쪽)은 시간 관계상 생략** — 대신 절반 페이지 전체(제목+사진+Contact Point/Tension·Compression/Barrier·Resistance 필드+페이지번호)를 그대로 씀. 이유: 이 필드들 자체가 "손 위치·압력 방향" 정보라 F3 목적에 오히려 도움이 되고, 마커 좌표를 페이지마다 수동으로 잡는 것보다 훨씬 빠름. 필요하면 다음 세션에서 `app/images/*.jpg`를 원본(`테크닉 소책자.pdf`)에서 다시 정밀 크롭 가능.
7. **이미지 최적화:** 900px 폭으로 리사이즈 + JPEG quality 78 재압축 (72장 기준 25MB → 8.6MB, PWA 오프라인 캐시 고려).

9. **카드 35장의 frontmatter+본문 갱신 완료:**
   - `책_페이지:` 필드를 실측값으로 교체 + `(2026-07-19 소책자 실측 확인)` 태그 추가.
   - 새 필드 `F3_이미지: [../app/images/xxx_1.jpg, ...]` 추가(기존 배열형 필드와 동일 문법).

import fitz
doc = fitz.open('테크닉 소책자.pdf')
for i in range(doc.page_count):
    xref = doc[i].get_images(full=True)[0][0]
    data = doc.extract_image(xref)['image']
    open(f'pdfpage_{i+1:02d}.jpeg','wb').write(data)

## 이어서 진행 (2026-07-19, 후속 세션) [hyemi]

지시받은 우선순위 1~3번을 순서대로 처리. `app/viz.js`·`app/index.html`은 지시대로 건드리지 않음(둘 다 다른 세션이 3D 뷰어 관련으로 이미 수정 중이었음 — `git diff --stat`로 확인, 내가 만든 변경분이 아님).

### 1. `app/app.js`에 F3 이미지 렌더링 로직 추가 — 완료, 실기기(브라우저)로 확인 완료

- `app/app.js`의 `renderTechniqueDetail()` 함수(테크닉카드 상세 화면 담당) 안에 새 헬퍼 `f3ImagePath()`(`../app/images/xxx.jpg` → `images/xxx.jpg` 경로 변환)와 `renderF3Images()`(F3_이미지 배열을 실제 `<img>` 태그로 렌더링)를 추가하고, 탭 카드 바로 아래에 "📷 F3 참고 이미지" 카드로 삽입.
- 기존에 "F3 참고 이미지 (소책자)"라는 섹션이 이미 본문(마크다운) 안에 `![alt](경로)` 형태로 들어있었는데, 이걸 그대로 텍스트 탭으로 두면 이스케이프돼 `![...](...)` 글자 그대로 깨져 보이는 문제가 있어서, 이 섹션은 텍스트 탭 자동생성 목록에서 제외하고 대신 F3_이미지 배열 기반의 실제 이미지 렌더링으로 대체.
- `app/style.css`에 `.f3-images`/`.f3-img` 스타일 최소 추가(그리드 배치 + 클릭하면 확대 토글). 3D 뷰어 CSS 블록(`/* ---- 3D 뷰어 ---- */`) 앞부분에 추가해 그 섹션은 건드리지 않음.
- **확인 방법**: `app/server.py`를 로컬(포트 8901)로 띄우고 Claude Browser 도구로 실제 접속해 확인. 처음엔 이미지가 안 뜨는 것처럼 보였는데(naturalWidth 0) — 원인은 코드 버그가 아니라 이 브라우저 탭에 **이전 세션이 등록해 둔 서비스워커(`sw.js`)가 옛날 `app.js`를 캐시해서 계속 그걸 서빙**하고 있었던 것(개발 중 흔한 PWA 캐시 문제). 서비스워커 등록 해제 + 캐시 삭제 후 재접속하니 정상 작동. 실제 사용자가 새로 앱을 켤 때는 문제 없음(캐시가 아직 없으므로) — 다만 **이미 이 앱을 써본 적 있는 실제 기기(혜미 폰 등)에서는 서비스워커가 옛 파일을 계속 보여줄 수 있어서, 앱을 새로고침해도 안 바뀌면 "브라우저 데이터 지우기" 또는 앱 재설치가 필요할 수 있음** — 다음 배포/공유 시 참고.
  - `technique-20-DNF`(이미지 1장), `technique-03-견갑거근`(3장 공유), `technique-52-가자미근`(F3 이미지 없는 카드) 세 가지 케이스 모두 브라우저에서 접속성 트리(accessibility tree)로 직접 확인 — 이미지 정상 표시, naturalWidth 900(정상 로드), 이미지 없는 카드는 섹션 자체가 조용히 생략됨(레이아웃 깨짐 없음).
  - 3D 뷰어 라우트(`#/model/technique-20-DNF`)도 콘솔 에러 없이 정상 작동하는 것 확인(F3 작업이 3D 뷰어에 영향 없음).

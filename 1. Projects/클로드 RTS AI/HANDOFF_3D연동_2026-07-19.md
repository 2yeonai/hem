### 막힌 지점: GLTFLoader가 ES 모듈이라 구조 변경 필요
- `app/vendor/three.min.js`는 UMD 빌드(`<script src=...>`로 전역 `THREE` 노출)인데, three.js r149의 `GLTFLoader`는 **ES 모듈 전용**(`import ... from 'three'`, `../utils/BufferGeometryUtils.js` 등 상대경로 의존)이라 지금 방식과 안 맞음. r149에는 예전의 UMD용 `examples/js/loaders/GLTFLoader.js`도 이미 제거되어 없음(404 확인함).
- 즉 근육 실제 렌더링을 붙이려면 **`three.module.js`(ESM 빌드) + GLTFLoader.js + 의존 유틸 파일들을 받고, `index.html`에 importmap 설정 + `app.js`/`viz.js`를 module 방식으로 전환**하는 구조 변경이 필요함. 이건 기존에 잘 작동하는 3D 마네킹 뷰어(포즈 5종, 체인 연결 등)에 영향을 줄 수 있는 더 큰 리팩토링이라, 이번 세션에서는 데이터 준비까지만 끝내고 멈춤(브라우저에서 회귀 검증할 시간 부족 판단).
- (참고) GLTFLoader.js(ESM, 100KB) 다운로드는 성공했음(작은 텍스트 파일이라 자동 모드 차단에 안 걸림 — 87MB 대용량 바이너리만 차단되는 것으로 보임) — 필요하면 재다운로드만 하면 됨: `curl -sL -o GLTFLoader.js https://raw.githubusercontent.com/mrdoob/three.js/r149/examples/jsm/loaders/GLTFLoader.js`

## 3D 근육 실제 형상 오버레이 — 코드 통합 완료 (2026-07-19, [hyemi])
혜미가 Blender 설치 완료 후 이어서 진행. **실제 근육 3D 렌더링이 앱에 연결됐다.**

### 한 일
1. **three.js를 ES 모듈로 전환**: `vendor/three.module.js`(r149) + `vendor/loaders/GLTFLoader.js` + `vendor/utils/BufferGeometryUtils.js` 추가(작은 텍스트 파일이라 curl 다운로드 정상 동작). `index.html`에 importmap 추가, `viz.js`/`app.js` 둘 다 `type="module"`로 전환. `vendor/three.min.js`(UMD, 기존)는 이제 안 쓰지만 삭제하지 않고 남겨둠(참고용).
2. **`app.js` 호환성 수정**: `!window.THREE` 체크로 "3D 엔진 로드 실패"를 띄우던 기존 로직이 있어, `viz.js`에서 `window.THREE = THREE`로 하위호환 노출(한 줄 추가, app.js는 최소 침습).
3. **이름 충돌 버그 발견·수정**: Blender→glTF export 시 오브젝트 이름의 공백/마침표가 자동 정규화되며 좌우 근육 이름이 충돌해(예: "Clavicular head of pectoralis major muscle.l"과 ".r"이 둘 다 "...muscle"로 겹쳐 "001"/"_1" 접미사로 뒤섞임) 매핑이 깨지는 문제를 실측(실제 브라우저에서 glb 로드해 이름 덤프)으로 발견. **해결**: `scripts/blender_export_muscles.py`를 수정해 export 직전 각 오브젝트를 `muscle_0000`~`muscle_0179` 안전 ID로 임시 개명하고, `{안전ID: 원본이름}` 대응표(`muscles_export_idmap.json`)를 별도 저장하도록 변경. `muscle_object_mapping.json`도 이 안전 ID 기준으로 재생성. `viz.js`에서도 접두어(prefix) 일치로 매칭(material 슬롯이 여러 개인 오브젝트는 `muscle_0042_1`처럼 자식 노드가 더 생길 수 있어서).
4. **좌표 정렬(근사)**: 마네킹 "서기" 자세 실측 치수(발바닥 y=-0.31, 키 2.34, `viz.js`의 `debugBounds()` 디버그 함수로 측정)에 맞춰 근육 그룹 전체를 일괄 스케일(≈1.49배)·위치 정렬. **개별 근육 단위 정밀 정합은 아님** — 전신 단위로 대략 맞춘 근사치.
5. **자세 추종**: 근육 그룹을 마네킹의 `rootPivot`(전신 회전/이동 담당 노드)의 자식으로 붙여서, "누움"/"엎드림" 등 자세를 바꾸면 근육도 마네킹과 함께 통째로 회전·이동한다. **단, 관절별 리깅은 없어 팔다리 개별 각도까지는 못 따라감**(전신 방향만 근사 추종).
6. **표시 로직**: `viz.js`의 `setHighlight(meshKeys, face, label, zAnatomy)` — 4번째 인자로 카드의 `z_anatomy` 값을 받아(`app.js`의 `renderModelDetail`에서 `mapCard(id).z_anatomy`로 전달) `assets/muscle_object_mapping.json`에서 실제 오브젝트 이름을 찾아 해당 근육만 노출(주황색 강조), 나머지는 숨김. 근육 자산은 앱 로드 후 1회만 비동기 로드해 캐시 재사용.
7. **안내 문구 갱신**: 3D 홈 화면 문구를 "부위 단위 강조(1차)만 있고 실제 형상은 다음 단계"에서 "부위 강조 + 일부 근육은 실제 형상(CC BY-SA)도 함께 표시, 관절별 자세 연동은 없음"으로 정정(`app.js` `renderModelHome`).

## 좌표 정합 버그 수정 + 근육 상시 표시로 전환 (2026-07-20, [hyemi])

혜미 지적: "3d장난쳐? 저게 너는 근육으로 보여??? 막대기로 보여??????" — 실제로 열어보니 목 부위에 회색 막대·상자 마네킹 위에 작은 주황 상자 하나만 떠 있고, "근육 형상"은 전혀 안 보이는 상태였음. 원인 조사 후 수정 완료, 브라우저 스크린샷으로 실측 검증까지 마침.

### 근본 원인 (2가지 겹쳐서 발생)

1. **좌표 정렬 버그(치명적)** — 이전 세션이 `muscles.glb`(Z-Anatomy 근육 182개 메시)를 "메시 뭉치 전체의 위아래 끝"을 "마네킹의 발바닥~머리 꼭대기"에 늘려 맞추는 방식으로 정렬했다(`MANNEQUIN_STANDING_HEIGHT / size.y` 스케일). 문제는 근육 데이터가 **목에서 끝나고 머리·손·발이 없다**는 것 — 그런데도 "머리 꼭대기까지" 늘려 맞춰버려서 모든 근육이 실제보다 위로 붕 떠 있었다. 실측해보니 예를 들어 어깨올림근(목 근육)이 마네킹 몸통보다 위, 사실상 머리 근처에 가 있었음.
   - **해결**: Z-Anatomy 원본이 "바닥 y=0인 실제 사람 키(약 1.75m) 좌표계"라는 걸 브라우저 콘솔에서 개별 근육 좌표를 직접 재서 확인(정강이근 0.04~0.41m, 대둔근 0.71~0.98m, 대흉근 1.17~1.41m, 어깨올림근 1.39~1.54m — 전부 해부학적으로 맞는 순서). 이제는 "실제 키(1.75m) 기준 스케일 + 바닥을 발바닥에 맞춤" 방식으로 정렬(`viz.js`의 `ZANATOMY_STATURE` 상수). 이 방식은 목까지만 있는 근육 데이터를 억지로 안 늘리므로 정확하다.
2. **표시 방식 문제** — 근육이 로드돼도 기본은 전부 숨김 처리였고, 선택된 근육만 불투명(opaque)으로 그려서 (a) 마네킹의 회색 상자·막대가 근육 뒤에서 그대로 다 보이고 (b) 몸 안쪽 깊은 근육(예: 견갑하근)은 겉근육에 가려 아예 안 보였다.
   - **해결**: 이제 앱을 열면 **전신 근육이 항상 살색으로 은은하게(반투명 32%) 보인다** — "몸이 진짜 근육으로 덮여 보이는" 느낌. 선택된 근육은 주황으로 밝게 강조하되 `depthTest:false`로 항상 맨 위에 그려지게 해서 몸 안쪽 근육도 또렷이 보임. 근육이 덮는 몸통·팔다리 부위의 마네킹 상자·막대는 아예 숨겨서(뒤에 "의자"처럼 비치던 문제 제거) 근육만 보이게 함. 카메라도 강조 부위(앞/뒤/옆)를 자동으로 바라보도록 각도를 맞춤(전에는 등 근육인데 정면에서 보고 있어 몸에 가려 안 보이는 경우가 있었음).

### 검증 방법 (실제로 눈으로 확인함 — 이전 세션이 "간접 검증만 하고 실제 시각 확인 못 함"이라 남겼던 한계를 해소)
- 로컬 서버(`app/server.py`)를 백그라운드로 띄우고, 진단용 임시 서버(스크래치패드, 127.0.0.1 전용, 프로젝트 파일 아님)를 하나 더 만들어 브라우저 캔버스를 `canvas.toDataURL()`로 캡처해 PNG로 저장하는 방식으로 실제 렌더링 결과를 이미지로 확인했다. `requestAnimationFrame`이 비활성 탭에서 안 도는 문제(이전 세션이 겪은 것과 동일)는 `viz.js`에 진단 전용 `debugRenderNow()` 함수를 추가해 우회(rAF 기다리지 않고 즉시 1프레임 렌더).
- 어깨올림근(목), 대흉근(가슴), 대둔근(엉덩이), 가자미근(종아리), 견갑하근(깊은 근육) 5종을 스크린샷으로 확인 — 전부 해부학적으로 올바른 위치에 뚜렷하게 표시됨. 견갑하근처럼 겉근육(대흉근)에 덮이는 깊은 근육도 잘 보임.
- 사용한 진단 서버 스크립트·캡처 이미지는 세션 스크래치패드에만 있고 저장소에는 없음(프로젝트 서버 `app/server.py`는 건드리지 않음, 검증 후 프로세스 종료함).

### 남은 개선 과제 (선택, 이번엔 범위 밖)
- 개별 근육 단위 정밀 정합은 여전히 근사치(전신 단위로 실제 키 기준 정렬한 것 — 이전보다 훨씬 정확해졌지만 근육 하나하나를 손으로 맞춘 건 아님).

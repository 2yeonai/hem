# 7. Attachments 폴더 운영 지침 (CLAUDE.md)

## 문서 목적
- ✅ Claude Code(저)를 위한 컨텍스트 제공
- ✅ `7. Attachments/` 폴더의 목적·구조·규칙을 빠르게 이해
- ✅ 작업 시 자동 참조되어 맥락 파악 가속

---

## 폴더 역할
- **7. Attachments = 바이너리 첨부파일 보관소**
- 원칙: **노트는 텍스트만, 바이너리는 여기로 · 원본 보존 · 참조는 임베드로만**
- 대상: 이미지, PDF, 스크린샷/캡처, 오디오, 기타 마크다운이 아닌 첨부파일
- 어떤 폴더(1. Projects/2. Areas/3. Resources/5. Zettelkasten 등)의 노트에서 첨부하든 실제 파일은 전부 이 폴더에 모은다 — 노트 폴더에 이미지를 직접 두지 않는다

---

## 여기 들어가는 것 / 안 들어가는 것

**들어가는 것**:
- 이미지 (`.png`, `.jpg`, `.gif` 등) — 스크린샷, 다이어그램, 사진
- PDF (`.pdf`) — 참고 문서, 스캔본
- 오디오/영상 등 기타 바이너리 파일

**안 들어가는 것**:
- 마크다운 노트(`.md`) — 이건 각 폴더(1. Projects/2. Areas/3. Resources/5. Zettelkasten/0. Docs 등)로
- 노트에서 다루는 실제 "내용"(요약, 통찰 등) — 첨부파일은 원본 그대로 보관하고, 요약/발췌는 노트 본문에 작성

---

## 파일명 규칙

- **날짜 접두사 권장**: `YYYY-MM-DD_설명.ext` (예: `2026-07-08_사업계획서-1페이지.png`)
  - 같은 날 여러 파일을 저장해도 시간순 정렬이 되고, 이름 충돌을 줄임
- **의미 있는 설명 사용**: `Screenshot 2026-07-08 143022.png` 같은 OS 기본 파일명은 리네임 후 저장 권장 — 나중에 어떤 노트에서 썼는지 파일명만 보고 유추 가능해야 함
- **주제별 하위 폴더 허용**: 파일이 많아지면 `7. Attachments/프로젝트명/`, `7. Attachments/2026/` 처럼 하위 폴더로 분류 가능 (평면 구조를 기본으로 하되, 강제하지 않음)
- 공백보다 하이픈(`-`)/언더스코어(`_`) 권장 — Obsidian 링크 문법에서 공백이 있어도 동작은 하지만, 파일 시스템/터미널 작업 시 다루기 번거로움

---

## 원본 보존 원칙

- **첨부파일은 원본 그대로 보관** — 노트에 삽입할 때 리사이즈/변환된 사본을 별도로 만들지 않는다 (Obsidian이 렌더링 크기는 임베드 문법으로 조절)
- **삭제 금지, 이동만**: 더 이상 어떤 노트에서도 참조하지 않는 첨부파일이라도 확실하지 않으면 삭제보다 보관을 우선 (4. Archive의 "삭제 금지" 원칙과 동일한 정신)
- **노트가 4. Archive로 이동해도 첨부파일은 여기 그대로 둔다** — 노트 본문의 `![[파일명]]` 링크는 파일이 이동하지 않는 한 계속 유효

---

## 노트에서 참조하는 규칙 (임베드)

- 실제 파일 복사 금지, 항상 임베드/링크로 참조:

```markdown
![[파일명.png]]                    # 기본 임베드 (원본 크기)
![[파일명.png|300]]                # 너비 300px로 축소
![[파일명.pdf]]                    # PDF 임베드 (뷰어로 표시)
![[파일명.pdf#page=3]]             # PDF 특정 페이지
[[파일명.pdf|보고서 원본]]         # 임베드 없이 텍스트 링크만 (별칭)
```

- 같은 첨부파일을 여러 노트에서 참조해도 괜찮음 — 파일은 하나만 유지하고 여러 노트가 그 파일을 가리키게 한다 (복사 금지, 참조 우선 원칙은 5. Zettelkasten의 "ZK → PARA는 복제본만" 원칙과 동일한 정신: 원본은 하나, 참조는 여러 곳)
- 노트를 옮기거나 이름을 바꿔도 Obsidian이 링크를 자동 갱신하지만, **파일 탐색기(OS)에서 직접 파일을 옮기면 링크가 깨진다** — 첨부파일 이동은 Obsidian 안에서(파일 탐색기 패널 드래그 등) 수행할 것

---

## 민감정보 주의

- **스크린샷/캡처에 개인정보·계정정보·사업자등록번호 등이 노출되는 경우가 많다** — 저장 전에 가려야 할 정보가 있는지 확인 (특히 `클로드 방역 ai`, `클로드 정부지원사업 ai`처럼 실제 고객/사업자 데이터를 다루는 프로젝트에서 캡처를 뜰 때)
- 민감한 첨부파일은:
  - 꼭 필요한 경우가 아니면 애초에 저장하지 않기
  - 저장이 필요하면 파일명에 그 사실을 표시(예: `_민감` 접미사)하거나 별도 하위 폴더(`7. Attachments/민감/`)로 분리해 실수로 공유되지 않게 함
  - 이 볼트를 클라우드 동기화(OneDrive 등)하고 있다는 점을 감안 — 동기화 자체가 곧 외부 저장이라는 걸 잊지 말 것
- 저작권이 있는 자료(구매한 PDF, 스캔한 책 등)는 개인 학습 목적으로만 사용하고 재배포하지 않는다

---

## 대용량 파일 관리

- **PDF/이미지 원본이 큰 경우**: 필요하면 노트에는 저해상도 미리보기용 이미지(`![[파일명.png|400]]`)만 임베드하고, 원본 고해상도 파일은 그대로 두되 굳이 매번 렌더링하지 않도록 링크(`[[파일명|원본 보기]]`)로 대체
- **정기 정리**: 어느 노트에서도 더 이상 참조하지 않는 첨부파일이 쌓이면 분기 1회 정도 점검 (고아 첨부파일 확인 — 아래 Dataview 참고)
- **클라우드 동기화 부하**: 이 볼트는 클라우드로 동기화되므로, 대용량 파일이 많아지면 동기화 지연/충돌 가능성이 커진다 — 정말 볼트에 넣어야 하는 파일인지, 외부 저장소(별도 클라우드 폴더 등) 링크로 대체 가능한지 먼저 판단

---

## Dataview 활용 예시

### 고아 첨부파일 후보 (전체 노트에서 참조 확인은 Dataview만으로는 완전 자동화가 어려움 — 파일 목록만 나열)

```dataview
LIST
FROM "7. Attachments"
SORT file.mtime DESC
```

### 최근 추가된 첨부파일 (30일 이내)

```dataview
LIST file.mtime as "추가일"
FROM "7. Attachments"
WHERE file.ctime >= date(today) - dur(30 days)
SORT file.ctime DESC
```

> 첨부파일 자체는 frontmatter가 없는 바이너리라 `type`/`status` 같은 필드 기반 쿼리는 불가능 — 파일명/날짜 기준 조회만 가능하다.

---

## 금지/주의(안티패턴)

- 노트 폴더(1. Projects/2. Areas/3. Resources/5. Zettelkasten 등)에 이미지를 직접 저장 — 전부 `7. Attachments/`로
- 같은 파일을 여러 번 복사해서 여러 폴더에 흩어놓기 — 원본 하나 + 임베드 참조만
- 민감정보가 그대로 보이는 캡처를 확인 없이 저장
- OS 파일 탐색기에서 직접 첨부파일을 옮기거나 이름 변경 (Obsidian 링크 깨짐 — Obsidian 안에서 이동)
- 첨부파일을 "혹시 몰라서" 무한정 쌓아두고 정리하지 않기

---

## 유지보수 체크리스트(분기 1회)

- 어느 노트에서도 참조되지 않는 파일(고아 첨부파일) 확인 및 정리 판단(삭제 vs 보관)
- 파일명 규칙(날짜 접두사, 의미 있는 설명) 준수 여부 점검
- 민감정보 포함 파일 재확인 — 필요 없어진 캡처는 삭제 검토
- 대용량 파일 목록 확인 — 저해상도 대체본으로 바꿀 대상 선별

## §0. 모델 티어표 (제품별 — 이름이 아니라 티어로 배정한다)

모델명은 계속 바뀌므로 규칙은 티어를 가리키고, 티어→실제 모델 매핑만 여기서 갱신한다.

| 제품 | 경량(빠름·싼 소비) | 기본(디폴트) | 상위(추론 강화) | 특수 |
|---|---|---|---|---|
| ChatGPT | 기본 모델의 빠른 응답 | 기본 모델 | Thinking/추론 모드 | Deep Research |
| Claude Code | Haiku (서브에이전트용) | Sonnet | Opus (에스컬레이션) | Advisor(⚠️실험) |
| Claude Cowork/챗 | — | 기본 배정 모델 | 상위 모델 선택 시 | — |
| Codex | — | codex 기본 | 고추론 설정 | — |
| Gemini | Flash급 | Pro급 | Pro+추론 설정 | Deep Research급 조사 |

**티어 운용 원칙 (모델이 바뀌어도 불변):**
- 항상 **기본 티어로 시작**한다.
- 결과 불만족·실패 시 **같은 제품의 상위 티어로 1회만 승급**한다.
- 상위 티어로도 불만족이면 **모델을 더 올리지 말고 교차 검증으로 전환**한다(다른 노드에 같은 패킷). 같은 계열을 더 굴리는 것보다 다른 두뇌가 싸다.
- 경량 티어는 사람이 고르지 않는다 — Claude Code 내부에서 서브에이전트가 자동 사용.
- ※ Fable 5 무료 이벤트: 종료 시점 구체화(2026-07-16 확인) — 7/19 PT = **한국시간 7/20(월) 15:59까지**. "한 번 하면 오래 남는 판단"(설계 감사·템플릿 검토)에만 투입. 이후 이 칸은 삭제하고 규칙 변화 없음.

## §1. 1차 판정 — 시작 제품 (위에서부터 첫 매칭)

| # | 조건 (작업 한 문장에서 판단) | 시작 제품 | 시작 티어 |
|---|---|---|---|
| R1 | **코드를 쓰거나 고친다** | Claude Code | 기본 (내부 전환 자동) |
| R2 | 코딩이지만 **설계·방향이 불확실** | ChatGPT로 계획 먼저 → 계획 확정 후 R1 | ChatGPT 상위 |
| R3 | **로컬 파일·문서·폴더를 만지는 비개발 업무** (정리·변환·docx/pptx·취합) | Claude Cowork | 기본 |
| R4 | **조사 규모 큼** (출처 5개+, 비교·검증 필요) | ChatGPT Deep Research | 특수 |
| R5 | 조사 대상이 **초장문·영상·이미지·Google 문서/시트** | Gemini | Pro급 (대규모면 Deep Research급) |
| R6 | **전략·기획·사업계획·글쓰기·아이디어** | ChatGPT | 기본 (복잡하면 상위) |
| R7 | 위 어디에도 안 걸림 | 개발 문맥이면 Claude Code, 아니면 ChatGPT | 기본 |

## §2. 2차 판정 — 병렬·검수 (신호 3개만 본다)

**신호 ① 실패 위험** (돈·삭제·대량 수정·외부 발송·법률/세무가 걸리는가?)
- 높음 → **병렬 필수**: 코딩이면 Codex 교차, 비코딩이면 다른 노드 1개에 같은 패킷. 최종 승인 나 + 검수자 1.
- 낮음 → 병렬 생략 가능.

**신호 ② 조사 성격** (사실이 틀리면 안 되는가?)
- 예 → ChatGPT·Gemini **2노드 병렬** 후 Cowork가 불일치 대조표 → 검수자가 종합.

**신호 ③ 남은 사용량** — `vault/ops/usage.md` 신호등 확인:
- 🟢 여유 / 🟡 주의(한도 50%+) → 그대로
- 🔴 임박 → **대체표(R8)** 적용:

| 소진된 노드 | 대체 | 대체 시 주의 |
|---|---|---|
| Claude Code 🔴 | Codex (같은 repo) | 서브에이전트 없음 → 패킷의 완료 기준을 더 잘게 |
| ChatGPT 🔴 | Claude 챗/Cowork (전략·글), Gemini (조사) | 감사 역할은 사람이 더 꼼꼼히 |
| Gemini 🔴 | ChatGPT Deep Research | 멀티모달 자료는 요약본으로 변환 후 |
| Codex 🔴 | Claude Code reviewer 강화 (교차성은 약해짐 — 카드에 명시) | |

## §3. 검수자 규칙 (실행자 ≠ 검수자, 불변)

| 작업 유형 | 1차 검수 | 최종 |
|---|---|---|
| 코드 | Codex (또는 Claude Code reviewer + GPT 카드 검토) | 나 (merge 버튼) |
| 조사·사실 | ChatGPT (출처 비교·종합) — 단 GPT가 실행자였으면 Gemini 또는 Claude | 나 |
| 전략·문서 | ChatGPT가 썼으면 Claude 검토, Claude가 썼으면 GPT 검토 | 나 |
| 저난도 단발 | 검수 생략 | 나 (즉시) |

## §4. 판정 예시 (한 줄 훈련)

- "주문 폼에서 결제 오류" → R1 + 위험 높음(결제) → Claude Code 기본, Codex 병렬, 나+Codex 검수
- "지원사업 3건 비교" → R4 + 사실 중요 → GPT DR + Gemini 병렬(파생 패킷으로 비범위 분할), GPT가 90_FINAL 종합, 나 최종
- "다운로드 폴더의 영수증 정리해서 엑셀로" → R3 → Cowork 단독, 검수 생략, 나 확인
- "신사업 아이디어 벽치기" → R6 → ChatGPT 기본, 절차 없음 (START-HERE 예외 규칙)

※ 판정 결과는 TASK-PACKET의 "담당" 란에 그대로 적고, 저장·상태 관리는 TASK-PACKET.md의 작업 폴더 규칙(vault/ops/tasks/…)을 따른다.

## §5. 카나리 레인 (9Router · chatgpt2codex) — 2026-07-16 추가

§1 판정과 별개로, Dispatcher에는 Stable 레인(공식 API, 기본) 옆에 **카나리 레인**이 존재한다.
- 카나리로 흘릴 수 있는 것: **작업 클래스 allowlist에 등록된 경량 작업만** (C1 태깅·추출·요약 → C2 초안 → C3 코드, 단계별 게이트 통과 후). Fable 판단 작업과 게시·발송·삭제 등 비가역 작업은 **영구 제외**.
- 카나리 실패(오류·타임아웃·품질 미달) 시 **같은 요청이 Stable 레인으로 자동 재시도**된다 — 사용자에게는 지연으로만 보여야 한다.
- 킬스위치 `CANARY_ENABLED=false` 한 개로 즉시 Stable 100% 복귀.
- 상세 설계(5중 격리, G0~G4 승격 게이트, 로그 필드): `1. Projects/ai공장짓기/9Router_chatgpt2codex_카나리통합설계.md`

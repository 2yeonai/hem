# 나의 핵심 맥락

## 나는 누구인가

두 정체성이 하나로 이어져 있다.

- **mo,on(모온) 대표** — 진주·경남 지역에서 산후조리원 퇴소 후 2주~3개월 회복 공백기 산모를 대상으로 한 가정방문형 회복 컨디셔닝 서비스 창업자. 미용사(피부) 국가자격증 보유, 본인이 직접 산후 회복 공백을 겪은 당사자로서 서비스를 기획·운영 중. 예비창업패키지 등 정부지원사업에 이 사업으로 신청 준비 중.
- **비개발자 AI 공장 설계자** — 코드를 모르지만, Claude(대화)·Cowork(실행)·Fable 5(설계 승격) 세 도구를 역할 분담시켜 실제로 작동하는 AI 자동화 시스템을 여러 개 짓고 있음.

즉 AI 공장을 짓는 이유 자체가 추상적인 "플랫폼 사업"이 아니라, **본인 사업(mo,on)을 굴리며 필요했던 도구(지원사업 매칭 자동화)를 직접 만든 것이 출발점**이고, 거기서 얻은 구조를 다른 소상공인 업종(꽃집, 방역 등)에도 반복 적용 가능한 형태로 일반화하는 방향으로 확장 중.

## 일하는 방식의 특징

- **역할 분담이 뚜렷함**: Claude(전략·설계 대화) / Cowork(파일 작성·코드 실행, Sonnet 5 기본) / Fable 5(구조가 안 정해졌거나 충돌할 때만, 반드시 먼저 승인받고 사용).
- **"Gemini식 확장 + Monday식 방어"** — 무조건 긍정하지 말고 항상 위험·거절 이유부터 짚을 것을 스스로 원칙으로 세움.
- **끊김에 대비해 기록을 습관화**: HANDOFF.md(세션 간 인수인계), decision-log(Fable 5 결정 근거), verify_write.py(파일 잘림 버그 자동 복구) 등, 세션·크레딧이 끊기는 환경에서도 이어받을 수 있게 매번 기록을 남김.
- **"앱 하나 만들어줘"가 아니라 "계속 만들고 고치고 운영할 수 있는 공장을 만들어줘"**가 기본 태도. 완성 선언의 최소 조건: quality_gate + sample_input + expected_output + failure_log.
- **속도를 위해 통제를 포기하지 않음**: 법적 리스크나 되돌릴 수 없는 지점(문서 발송, 지원사업 접수 제출 등)엔 반드시 사람 승인 게이트를 남김.

## 왜 기록하고 싶은가

1. 정부지원/꽃집/방역/AI회사 플랫폼(4단계)을 혼자 동시에 굴리고 있어 기록 없이는 스스로도 진행상황을 놓침.
2. Cowork 세션 5시간 한도, 모델 세션 중 고정 등 구조적 제약 때문에, 기록이 없으면 끊긴 지점부터 이어갈 수 없음.
3. 지금 세 스킬(정부지원/꽃집/방역)에서 얻는 설계 판단(왜 이렇게 결정했는지)이 나중에 4단계 AI회사 플랫폼과 9개 신규 공장 후보의 공통 뼈대로 그대로 재사용될 것이기 때문 — 지금의 기록이 곧 미래의 설계 근거.
4. mo,on 회복 기록 관리 철학("회차별 기록 누적 → 다음 단계 판단 근거")과 AI 개발 방식의 기록 습관이 같은 원칙을 공유함.

### 최종 목적

**소상공인이 자기 업종에 맞는 공장(꽃집이면 꽃집 공장, 방역이면 방역 공장 등)을 골라, 메인 러너 위에 조립해 쓰는 플랫폼.** 각 공장은 독립적으로 개발되지만 전부 같은 계약 언어(manifest, stage, quality_gate, human_review)로 말하기 때문에 서로 호환되고, 메인 러너 하나가 이를 전부 실행한다.

### 정부지원사업 공장의 실제 전체 그림 (완전 자동화의 의미)

```
[공고 스크래핑] → [내 사업과 자동 매칭] → [신청서 자동 작성]
                                              ↓
                                        [접수 — 사람이 직접 제출]
                                              ↓
                                        [선정 시] → PPT 자동제작 → 발표대본 자동제작 → 예상 Q&A 자동제작
```
사람이 개입하는 지점은 "접수 제출" 단 하나. (2026-07-16 갱신) 앞단 스크래핑은 T7로 구현 완료(가동은 bizinfo API 키 대기), 뒷단 PPT/발표대본/QNA는 T8로 8 stage mock 구현·테스트 완료 — 남은 것은 실 LLM 연동뿐.

## 공통 아키텍처 원칙 (모든 공장에 적용)

| 항목 | 원칙 |
|---|---|
| 폴더 구조 | SKILL.md, manifest.yaml, scripts/, templates/, checks/ |
| 공장 정체성 | 내부 코드가 아니라 io_contract |
| 모델 라우팅 | 모델명 하드코딩 금지, 티어명(low_cost/mid/high)만 사용 |
| 품질 판정 | quality_gate 없는 공장은 완성으로 보지 않음 |
| 검수 | 자동 점수 + 사람 승인 이벤트의 이중 게이트 |
| 장기 유지 | failure_log, sample_input, expected_output, HANDOFF 필수 |
| 시작 방식 | triggers(event/schedule) — 문의가 오면 vs 날짜가 되면, 둘 다 표현 가능 |
| 조건부 실행 | run_if — 조건 맞을 때만 특정 단계 실행 |
| 법정 문서 흐름 | 문서 자체에 승인 상태(초안/승인대기/승인완료/반려) 필드 + human 게이트 1개, 승인완료 아니면 발송 절대 금지 |

- **강점**: 기록 습관(HANDOFF/decision-log), 승인 게이트 감각(되돌릴 수 없는 건 사람이), 구조적 사고(비개발자인데 계약·게이트 개념을 체득). 이 세 가지는 상위 수준 — 시스템이 이미 트렌드 글들이 권하는 것보다 정교함.
- **병목**: 실행력이 아니라 **판단 과부하**. 정보 유입량 >> 판단 처리량이라 "뭐부터/어떤 모델/얼마 걸리나"에서 멈춤. 해결책은 더 많은 정보가 아니라, 판단을 파일에 위임하는 것 — 순서는 실행계획, 모델은 모델정책, 중요도는 감사_로드맵이 이미 답을 들고 있음.
- **운영 규칙 3줄**:
  1. 새 자료를 보면 저장하지 말고 클로드에게 "볼트 기준으로 평가해줘" 한 줄만 던진다. 평가 전엔 세팅을 바꾸지 않는다.
  2. 판단은 클로드가 선택지+추천으로 만들어 오고, 혜미는 고르기만 한다. (클로드는 제안 시 항상 모델·소요시간·혜미 몫을 명시할 것)
  3. 혜미의 시간은 "사람만 아는 정보 입력"(골든셋, 실제 사례, 승인)에만 쓴다 — 그 외는 전부 위임.

# 1부. 대표님용 사용 안내

## 이 스킬이 지금 하는 일 (한 줄 요약)
전화/카톡/문자로 들어온 방역 문의를 정리해서 고객DB에 저장하고, 방문일이 다가오면 알려주고, 현장 작업이 끝나면 완료보고서·소독증명서 초안을 자동으로 만들어줍니다. 단, **대표님이 승인 버튼을 누르기 전까지는 어떤 문서도 고객에게 발송되지 않습니다.**

## 문서 만들고 보내는 흐름
1. 현장 작업이 끝나면 담당자가 사용 약품·처리 구역 등을 입력합니다.
2. 완료보고서·소독증명서가 필요한 건에 한해 초안이 자동으로 만들어집니다.
3. 대표님이 초안을 검토합니다. 이때 상태는 "승인대기"입니다.
4. **승인**하면 → 문서가 잠기고("승인완료" 상태), 그제서야 발송이 가능해집니다.
5. **반려**하면 → 이유를 두 가지 중 하나로 고릅니다.
   - "문구/서식이 이상해요" → 문서만 다시 만듭니다 (현장 데이터는 안 건드림).
   - "약품명·면적·날짜 같은 내용 자체가 틀렸어요" → 현장 입력부터 다시 고친 뒤 문서를 다시 만듭니다.
6. 반려된 예전 버전은 지우지 않고 남겨둡니다 (나중에 확인 필요할 때 대비).

## 가장 중요한 안전장치
**"승인완료" 상태가 아닌 문서는 시스템이 절대 발송하지 않습니다.** 실수로 승인 안 된 문서가 나가는 것을 막기 위한 장치이며, 이번 세션에서 실제로 작동하는지 테스트까지 마쳤습니다 (아래 "검증 결과" 참고).

## 사업자 정보 관련 새 주의사항 (2026-07-07 추가, 2026-07-07 갱신)
완료보고서·소독증명서 초안에 "사업자 정보"(상호명/대표자명/사업자등록번호) 칸이 새로 추가됐고, **이제 이 값은 현장완료 입력 단계에서 한 번 입력되면 시스템이 저장해뒀다가 문서를 만들 때 자동으로 불러와 채웁니다** (더 이상 문서마다 손으로 다시 적을 필요 없음). 다만 시스템은 "값이 저장되고 불러와지는 것"만 보장할 뿐, **그 값이 실제로 맞는지는 여전히 사람이 확인해야 합니다.** 실제 문서를 발급하기 전에는 반드시 국세청에서 발급하는 공식 사업자등록증명과 대조해서, 잘못 입력된 값이 없는지 확인해주세요.

## 지금 안 되는 것 / 아직 조심해야 하는 것
- **이 문서 초안을 그대로 법적 효력이 있는 최종 문서로 쓰면 안 됩니다.** 소독증명서의 법정 필수 기재항목은 아직 국가법령정보센터 원문·관할 보건소 서식으로 최종 확인되지 않았습니다.
- **지금까지의 테스트는 전부 가상의 합성 데이터(예: "해오름카페", "청솔유치원")로 한 것입니다.** 실제 방역 문의·현장 데이터로는 아직 검증되지 않았습니다.
- 대표자 1인이 최종 검수하는 구조를 전제로 만들어졌습니다. 직원 여러 명이 각자 다른 권한으로 쓰는 구조는 아직 준비되지 않았습니다.
- 현장 인터넷이 끊기는 상황(오프라인)에 대한 대비는 아직 설계되지 않았습니다.
- 아직 사람이 판단해서 결정해야 할 구조적인 사항이 5가지 남아있습니다 (2부 참고).

## 핵심 메커니즘
- **run_if**: stage 조건부 실행. 문자열 식별자를 `evaluate_run_if()`가 평가하는 방식이며, 정식 표현식 언어가 아님.
- **entry_points**: 한 stage가 여러 경로(OR)로 도달 가능함을 나타냄. `build_graph()`가 entry_points 엣지를 추가하고, `topo_order()`가 그 그래프를 그대로 재사용하도록 되어 있음 (예전엔 depends_on만 봐서 스케줄 트리거가 리마인더봇에서 막히는 버그가 있었음 — 이번에 수정됨).
- **승인 블록 (approval_block)**: `document_draft.report`/`certificate` 각각에 `status`(초안/승인대기/승인완료/반려)/`version`/`approved_by`/`approved_at`/`rejection_reason`/`rejection_target`/`version_history`. `approval.status=승인완료`가 되면 `locked=true`로 전환 — 이후 수정 불가, 새 버전만 가능.
- **on_fail.human_reject.return_to**: 기존엔 고정값 1개만 가능했으나, 이제 `{rejection_target_key: stage_id}` 형태의 매핑도 지원 (기존 단일값 방식도 하위호환 유지).
- **문자장부봇의 발송 차단 안전장치**: `approval_required=true`인 문서 중 하나라도 `status != 승인완료`면 `RuntimeError`를 던지고 발송을 막음. run.py에서 실제로 발동하는지 테스트 완료(시나리오 C).
- **재시도 루프는 DAG 사이클이 아님**: `_run_document_approval_cycle()`이 문서생성봇↔문서승인을 명령형으로 최대 3회 반복시키는 방식이며, manifest의 depends_on 그래프 자체는 순환하지 않음 (문서화용 메타데이터로만 return_to를 씀).
- **알려진 동기화 버그**: 이 폴더에서 큰 파일을 Edit할 때 Windows쪽 파일도구와 Linux bash 마운트가 잠깐 어긋나는 현상이 반복됐음. 대응: `mv file file.stale-$(date +%s)` 후 bash heredoc(`cat > file << 'EOF'`)으로 재작성 → `python3 -m py_compile` 또는 `yaml.safe_load`로 검증. 큰 파일 리라이트는 Write/Edit 도구보다 bash heredoc이 더 안정적이었음.

## 지난 전체 점검에서 발견된 사항 3건 — 전부 해결됨 (2026-07-07 갱신)
아래 3건은 직전 점검에서 발견돼 남겨뒀던 것이며, 이번 라운드(사용자 지시)에서 전부 필드 추가/문구 정리 수준으로 해결했습니다(구조 변경 아님).
1. **[해결] 템플릿 상태값 표기와 스키마 enum 통일**: 템플릿 3종의 "작성상태" 줄을 "초안 / 승인대기 / **승인완료** / 반려"로 수정해 스키마 enum과 철자까지 동일하게 맞춤. 다른 상태값(초안/승인대기/반려)도 재대조 완료 — 불일치 없음.
2. **[해결] 사업자 정보 필드를 manifest 데이터모델에 연결**: `shared_context.business_info`(상호명/대표자명/사업자등록번호) 신규 추가 — `현장완료입력봇`이 쓰고 `문서생성봇`이 읽어 `document_draft.report`/`certificate` 양쪽에 복사. `certificate.상호`는 `business_info.상호명` 값으로 자동 채워짐(중복 입력 불필요). `run.py`에서 실제로 저장→전달되는지 배치 테스트로 확인함.
3. **[해결] `SKILL.md`의 "실행코드 없음" 문구 갱신**: `run.py`가 실제로 존재하고 18 stage + 문서승인워크플로우까지 동작한다는 점을 반영해 "실행 방법"/"알려진 한계" 절을 다시 작성함.

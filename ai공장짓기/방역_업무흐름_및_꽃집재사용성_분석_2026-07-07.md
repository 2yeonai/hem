## 1. 방역 실제 업무 흐름 검증

**보고서 원문(기업현황 섹션)에 명시된 실제 흐름:**

> 신규 견적 문의 접수 → 상담 내용 정리 → 방문일정 확인 → 현장 방역 수행 → 완료보고서·소독증명서 요청 대응 → 사후관리

사용자가 제시한 5단계(문의접수→방문→완료→문서발급→사후관리)와 비교하면 **큰 방향은 맞지만, 3가지 차이가 있음**:

1. **"방문" 앞에 두 단계가 숨어 있음** — "상담 내용 정리"와 "방문일정 확인"이 문의접수와 현장방역 사이의 별도 단계로 명시돼 있음. 특히 "방문일정 확인"은 정기 고객의 "방문주기·D-day 리마인더"와 직결되는 단계라 따로 떼어둘 필요가 있음.
2. **"문서발급"은 항상 일어나는 고정 단계가 아니라 조건부 단계** — 원문이 "완료보고서·소독증명서 **요청 대응**"이라고 쓴 것에 주목. 모든 방문 후 문서가 자동 발급되는 게 아니라, 고객이 요청했을 때만 발급되는 분기(branch)로 보임. (요청 여부 자체가 `order_draft_bot` 단계에서 다뤄야 할 필드 중 하나 — "문서 요청 여부"가 보고서에 명시된 구조화 대상 필드임.)
3. **선형 흐름과 별개로 "정기 고객 방문주기 기반 리마인더"라는 순환 트리거가 병행 존재** — 이건 "문의 접수"라는 이벤트로 시작하는 게 아니라, 이미 저장된 고객DB에서 날짜가 되면 자동으로 재트리거되는 구조(정기 고객사 약 110곳, 방문주기 기준). **꽃집 14봇 설계에는 이런 유형의 트리거가 아예 없음** — 꽃집은 100% "통화/문자 수신"이라는 이벤트로만 파이프라인이 시작됨. 이 점이 이번 분석에서 가장 큰 구조적 차이로 보임(4번 섹션에서 다시 다룸).

**정리된 실제 흐름(6단계 + 1개 병행 트리거):**

| 순서 | 단계 | 비고 |
|---|---|---|
| 1 | 신규 견적 문의 접수 | 전화/카톡 등 다채널 추정(보고서에 채널 명시는 약함 — 확인 필요) |
| 2 | 상담 내용 정리 | AI 입력 양식 구조화 대상 |
| 3 | 방문일정 확인 | 정기고객은 리마인더(D-day)와 연결 |
| 4 | 현장 방역 수행 | 대표/직원 2인이 직접 수행(배송기사 같은 별도 인력 없음) |
| 5 | 완료보고서·소독증명서 요청 대응 | **조건부** — 요청 있을 때만 |
| 6 | 사후관리 | 만족도·불편사항 수집, 시즌 리포트, 재방문 상담 근거로 환류 |
| 병행 | 정기 고객 방문주기 D-day 리마인더 | 이벤트 트리거 아님 — 날짜/배치 트리거. 선형 흐름 밖에 존재 |

## 2. 꽃집 14봇 재사용 가능성 표

꽃집 `12봇_kind분류표.yaml`(실제 14 stage: local 7 / model 6 / human 1)에는 이미 각 stage마다 `domain_neutral: true/false` 플래그와 근거가 기록돼 있어, 이를 기준으로 방역 대응 여부를 판정함.

| 꽃집 봇 | 방역에서의 대응 역할(추정) | 재사용 판정 | 근거 |
|---|---|---|---|
| collection_bot (수집봇) | 견적문의/상담 접수(전화·카톡 등) 수집 | **겉모양 재사용 가능** | 원본이 이미 domain_neutral: true — 입력 채널 수집은 도메인 무관 |
| transcription_bot (받아쓰기봇) | 전화상담 STT/전사 | **겉모양 재사용 가능** | domain_neutral: true |
| order_classification_bot (주문판정봇) | "견적문의/방역요청 vs 일반문의·스팸" 판정 게이트 | **겉모양 재사용 가능** (enum 값만 교체) | domain_neutral: true — "이게 애초에 처리 대상인지" 판정 게이트는 도메인 무관이라고 원본에 이미 명시 |
| order_split_bot (주문분리봇) | 한 통화에 여러 현장 건이 섞였을 때 분리 | **부분재사용 — 필요성부터 확인** | 구조 자체는 domain_neutral: true(원본 파일에 "방역에서도 한 통화에 여러 현장 이슈가 섞일 수 있음"이라고 이미 언급돼 있음)이나, 방역 보고서엔 이런 케이스의 실제 빈도 근거가 없음 |
| correction_bot (말귀봇) | 사투리/구어체 보정(진주 지역) | **겉모양 재사용 가능** | domain_neutral: true |
| order_draft_bot (주문정리봇) | 고객명/공간유형/방문희망일/요청사항/문서요청여부/후속연락여부로 필드 구조화 | **겉모양 재사용 가능** (필드셋만 교체) | domain_neutral: true, 방역 보고서에 이미 구조화 대상 필드가 구체적으로 나열돼 있어 매핑이 쉬움 |
| ribbon_price_bot (리본상품금액봇) | 대응 개념 없음 | **완전 신규설계 필요(또는 제외)** | 원본에 이미 domain_neutral: false로 표시됨(리본 문구=꽃집 전용). 방역 보고서에도 표준가격표/리본 유사 산출물 언급 없음 |
| review_manager_bot (검수매니저) | confidence·후보값 종합 → 자동확정 vs 대표자 에스컬레이션 | **겉모양 재사용 가능** (임계값만 재조정) | domain_neutral: true |
| human_reviewer (사람 검수자) | 대표자 최종 검수 | **겉모양 재사용 가능, 그대로** | 방역 보고서에 "대표자 입력 → AI 구조화 → 산출물 생성 → 대표자 최종 검수" 원칙이 이미 명문화돼 있어 꽃집의 human_reviewer 개념과 정확히 일치 |
| storage_bot (저장봇) | 고객DB/오더메모 저장·버저닝 | **겉모양 재사용 가능** | domain_neutral: true |
| print_prep_bot (출력준비봇) | 완료보고서·소독증명서 초안 템플릿 생성 | **부분재사용** (골격만) | "확정데이터→템플릿 채우기" 골격은 재사용 가능하나 템플릿 내용(증명서/보고서 양식)은 완전히 새로 작성 필요 |
| dispatch_bot (배송전달봇) | 방문일정/현장 정보 전달 | **완전 신규설계 필요** | 원본에 이미 명시: "배송"이라는 개념은 방역의 "현장방문"으로 치환 가능하나, stage 내용은 백지에서 다시 짜야 함(복붙 금지) |
| delivery_photo_bot (배송사진봇) | 현장 완료 사진(있다면) | **부분재사용 — 필요성 확인** | 방역 보고서엔 "현장완료메모"만 언급되고 사진 여부는 명시 안 됨 — 확인 필요 |
| sms_ledger_bot (문자장부봇) | 문자·알림톡 발송 + 이력 기록 | **겉모양 재사용 가능** | domain_neutral: true, 보고서에도 "문자·알림톡" 운영비 항목이 실제로 있어 대응됨 |

**방역에만 필요해 보이는 신규 stage (꽃집 14개 목록엔 대응 없음):**
- **정기방문 D-day 리마인더봇** — 고객DB의 방문주기 기준으로 날짜를 비교해 알림을 트리거. 판단이 필요 없는 카운트/날짜비교 성격이라 `kind: local`에 가까우나, 트리거 방식 자체가 "접수 이벤트"가 아니라 "시간 배치"라 v2 스키마의 표현 범위를 벗어남(3번 섹션에서 다룸).
- **만족도·불편사항 수집봇** — 사후관리 단계 산출물. 꽃집 파이프라인엔 대응 stage 없음.

## 3. 재사용률 추정 (러프 추정 — 실측 아님)

| 분류 | 개수 | 비율(14개 기준) |
|---|---|---|
| 겉모양(구조) 그대로 재사용 가능 | 9개 (collection, transcription, order_classification, correction, order_draft, review_manager, human_reviewer, storage, sms_ledger) | 약 64% |
| 부분재사용 (골격만 재사용, 내용 상당 부분 새로 작성 또는 필요성 확인 필요) | 3개 (order_split, print_prep, delivery_photo) | 약 21% |
| 완전 신규설계 필요 (대응 개념 없음) | 2개 (ribbon_price, dispatch) | 약 14% |

→ 넓게 잡으면(부분재사용 포함) **약 86%**, 좁게 잡으면(겉모양만) **약 64%**가 재사용 가능권.
단, 이 비율은 꽃집 14개 안에서만 계산한 것이고, **방역 전용 신규 stage 2개(리마인더봇, 만족도수집봇)는 이 14개 밖에 있어 비율 계산에 포함 안 됨** — 실제 방역 파이프라인 전체 규모는 14개보다 커질 가능성이 있음. 실측(실제 구현 후 diff)까지는 이 숫자를 확정치로 쓰지 않는 걸 권함(꽃집 문서들도 같은 이유로 "확인 필요"를 자주 남겨둠).

## 4. Architecture 후보 (확정 아님 — 후보만 제시, 승인 필요)

`decision-log_skill-factory-architecture.md`의 Fable 5 결정(2026-07-06)에 따르면 방역은 "3번째 도메인"이라 core+plugin 구조 전환 대상으로 이미 지목돼 있음. 이번 분석에서 새로 드러난 문제는 **v2 스키마(`manifest.schema.v2.yaml`)가 "접수 이벤트 기반 순차 실행"만 표현 가능하고, "시간/배치 기반 트리거"(정기방문 리마인더)를 표현할 방법이 없다**는 점 — 이건 기존 open_issues 목록에도 없던 새로운 갭임.

**후보만 나열 (확정하지 않음, 승인 후 진행):**

- **후보 A** — core+plugin 구조를 그대로 채택하되, v2 스키마에 `trigger_type: event | schedule` 같은 필드를 신규 추가해서 리마인더 stage를 배치 트리거로 표현.
- **후보 B** — 리마인더 기능은 파이프라인 스키마 밖에 완전히 별도의 "스케줄러 모듈"로 분리하고, 방역 스킬 manifest에는 이벤트 기반 부분(문의→방문→완료→문서)만 남긴다.
- **후보 C** — 지금은 방역도 꽃집처럼 "폴더 복제" 1차 구현으로 진행(Fable 결정 그대로 유지, 변경 없음)하고, 리마인더 트리거 문제는 별도 open_issue로만 기록해두고 구조 변경 논의는 나중으로 미룬다.

이 중 무엇을 택할지, 혹은 Fable 5 승격이 필요한 판단인지는 여기서 결정하지 않음 — 프로젝트 규칙(3번)대로 진행 전 사용자 승인 필요.

## 5. 확인이 필요한 항목 (임의로 결정하지 않고 남김)

- 신규 견적 문의 접수 채널이 정확히 무엇인지(전화/카톡/문자 중 실제 비중) — 보고서에 명시 약함
- order_split_bot(한 통화 다중 현장 분리), delivery_photo_bot(현장 사진) 두 stage의 실제 필요 여부
- 문서(완료보고서/소독증명서) 발급이 "요청 시에만"이라는 조건부 해석이 맞는지 — 원문 "요청 대응"이라는 표현에 근거한 추정
- ribbon_price_bot 대응 개념이 방역에 정말 없는지, 아니면 "견적 금액 산정" 같은 형태로 존재하는데 보고서에 안 드러난 것인지
- v2 스키마의 event/schedule 트리거 갭을 방역 진행 전에 먼저 메울지, 나중으로 미룰지 (4번 섹션 후보 참고)

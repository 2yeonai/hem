# 볼트 전체 스캔 발견카드 (2026-07-13, 0단계)

## FACT (이미 확인·조치 완료 — Fable5가 다시 볼 필요 없음)

1. **알려진 Edit/Write 잘림 버그가 오늘 4개 파일에 실제로 발생해 있었음** — `0. Docs/명령어_사전.md`, `LLM위키_홈.md`, `2. Areas/핵심맥락.md`, `ai공장짓기/착수_2026-07-12.md`. 전부 "관련 문서" 목록 등 파일 맨 끝에서 문장 중간 바이트로 끊겨 있었음. git HEAD와 대조해 3건은 완전 복구, 1건(`착수_2026-07-12.md`)은 **최초 커밋부터 이미 잘려 있어 원래 링크 대상 1개가 영구 유실**(문장 자체는 완결이라 실질 내용 손실은 없어 보임). 전부 heredoc 방식 + 바이트 검증으로 재작성 완료.
2. **이 버그가 이번 세션 중 제 Edit 호출로 2번 더 재발함** (`명령어_사전.md`) — 작은 한 줄 수정에도 발생. `verify_write.py`가 이미 있다는데 사후 검증용이지 예방은 안 되는 듯. **재발 방지책 필요** — 아래 CONFLICT 1번.
3. **명령어_사전.md 관련문서 링크 2건에 "1. Projects/" 프리픽스 누락** (콘텐츠_작업스타일, channel_config) — 리오르그 후 안 고쳐진 경로. 수정 완료.
4. **4개 공장 기계 검증 전부 통과**: `scripts/run.py`(방역/꽃집/정부지원) py_compile 통과, 콘텐츠 공장은 설계상 자체 run.py 없이 `handlers.py`로 러너에 얹혀 있음(정상 — 신축 방식 그대로). 4개 manifest.yaml 전부 validate_manifest.py FAIL 0 / WARN 0. 러너 3개 파일(runner.py/webapp.py/cowork_provider.py) + verify_write.py도 전부 컴파일 통과. **코드 레벨 버그는 못 찾음.**
5. **LLM위키_홈.md의 `[[클로드 꽃집 ai.md]]` 링크는 대상이 없음** — 원인은 새 버그가 아니라 이미 감사_로드맵 R4에 있는 그 문제(꽃집 폴더 루트가 정부지원 사본으로 오염되어 진짜 허브 파일이 없음)와 동일 뿌리. 중복 기록 안 함.
6. **샌드박스 접근 이슈 3건은 실제 버그 아님** — `1. Projects/클로드 ai 자동화/company_charter.md` 등 3개 파일이 이번 Cowork 세션의 리눅스 마운트에서 유니코드 정규화 문제로 파이썬 스캔에 안 걸렸지만, Read 도구(Windows 실제 경로)로는 정상 열람 확인함. 혜미 실제 컴퓨터에서는 문제 없을 가능성이 높음 — 참고만.

## CONFLICT (판단 필요 — 서로 다른 정보가 부딪힘)

1. **잘림 버그 재발 방지 방법 결정 필요.** `verify_write.py`가 사후 탐지용인지 사전 방지용인지 볼트 문서에 명확하지 않음. 오늘처럼 매번 heredoc+바이트검증으로 우회하는 게 임시방편인데 지속 가능한 해법(예: 모든 세션이 Edit 대신 항상 Write+검증 쓰게 규칙화? verify_write.py를 세션 시작 시 자동 실행?)은 볼트에 아직 없음.
   신뢰도: 높음(오늘 3번 실측).

## UNKNOWN (사실관계 자체가 불명 — 혜미만 답할 수 있음)

1. **`_inbox/옵시디언 변환본/00 인박스 자료 인덱스.md`가 `3. Resources/정부지원사업_자료/00 인박스 자료 인덱스.md`의 이전 초안으로 보임(154줄 vs 151줄, 내용 거의 동일하되 경로 패턴이 다름).** 둘 다 살아있음 — `_inbox`판을 삭제해도 되는지, 아니면 뭔가 다른 용도로 남겨둔 것인지 불명.
   신뢰도: 중간(diff로 유사성만 확인, 왜 두 개인지는 세션 기록에 없음).
2. **두 인덱스 파일이 참조하는 "옵시디언 변환본/..." 경로 자체가 실제 폴더명과 다름.** `3. Resources/정부지원사업_자료/` 안에는 "옵시디언 변환본"이 아니라 `_원문`이라는 폴더가 있고, 그 안 파일명도 인덱스가 가리키는 이름과 다름(`...__원문.md` 접미사). 이 때문에 위키링크 약 150개가 대상을 못 찾음. **인덱스가 옛날 경로 규칙으로 쓰였고 그 뒤 `_원문`으로 이름이 바뀌면서 인덱스만 안 따라간 것으로 추정**되지만 확정 아님.
   신뢰도: 중간.
3. **hwp/pdf/pptx/mp4 등 "원본" 링크 약 250개가 전부 vault 안에 실제 파일이 없음** (변환된 .md 텍스트만 있음). 원본 바이너리를 vault에 안 넣기로 한 의도적 설계인지(용량 때문에), 복사가 누락된 것인지 불명. 의도적이라면 "버그"가 아니라 그냥 안 채워진 참조일 뿐.
   신뢰도: 낮음(패턴만 확인, 의도는 혜미만 앎).

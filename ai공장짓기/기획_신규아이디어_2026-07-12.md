# 신규 아이디어 기획서 4건 (2026-07-12, 혜미 발화 기반)

> 각 기획: 문제 → 타깃 → 핵심 기능 → MVP 범위 → 공장 구조 스케치 → 리스크 → 모델/시간/혜미 몫.
> 우선순위 배정은 혜미 몫. 기존 9개 후보와 병합해 Fable 5 총설계 1회로 묶기 권장.

## 기획 1 — 초민감자 AI (심리 결에 민감한 대화 AI)

- **문제**: 초민감자(HSP)는 일반 챗봇의 둔한 공감("힘드시겠어요")에 오히려 상처받음. 생각의 미세한 결을 따라와 주는 AI가 없음.
- **타깃**: 1차 혜미 본인(검증자 겸 골든셋 공급자) → 2차 mo,on 산모 고객(회복기 정서 케어 — 산후우울 경계 구간) → 3차 HSP 일반.
- **핵심 기능**: ①대화에서 감정 결 감지(표면 단어가 아니라 문장 리듬·망설임) ②"해결하려 들지 않고 먼저 정확히 되짚기" 원칙 ③민감도 프로필 누적(회차별 기록 → 다음 대화 판단 근거 = mo,on 철학 그대로).
- **MVP**: 앱이 아니라 **프롬프트 페르소나 + 대화 로그 누적 구조**부터. Claude 프로젝트 지침 1개 + 혜미 실사용 2주 → 골든셋(좋았던/상처받은 응답) 수집.
- **공장 구조**: intake(local) → 감정결분석(model:sonnet) → 응답생성(model:sonnet) → 민감도프로필갱신(local) → 위험신호 감지 시 human 게이트(전문가 연계 안내).
- **리스크 (정직하게)**: 심리 상담 아님 — 의료·상담 경계 명확화 필수. 위기 신호(자해 등) 시 전문기관 안내로 넘기는 게이트가 헌장상 필수. 이 경계 설계가 Fable 5 상정 대상.
- **추정**: MVP는 Sonnet/2h/혜미 2주 사용 피드백.

## 기획 2 — 스텝0~10 티칭 앱 (무지랭이 전용 단계 교습)

- **문제**: 오픈소스·툴 소개 글은 "링크 던지고 끝" — 비개발자에겐 스텝0(설치 전 준비)부터가 벽.
- **타깃**: 비개발자 소상공인·주부 — 혜미가 걸어온 길 그 자체가 콘텐츠.
- **핵심 기능**: ①링크/툴 이름 입력 → 스텝0(준비물)~10(실전 운영) 커리큘럼 자동 생성 ②각 스텝에 "확인 문제 1개"(통과해야 다음) ③막히면 스크린샷 올려 진단.
- **MVP**: 앱 이전에 **콘텐츠 시리즈로 검증** — "무지랭이 시리즈"(백로그 4번)에서 스텝0~10 포맷으로 3편 발행 → 반응 보고 앱화 결정. 콘텐츠가 곧 시장조사.
- **공장 구조**: 링크수집(local) → 난이도분해(model:sonnet) → 스텝0~10 생성(model:sonnet) → 확인문제 생성(model:haiku) → 형식게이트(local) → 발행(기존 콘텐츠 공장 재사용).
- **리스크**: 기존 9개 후보의 "학습변환 공장"과 60% 겹침 → 별도 공장이 아니라 **학습변환 공장의 출력 포맷 1개**로 흡수하는 게 경제적일 수 있음 (Fable 총설계 때 병합 판단).
- **추정**: 시리즈 3편 Sonnet/편당 30분/혜미는 소재 승인만.

## 기획 3 — 링크→낭독(TTS)+요약 파일 앱

- **문제**: 읽을 시간 없음(육아+사업 병행). 링크는 쌓이는데 소비가 안 됨.
- **타깃**: 혜미 본인 → 같은 처지의 워킹맘·소상공인.
- **핵심 기능**: ①링크 입력 → 본문 추출 → 요약 파일(md) 생성 ②TTS로 낭독(운전·설거지 중 청취) ③요약은 볼트에 자동 저장(기획 4와 연결).
- **MVP**: 오늘부터 가능한 반쪽 — "링크 주면 요약 md 저장"은 기존 Cowork로 즉시. TTS는 ①폰 기본 기능(안드로이드 '듣기'/아이폰 화면 낭독 — 설치 0) ②Edge 브라우저 소리내어 읽기 ③앱화는 나중.
- **공장 구조**: 링크수집(local) → 본문추출(local) → 요약(model:haiku~sonnet) → md저장+볼트연결(local) → (선택) TTS 변환(local, 외부 TTS).
- **리스크**: 유료 기사·로그인 콘텐츠는 추출 불가(차단 도메인 우회 금지 규칙). 저작권 — 요약은 짧게, 원문 복제 금지.
- **추정**: 반쪽 MVP는 즉시(0h 추가 개발), 명령어 사전에 등록만.

## 기획 4 — 인스타 저장물 아카이브 ("저장의 무덤" 해결, 혜미 지시 2번)

- **문제**: 인스타에서 저장한 릴스·게시물은 다시 안 보게 되고 검색도 안 됨. 혜미가 원하는 형태: **글로 남기고, 영상은 하단 링크로.**
- **해결 구조 (3단)**:
  1. **즉시 루틴(오늘부터)**: 혜미가 링크 붙여넣고 "아카이브해줘" → Claude가 Chrome(인스타 로그인 후)으로 열어 캡션·내용 추출 → **요약 글(핵심 포인트 3~5줄) + 하단에 원본 영상 링크** 노트를 `5. Zettelkasten/00. Inbox`에 저장, 태그 자동(#요리 #육아템 식) → 옵시디언 검색 가능해짐.
  2. **배치 처리**: 인스타 "저장됨" 컬렉션은 공식 API 없음 → 로그인된 Chrome으로 저장됨 페이지를 열어 한 번에 N건 아카이브 (주 1회 예약작업 후보).
  3. **재소비 루프**: 주간 예약작업이 Inbox의 아카이브 노트를 주제별로 묶어 "이번 주 저장한 것 다시보기" 요약 발행 → 기획 3의 TTS로 들을 수도 있음.
- **리스크**: 인스타 화면 구조 변경 시 추출 로직 손봐야 함(유지보수 카드로 관리). 저장 컬렉션 대량 처리는 속도 제한 고려.
- **추정**: 루틴 1은 명령어 등록만(0h) — 단, **Chrome 인스타 로그인 선행 필요**(게시 대기와 같은 열쇠). 배치는 Sonnet/1h.

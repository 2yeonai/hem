# 기록체계 재설계 (2026-07-17) [k9cjhmw7z9]

> 작성: Fable 5 (설계 판단). 근거: 같은 날 Sonnet 실태조사(세션로그 2026-07-17 참조).
> 한 줄 요약: **기록 파일이 너무 많고(맥락 파악에 12~20개), 같은 사실이 최대 6곳에 중복되며, "HANDOFF 갱신" 규칙엔 강제 장치가 없어서 안 지켜진다. → 정본을 종류별로 1개씩만 지정하고, 규칙은 기억 대신 스크립트로 검사한다.**

## 1. 진단 (실태조사로 확인된 사실)

1. 진입점이 3개다: `0. 최우선_확인파일.md` / `START-HERE.md` / `2. Areas/핵심맥락.md` — 셋 다 "먼저 읽어라"를 자처함.
2. HANDOFF(인수인계 정본)가 밀려 있다: ai공장짓기 약 9일, 정부지원 약 5일. 정부지원은 최근 작업(T7~T11, §14~§19)이 HANDOFF에 한 줄도 없어 **"이미 만든 기능을 재설계"하는 사고(§18)가 실제로 발생**했다.
3. 같은 사실이 최대 6개 파일에 중복 기록된다(예: "꽃집 14봇 구현 완료"). 중복이 많을수록 갱신 비용이 커지고, 비용이 크니 안 지켜진다 — 악순환.
4. "종료 시 HANDOFF 갱신" 규칙은 `혜미_자동화_운영표.md` §3 한 곳에만 명문화돼 있고, 지켜졌는지 확인하는 장치가 없다.
5. 설계만 되고 안 쓰는 절차가 있다: TASK-PACKET/RESULT-CARD(`ops/tasks/` 실물 0건).
6. 반복 버그 3종은 "고침"이 아니라 매번 "재발→우회" 기록만 쌓인다: ①파일 잘림(truncation) ②bash 캐시로 인한 "유실 오판"(2026-07-15 AI OS, 07-16 §20 — 2회) ③`.git/index.lock` 충돌(07-13, 07-17 — 2회).

## 2. 재설계 원칙 5개

### 원칙 1 — 사실은 한 곳에만 (정본 지정)
| 알고 싶은 것 | 정본 (여기만 갱신) | 성격 |
|---|---|---|
| 지금 어디까지 했고 다음은 뭔가 (공장별) | 그 공장의 `HANDOFF.md` **맨 위 "현재 상태" 섹션 1화면** | 덮어쓰기 문서 |
| 이미 어떤 기능이 있나 (공장별) | 그 공장의 `기능_인덱스.md` | 덮어쓰기 문서 |
| 왜 그렇게 결정했나 | `ai공장짓기/decision-log_...md` (§번호) | append 전용 |
| 오늘 무슨 일이 있었나 | `2. Areas/Claude 세션로그/날짜.md` | append 전용 |
| 전체 작업 큐 | `1. Projects/완전자동화_실행계획.md` T-테이블 | 덮어쓰기 문서 |
| 누가 지금 작업 중인가 | `ai공장짓기/현재작업현황.md` | 덮어쓰기 문서 |

다른 모든 파일에는 사실을 다시 쓰지 않고 **[[링크]]만** 건다. 이미 있는 중복은 소급 삭제하지 않는다(동시작업 규칙 ② 보호) — 다만 앞으로 새 중복을 만들지 않는다.

### 원칙 2 — 진입점은 1개
`0. 최우선_확인파일.md`가 유일한 진입점. 세션 시작 필독은 **딱 3개**: ① 최우선_확인파일 → ② 최신 세션로그 1개 → ③ 작업할 공장의 HANDOFF + 기능_인덱스. `START-HERE.md`는 AI_OS 문서의 목차 역할로 강등(진입점 아님), `핵심맥락.md`는 "혜미가 누구인가" 배경 문서로 유지(매 세션 필독 아님).

### 원칙 3 — 규칙은 기억이 아니라 스크립트로 검사
`ai공장짓기/scripts/handoff_check.py` 신설: 각 공장 폴더의 최근 git 작업일과 HANDOFF의 "현재 상태" 날짜를 비교해 **2일 이상 밀리면 [WARN]** 출력. 세션 시작 시 + 22:30 예약 작업에서 실행. 사람이 기억할 일을 스크립트가 대신 잔소리한다.

### 원칙 4 — append 전용 vs 덮어쓰기 문서를 명시적으로 구분
- **append 전용**(세션로그, decision-log): 기존 내용 삭제 금지, 새 섹션만 추가, `[계정태그]` 필수 — 동시작업 규칙 ② 그대로.
- **덮어쓰기 문서**(HANDOFF "현재 상태" 섹션, 기능_인덱스, T-테이블, 현재작업현황): 최신이 정의이므로 덮어써도 된다. 단, **저장 직전 다시 읽기**(동시작업 규칙 ④)는 여전히 적용. HANDOFF의 과거 이력 부분은 건드리지 않고 "현재 상태" 섹션만 교체한다.

### 원칙 5 — 안 쓰는 절차는 보류 선언
TASK-PACKET/RESULT-CARD 절차는 실사용 0건이므로 **보류**로 선언하고 세션 필독 목록에서 제외한다. 삭제가 아니라 보류 — 여러 AI 도구를 병렬로 굴리는 시점이 오면 재검토. (AI_OS_설계서 자체는 보존.)

## 3. 반복 버그 3종 대응 확정

| 버그 | 확정 대응 | 상태 |
|---|---|---|
| ① 파일 잘림(truncation) | heredoc 작성 + `<!-- ok -->` sentinel + verify_write.py — **이미 유효 실증**(failure_log "이번 세션 0건") | 유지 |
| ② bash 캐시 "유실 오판" | bash `ls`/`cat`/`grep`이 "없다"고 해도 **Read 도구로 재확인 전엔 유실로 단정 금지**. 진입점 문서에 3줄 요약 게시 | 규칙 승격 |
| ③ git index.lock 충돌 | git add/commit/push는 **한 번에 한 세션만**: 실행 전 `현재작업현황.md`에 `[git]` 선언 → 끝나면 삭제. lock 파일이 있으면 1분 대기 후 재시도, 그래도 있으면 `mv`로 치우고 진행(기존 우회법 표준화) | 신설 |

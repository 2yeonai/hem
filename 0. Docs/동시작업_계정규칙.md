# 다중계정 동시작업 규칙 — hem 볼트를 여러 계정이 같이 쓸 때 #ai공장 #지침

## 왜 필요한가
hem 볼트를 여러 계정(가족/직원 등)이 동시에 열어 작업하는 상황이 실제로 발생했다(2026-07-17). 같은 날 서로 다른 세션이 서로 모르게 같은 파일(`완전자동화_실행계획.md`의 T3 등)을 동시에 고쳤고, 한 세션의 git 작업이 다른 세션의 변경과 부딪혀 index 충돌까지 났다(다행히 데이터 손실 없이 복구됨). 완전한 "잠금" 시스템을 만들 여력은 없으니, **저비용으로 충돌 확률을 낮추고, 충돌이 나도 누가 뭘 했는지 바로 알 수 있게** 하는 게 목표다.

## 규칙 1 — 계정 태그
공유 로그성 문서(`decision-log`, 각 스킬 `HANDOFF.md`, `완전자동화_실행계획.md`)에 새 기록을 남길 때는 날짜 옆에 계정 태그를 붙인다.

```
(2026-07-17, [gombeck1] Fable 5, 혜미 지시)
```

- 태그는 그 계정 이메일의 `@` 앞부분을 쓴다(예: `gombeck1@gmail.com` → `[gombeck1]`).
- 계정이 여러 개면 각자 자기 계정 이메일 앞부분을 그대로 쓰면 된다 — 별도 등록 절차 없음.
- 이미 있는 과거 기록에 소급 적용하지 않는다(생산성 낭비). 오늘 이후 새로 쓰는 기록부터.

## 규칙 2 — 공유 문서는 append-only
decision-log(§ 번호 추가), HANDOFF(회차 섹션 추가), 실행계획 T-테이블(행 추가)은 **기존 섹션의 내용을 지우거나 고쳐쓰지 않고, 새 섹션/새 행만 추가**한다. 두 세션이 동시에 편집해도 서로 다른 위치에 추가하면 자동으로 병합되고, 충돌이 나도 diff가 작아서 복구가 쉽다.

## 규칙 3 — 핵심 스킬 파일을 크게 고치기 전엔 작업 선언
`run.py`, `manifest.yaml` 같은 핵심 코드/스펙 파일을 **여러 함수/여러 섹션에 걸쳐** 고칠 예정이면, 시작 전에 `1. Projects/ai공장짓기/현재작업현황.md`에 한 줄 추가:

```
- [gombeck1] 2026-07-17 14:20~ : 정부지원 run.py draft_application() 예산/기대효과 로직 수정 중
```

작업이 끝나면 그 줄을 지운다. 다른 세션은 새 작업을 시작하기 전에 이 파일을 먼저 확인해서, 같은 파일·같은 함수를 건드릴 예정이면 잠깐 기다리거나 사람에게 물어본다. 작은 오타 수정, 한두 줄짜리 편집은 선언 없이 바로 해도 된다 — 이건 "여러 함수를 넘나드는 큰 작업"에만 해당.

**완화 조항 (2026-07-17, Fable 5 승인)**: 선언 줄의 타임스탬프가 **24시간 이상 지났으면 stale로 간주**하고 무시하거나 지워도 된다 — 삭제를 깜빡한 세션 하나 때문에 이후 모든 세션이 영구적으로 막히는 걸 방지하기 위함.

## 규칙 4 — 저장 직전엔 항상 최신본을 다시 읽는다
긴 작업(특히 서브에이전트에게 위임한 작업) 끝에 파일을 저장하기 직전, 그 파일을 한 번 더 Read해서 자신이 시작했을 때와 달라진 부분이 있는지 확인한다. 달라졌다면 그 사이 다른 세션이 손댄 것이므로, 덮어쓰지 말고 그 변경사항을 살려서 병합한다(2026-07-17 세션에서 실제로 이 패턴으로 충돌을 피한 사례 있음 — decision-log §18 후속 참고).

## 규칙 5 — git은 자주, 계정 태그와 함께 커밋
작업 단위가 끝날 때마다 커밋한다(다음 세션이 오기 전에). 커밋 메시지에도 계정 태그: `[gombeck1] T9: 홍재우 9인 심사위원 반영`. 커밋이 잘게 쪼개져 있을수록 충돌 시 되돌리기 쉽다. `.git/index.lock`이 걸려 있으면 지우기 전에 다른 세션이 지금 커밋 중인 건 아닌지 먼저 확인(가능하면).

## 예외 — 이 규칙이 필요 없는 경우
1인 계정으로만 작업 중이거나, 짧은 세션에서 파일 1~2개만 살짝 고치는 경우는 이 프로토콜을 안 지켜도 실무적으로 문제없다. 이 규칙은 "여러 계정이 겹쳐서 돌아가는 상황"을 위한 것이다.

## 1. 이 채팅 이후, 순서대로 할 일

2. **[30~60분] 방역 골든셋 인터뷰** — 유일한 병목, 이것만 하면 라인 전체가 움직임.
   - Cowork 새 세션 열기 → 모델 **Sonnet** 확인 → 아래 복붙:
   ```
   클로드 방역 ai/golden_set_template.yaml 을 읽고 나를 인터뷰해서 실제 사례로 채워줘.
   질문은 한 번에 하나씩. 기억 안 나는 필드는 "미확인"으로 두고 넘어가.
   완료 후 yaml 문법 검증을 실행하고 통과 결과를 붙여넣어. 3건 이상 목표.
   ```
   - 혜미는 질문에 답만 하면 됨.
3. **[10분] 이전 채팅 백업 신청**: claude.ai → 설정 → Privacy → Export data → 받은 파일을 `2. Areas/Claude 세션로그/과거기록/`에 넣기 → 실행계획 0단계의 컴파일 프롬프트 복붙.

## 3. "진짜 됐는지" 믿는 법 — 검증 루틴 #검증

클로드의 "완료" 보고를 그냥 믿지 않아도 됨. 세 가지 도구:

1. **증거 요구 문장** (모든 작업 지시 끝에 붙이기):
   ```
   완료 보고할 때 실제 실행 결과(테스트 출력/파일 크기/검증 로그)를 붙여넣어.
   증거 없는 항목은 "미검증"으로 표시해.
   ```
2. **직접 확인 3가지** (클릭만):
   - 파일이 진짜 생겼나 → 옵시디언에서 해당 폴더 열어보기 (이 문서도 그렇게 확인 가능)
   - 예약 작업이 진짜 있나 → 클로드 앱 왼쪽 사이드바 **Scheduled** 메뉴 → daily-session-log, weekly-synthesis 2개 보이면 정상. **Run now**를 한 번 눌러두면 이후 자동 실행이 권한 질문 없이 돌아감 (추천)
   - 자동기록이 도나 → 내일 아침 `2. Areas/Claude 세션로그/`에 오늘 날짜 노트 확인
3. **검증 전용 세션** (중요 작업 후, Sonnet):
   ```
   [작업물 경로]를 새 눈으로 검증해줘. 실제로 실행/열어보고, 주장과 실제가 다른 곳을 찾아.
   문제없으면 증거와 함께 PASS, 있으면 목록만 (고치지 말고).
   ```

### 2026-07-09 실검증 결과 (조작 없는 실제 실행 출력 기반)

| 항목 | 결과 | 증거 |
|---|---|---|
| 방역 12건 배치테스트 | ✅ PASS | "[전체 결과] PASS - 모든 갭 해결 확인", 12건 전부 만족도수집봇 도달, 미승인 발송 차단 작동 |
| 방역 manifest 검증 | ✅ PASS | "오류(FAIL): 0건 / 경고(WARN): 0건" |
| 방역·정부지원·꽃집 run.py 문법 | ✅ PASS | py_compile 3건 통과 |
| 정부지원 샘플 실행 | ✅ 정상 | sample_input.json 실행, 매칭 결과 JSON 출력 |
| 예약 작업 2개 등록 | ✅ 등록 확인 | 생성 시 시스템 확인 메시지 — 사이드바 Scheduled에서 직접 확인 가능 |
| 알려진 버그 | ⚠️ 파일 잘림 버그 실재 | 이 세션에서 CLAUDE.md가 실제로 잘렸고, 기록된 우회법으로 복구함. 대응책이 CLAUDE.md에 있으니 클로드가 알아서 우회함 |

주의: PASS는 "파이프라인 구조가 올바르다"는 뜻. 실제 AI 호출은 아직 모의(mock)라, 진짜 품질 검증은 골든셋(1번 할 일) 이후 가능 — 이건 버그가 아니라 설계된 순서.

## 4. 사용량 읽는 법과 운영 규칙 #사용량

오늘 수치(Pro): 세션 67% / 주간 전체 10% / Fable 18% → **주간은 아주 여유, 세션만 소진 중.**

- **세션 한도(5시간)**: 지금 67%는 이 Fable 대화가 쓴 것. 5시간마다 자동 회복 — 기다리면 됨.
- **주간 한도**: 매주 목 오전 4시 리셋. 10%면 매우 건강.
- **운영 규칙 4개**:
  1. 긴 대화 1개 > 짧은 대화 10개 (매번 CLAUDE.md를 다시 읽으므로). 단, **주제가 바뀌면 새 세션**.
  2. Fable은 설계·충돌·주간종합만, 반드시 승인 후 (지금처럼). 실행은 전부 Sonnet.
  3. 세션 한도에 걸리면: "여기까지 HANDOFF에 기록해줘" 시키고 중단 → 회복 후 새 세션에서 "HANDOFF 읽고 이어서".
  4. 사용량이 아깝다고 느껴지면 그 작업은 Haiku로 내려도 되는지 물어보기.

## 5. 범용러너와 클로드의 관계 (질문 답)

**러너를 Fable로 "만들지"는 말 것.** 이유: 러너는 매일 수십 번 도는 반복 실행기라, 비싼 모델을 러너로 쓰면 비용 구조가 무너짐. 이미 설계(ai 공장 종합 기획.md)도 그렇게 돼 있음 — 모델명 하드코딩 금지, 티어로만.

올바른 분담: **러너 몸통 = 코드(run.py 확장) + Sonnet이 구현 / 러너 설계·충돌 판단 = Fable(승인 후) / 단순 분류 stage = Haiku.** 즉 Fable은 러너가 아니라 러너의 **건축가·감리**로 쓰는 게 혜미님 계획에 최적. 착수 시점은 로드맵 R6(방역 검증 후).

# 혜미 업무스타일 재진단 + 자동화 기획 (2026-07-20, Fable 5) [hyemi]

> 작성 배경: Fable 5 접근 종료가 다가와, 혜미 요청으로 Fable급 관찰 판단을 기록으로 남긴다.
> 이 문서는 **제안서**다 — 아래 자동화 5안 중 채택분만 [[0. Docs/혜미_자동화_운영표|운영표]]·routing-policy에 반영한다(§0-3 인테이크 원칙과 동일: 평가 전 세팅 변경 없음).

## 1. 무엇이 달라졌나 (07-09 진단 → 07-20 재진단)

- 07-09 진단의 병목은 **판단 과부하**(정보 유입량 ≫ 판단 처리량)였다. 대응책(판단을 파일에 위임 — 실행계획/운영표/routing-policy)은 **실제로 작동했다**. 증거: 실행계획 T1~T14 전부 완료, 4개 공장 실 LLM 연동, 꽃집 웹앱 실배포까지 11일 만에 도달.
- **새 병목은 "검수·승인 적체"다.** AI가 만드는 속도가 혜미가 확인하는 속도를 추월했다. 지금 혜미 대기함: bizinfo 매칭 후보 298건(156+142) 검수, T14 헌장 승인, T12 brand-pack 8곳+스레드 계정, T13 가격 결정, 꽃집 웹앱 실사용 확인, RTS 실기기 확인 등. 판단은 파일에 위임했지만 **검수는 아직 사람 눈에만 의존**하는 구조라서 생긴 병목이다.

## 2. 강점 (증거 기반, 6개 — 07-09판 3개에서 확장)

1. **기록·인수인계 시스템** — HANDOFF/decision-log/자동 세션로그로 세션·계정이 끊겨도 이어짐. 맥락 재설명 0회 운영. (이건 전문 개발팀도 잘 못 하는 수준)
2. **승인 게이트 감각** — 되돌릴 수 없는 것(게시/발송/제출/삭제)에만 사람 게이트. 게이트를 남발하지 않아 게이트의 무게가 유지됨.
3. **구조적 사고** — 비개발자인데 계약(manifest)·티어·게이트 언어를 체득, 새 업무를 "입력→판단→출력→승인→인계" 5칸으로 쪼갬.
4. **자원 배분 감각** — 모델 티어 라우팅, "토큰 도둑" 규칙(상위 모델이 하위 모델 일을 직접 하면 낭비), 병렬 서브에이전트 기본. 비용 개념이 운영 규칙에 녹아 있음.
5. **검증 문화** — "뭐가 진짜 되는지" 현황판 요구, 블라인드 테스트, 외부 AI 감사와 교차 대조. AI의 과장 보고("완성됐어요")를 스스로 거르는 장치를 가짐.
6. **규칙의 코드화** — 지키기로 한 규칙을 의지가 아니라 pre-commit 훅(커밋 자동 거부)으로 강제. 규칙이 사람 기억에 의존하지 않음.

## 3. 약점·위험 (5개 — 각각 §4의 대응안과 연결)

1. **검수·승인 적체 (새 1순위 병목)** — 혜미 몫 큐가 쌓이는데 AI는 계속 새 산출물을 만듦. 적체가 길어지면 "만들었지만 안 쓰는 것"이 늘어남. → 대응 A·B
2. **동시 진행 과다** — 4개 공장 + RTS + 콘텐츠 + mo,on 본업을 혼자 굴림. 동시에 벌인 일의 개수 제한이 없어 이월 항목이 계속 쌓임(스레드 게시 진단, 방역 카드 5개, 방역·정부지원 웹앱 등). → 대응 C
3. **만들기 > 쓰기** — 새 기능 착수 속도가 실사용 확인 속도를 앞지름. 꽃집 웹앱 실사용, RTS 실기기 확인, T5 1주 사용 같은 "써보기" 항목이 계속 뒤로 밀림. 만든 것의 가치는 실사용에서만 확정된다. → 대응 D
4. **문서·상태 증식** — 세션마다 문서가 늘어나는 구조적 압력. 단, 이미 스스로 인지하고 정본 2단계 규칙 + CLAUDE.md 다이어트로 대응 중 — 유지가 관건.
5. **단일 도구 의존** — 이번 Fable 5 종료처럼 모델·도구 변동에 계획이 흔들릴 수 있음. 다행히 판단 근거가 decision-log에 남아 있어 어떤 상위 모델로도 대체 가능 — 라우팅 규칙만 갱신하면 됨. → 대응 E
